## 🔍 Что смотреть первым

### 🚀 **При начале работы над модулем:**

1. **Спецификация** (`docs/spec/[MODULE].md`)
   - Изучить структуру и семантику
   - Проверить размер и выравнивание
   - Понять инварианты и контракты

2. **Runtime реализация** (`runtime/src/[module].rs`)
   - Сравнить со спецификацией
   - Проверить `repr(C)` и размеры
   - Изучить методы и валидацию

3. **Тесты** (`runtime/src/lib.rs` и `tests/`)
   - Запустить unit тесты
   - Проверить cross-spec тесты
   - Убедиться в 100% покрытии

### 📋 **Порядок работы:**

1. **Изучить спецификацию** → понять требования
2. **Проверить runtime** → найти расхождения
3. **Написать/исправить тесты** → обеспечить валидацию
4. **Оптимизировать структуру** → добиться соответствия размерам
5. **Запустить все тесты** → убедиться в работоспособности

## 🧪 Тестирование

### 📊 **Виды тестов:**

1. **Unit тесты** (`runtime/src/lib.rs`)
   - Базовая функциональность модулей
   - Валидация структур
   - Проверка методов

2. **Cross-spec тесты** (`runtime/tests/cross_spec_validation.rs`)
   - Соответствие спецификациям
   - Размеры и смещения полей
   - Инварианты и поведение

3. **Integration тесты**
   - Взаимодействие модулей
   - COM timeline монотонность
   - UPO decay и eternal memory

### 🚀 **Команды тестирования:**

```bash
# Все тесты
cargo test

# Unit тесты
cargo test --lib

# Cross-spec тесты
cargo test --test cross_spec_validation

# Отладка размеров
cargo test debug_print_sizes -- --nocapture
```

## 🔢 Версионирование проекта

### 📋 **Формат версий: V0.00.0**
- **V1** - основной релиз (Foundation v0.1.0, Domain v0.2.0, Ashti_Core v0.3.0)
- **V0.x0** - очередность изменений (v0.1.1, v0.1.2, v0.1.3...)
- **V0.00.xx** - незначительные изменения (v0.1.01, v0.1.02...)

### 🎯 **Примеры версионирования:**
- **V0.1.0** - Foundation complete (основной релиз)
- **V0.1.1** - Domain V1.3 implementation (очередное изменение)
- **V0.1.01** - Fixed Domain tests (незначительное изменение)
- **V0.2.0** - Domain Engine complete (следующий релиз)

### ✅ **Правила версионирования:**
- **Major релизы** - меняют первую цифру (V0.1.0 → V0.2.0)
- **Feature изменения** - меняют вторую цифру (V0.1.0 → V0.1.1)
- **Bug фикс** - меняют третью цифру (V0.1.0 → V0.1.01)

```bash
# 1. Проверяем текущее состояние

# 3. Проверяем размеры структур

```

### 🔍 **Поиск проблем в проекте**
```bash
# Размеры структур
grep -r "size_of" runtime/src/

# Проверить выравнивание структур
grep -r "repr\(" runtime/src/

# Найти все TODO/FIXME
grep -r "TODO\|FIXME" runtime/ docs/

# Проверить неиспользуемый код
cargo check --warnings
```

### 📝 **Шаблон нового модуля**
```rust
// 1. Спецификация: docs/spec/[MODULE].md
// 2. Реализация: runtime/src/[module].rs
// 3. Тесты: runtime/src/lib.rs + tests/
// 4. Обновить Core Invariants.md
// 5. Обновить STATUS.md и ROADMAP.md
```

### 🎯 **Частые проблемы и решения**
- **Размер структуры не соответствует спецификации** → проверь `repr(C)` и порядок полей
- **Тесты не компилируются** → проверь импорты в `lib.rs`
- **Cross-spec тесты fail** → проверь смещения полей `std::mem::offset_of!`
- **COM timeline проблемы** → проверь монотонность `event_id`

## 📋 **Управление проектом**

### 🗺️ **ROADMAP.md - единственный источник плана**
- **Основной документ** с планом развития проекта
- **Содержит** все цели, приоритеты и сроки
- **Обновляется** при изменении планов или завершении этапов
- **Никогда не дублируется** - другие документы ссылаются на него

### ⏸️ **DEFERRED.md - отложенные задачи**
- **"Заглушки"** - временные решения, которые нужно заменить
- **Отложенная реализация** - функциональность, которую нельзя реализовать сейчас
- **Проблемы** - технические долги, требующие внимания
- **Приоритеты** - порядок решения отложенных задач

### 🎯 **Правила управления:**
1. **План только в ROADMAP.md** - никаких других планов
2. **Все заглушки в DEFERRED.md** - отслеживаем технический долг
3. **Регулярный аудит** - проверяем DEFERRED.md и закрываем задачи
4. **Четкие приоритеты** - что делать сейчас, что потом

### 📝 **Примеры заглушек:**
```
### Технический долг
- [ ] Оптимизировать Event до 32 байт (текущий 64 байта)
- [ ] Реализовать Domain V1.3 physics engine
- [ ] Добавить performance тесты

### Временные решения
```
