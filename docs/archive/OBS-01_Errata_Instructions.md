## 0. Краткая сводка

OBS-01 выявил три errata. Корни установлены:

| # | Симптом | Корень | Природа |
|---|---------|--------|---------|
| **E1** | CR energy=0 → dominant=Unknown → V6 ActivityTrace пустой | FNV-позиции текстов не пересекаются с anchor-позициями примитивов | Архитектурный (TextPerceptor) |
| **E2** | 100% AE conflict rate (96/96) | разобран отдельно с Sonnet | — (вне этого документа) |
| **E3** | SutraDepthStore пуст → emergent заблокирован | `apply_dream_update` существует, но не вызывается из engine | Структурный (missing call site) |

E1 и E3 **независимы** — разные фазы (Wake vs Sleep), разные call sites, чинятся параллельно.

**Решения:**
- **E1 → путь А** (anchor-matching в TextPerceptor). Сложнее, но единственно правильный. Хеш-случайные позиции делают семантику декоративной.
- **E3 → call site** + проверка существования аккумулятора activations.

## 1. E3 — сначала, потому что дёшево

### 1.1 Что известно (диагностика Sonnet подтверждена)

- DREAM запускается корректно (idle_threshold=200, min_wake_ticks=1000; в OBS-01 последняя инъекция тик 1600, прогон 3000 → DREAM срабатывает ~тик 1800).
- DreamCycle проходит Stabilization → Processing → Consolidation.
- `apply_dream_update` в `crates/axiom-runtime/src/over_domain/context_recognizer/mod.rs` существует, протестирован (`test_apply_dream_update_no_panic`), внутри вызывает `learning::apply_dream_depth_update` + `fatigue_store.apply_dream_recovery`.
- **Grep по axiom-runtime: вызов `apply_dream_update` только в тесте. В engine.rs — ноль.** Мост не написан.

### 1.2 Что нужно сделать

**Шаг 1 — проверить аккумулятор activations.**

`apply_dream_update(activations, all_known_ids, event_id)` требует `activations` — статистику реактиваций Frame с момента прошлого DREAM (какой Frame, в каком октанте, сколько раз активировался).

Проверить: **существует ли накопитель этой статистики в Wake-фазе?**

- Если да (где-то по ходу tick собирается reactivation-статистика per-frame per-octant) — взять её, передать в `apply_dream_update`. Фикс тривиален.
- Если нет — добавить аккумулятор. Минимально: `HashMap<(SutraId, Octant), u32>` который инкрементируется когда Frame реактивируется в Wake, и очищается после каждого DREAM. Это +один аккумулятор, небольшая работа.

**Шаг 2 — добавить call site.**

После завершения Consolidation в DreamCycle (после `apply_dream_cycle_commands`), в engine.rs вызвать:

```rust
context_recognizer.apply_dream_update(
    accumulated_activations,   // из шага 1
    all_known_primitive_ids,   // уже доступны из AnchorSet
    current_event_id,
);
```

Точное место — там, где DreamCycle сигнализирует завершение цикла (переход Consolidation → Waking). Sonnet видит это место по коду точнее.

**Шаг 3 — очистить аккумулятор** после применения, чтобы следующий DREAM считал свежую статистику.

### 1.3 Критерий успеха E3

После фикса, на прогоне с DREAM:
- `depth_store_len > 0` после первого DREAM-цикла
- глубины растут согласно `MAX_GROWTH_PER_CYCLE=100` (V5 §3.4)
- глубины Frame, активных в конкретных октантах, растут в этих октантах

### 1.4 Что разблокируется после E3

Цепочка: глубины растут → ScanningPlan получает реальные depth_range → emergent detection получает данные → можно оценить пороги 8000/30/100 (вопрос 6 OBS-01, сейчас "нельзя оценить").

## 2. E1 — путь А: anchor-matching в TextPerceptor

### 2.1 Корень проблемы

Из `BLUEPRINT.md` строка 71:
```
обычный токен → хэш пути (XOR доменов, ASHTI/MAYA)
Frame-анкер   → FNV-1a(sorted sutra_id участников), identity
```

И строка 840: `AGENT-TD-01: TextPerceptor: FNV-1a → embeddings. После OBS-01.`

**TextPerceptor сейчас** кладёт токены текста по FNV-1a хешу → случайная позиция в `[0..32767]³`. Frame-анкеры этих токенов тоже по FNV. А subsystem-примитивы (`math_function`, `time_sequence`, `prim_vline`...) стоят на **осмысленных фиксированных позициях** из `config/anchors/`.

Результат: текст падает в случайную точку, далеко от примитивов своей подсистемы. `compute_energies` сканирует регион вокруг Frame, не находит рядом примитивов → energy=0 → dominant=Unknown.

### 2.2 Принцип решения (путь А)

**Позиция токена при восприятии определяется притяжением к подходящим примитивам, а не хешем.**

Это реализация "целевого пути" из `Writing_V1_0.md` §4.2 (против "FNV fallback" §4.3).

```
Сейчас (FNV fallback):
  текст → FNV-1a(текст) & 0x7FFF → случайная позиция

Путь А (anchor-matching):
  текст → разбор на единицы → сопоставление с примитивами →
        → позиция = функция от позиций сопоставленных примитивов
```

### 2.3 Архитектура anchor-matching

Поскольку embeddings — большая отдельная работа (полноценный AGENT-TD-01), путь А делается **поэтапно**. V1 anchor-matching — без нейросетей, на прямом сопоставлении.

**Этап A1 — таблица сопоставления (lookup-based).**

Создать таблицу декомпозиции: какие входные единицы соответствуют каким примитивам.

```rust
// crates/axiom-agent/src/text_perceptor/anchor_match.rs

pub struct AnchorMatchTable {
    // символ/токен текста → список примитивов с весами
    char_to_primitives: HashMap<char, Vec<(PrimitiveId, f32)>>,
    // распознанные слова-маркеры → подсистема
    word_to_subsystem: HashMap<String, SubsystemId>,
}
```

Примеры наполнения:
```
'+' → [(math_operation, 0.8), (prim_vline, 0.3), (prim_hline, 0.3)]
'=' → [(math_relation, 0.9), (prim_hline, 0.5)]
'f' → [(prim_vline, 0.4), (prim_hook, 0.4)]  # графически
"lim" → subsystem Mathematics, ассоциация с math_limit
"если" → subsystem Logic/Dilemmas
цифры '0'-'9' → math_element
буквы → соответствующие графические примитивы (Writing)
```

Источник таблицы: на старте — небольшая ручная таблица для частых символов (латиница, кириллица, цифры, базовые операторы). Расширяется постепенно.

**Этап A2 — вычисление позиции через притяжение.**

```rust
fn compute_position_from_anchors(
    matched: &[(PrimitiveId, f32)],
    anchor_set: &AnchorSet,
) -> [i16; 3] {
    if matched.is_empty() {
        return fnv_fallback_position();  // сохраняем fallback
    }
    
    // взвешенный центроид позиций сопоставленных примитивов
    let mut acc = [0i64; 3];
    let mut total_weight = 0.0;
    for (prim_id, weight) in matched {
        let prim_pos = anchor_set.get_primitive(prim_id).position;
        for i in 0..3 {
            acc[i] += (prim_pos[i] as f64 * weight) as i64;
        }
        total_weight += weight;
    }
    
    [
        (acc[0] as f64 / total_weight) as i16,
        (acc[1] as f64 / total_weight) as i16,
        (acc[2] as f64 / total_weight) as i16,
    ]
}
```

Токен ложится рядом с примитивами, которым соответствует. Математический `+` — рядом с `math_operation`. Буква — рядом с графическими примитивами.

**Этап A3 — fallback сохраняется.**

Если входная единица не в таблице — FNV-fallback как сейчас. Это гарантирует что система не падает на незнакомом вводе, просто такой токен пока "несемантичен" (как и сегодня все). Постепенное наполнение таблицы увеличивает покрытие.

### 2.4 Где менять код

```
crates/axiom-agent/src/text_perceptor/
├── mod.rs                  — изменить: вызов anchor_match перед fnv
├── anchor_match.rs         — НОВОЕ: AnchorMatchTable, compute_position_from_anchors
└── decomposition_table.rs  — НОВОЕ: начальная таблица char→primitives, word→subsystem
```

TextPerceptor получает доступ к `&AnchorSet` (позиции примитивов). Если сейчас не имеет — пробросить через конструктор.

### 2.5 Альтернатива/мост (путь Б) — НЕ выбран, но зафиксирован

Если путь А окажется слишком большим для одного шага, **временный мост**: CR сканирует не только регионы EXPERIENCE-Frame, но и напрямую subsystem-ref регионы (проверяет близость каждого Frame ко всем примитивам, где бы Frame ни стоял).

**Почему НЕ выбран:** лечит симптом. Frame в случайной FNV-позиции может случайно оказаться близко к Math-примитиву → ложная энергия. Это шум, выдающий себя за смысл. Путь Б оживит метрики, но они будут врать.

**Путь А чинит корень:** Frame живёт там, где его смысл.

Путь Б упомянут на случай, если нужно быстро разблокировать V6 для дальнейших тестов **до** готовности А. Тогда: Б как временный костыль с пометкой "energy ненадёжна", А как замена. Решение за Sonnet+chrnv по ситуации.

### 2.6 Критерий успеха E1

После фикса, на прогоне с теми же 8 текстами:
- `compute_energies` возвращает ненулевые значения для подсистем, к которым относятся тексты
- `dominant_subsystem != Unknown` для Frame, собранных из распознанных символов
- Математический текст → dominant=Mathematics (или близкая); текстовый → Writing
- V6 ActivityTrace начинает заполняться (`fill_count > 0`)

### 2.7 Связь с AGENT-TD-01 (embeddings)

Путь А этапы A1-A3 — это **lookup-based anchor matching**, не embeddings. Это сознательно: embeddings (полный AGENT-TD-01) — большая отдельная работа.

Lookup-matching — **мост между FNV-хаосом и будущими embeddings**:
- Сейчас: FNV (всё случайно)
- Путь А: lookup (частые символы осмысленны, остальное fallback)
- Будущее (AGENT-TD-01): embeddings (всё осмысленно через обученное представление)

Путь А не конфликтует с будущими embeddings — он их предшественник. Когда embeddings появятся, они заменят lookup-таблицу, а механизм `compute_position_from_anchors` (притяжение к примитивам) останется.

## 3. E2 — вне этого документа

E2 (100% AE conflict rate) разобран отдельно chrnv с Sonnet. В эту инструкцию не входит по решению chrnv. 

Замечание для координации: после фикса E1 стоит **перемерять conflict rate**. Гипотеза Opus была что E2 может частично быть следствием E1 (AxialEvaluator оценивает шумовые FNV-позиции → analytic и synthetic расходятся систематически). Если после E1-фикса rate упадёт с 100% — подтвердит связь. Если останется 100% — E2 полностью самостоятелен (разбор Sonnet это покрывает).

## 4. Порядок работ

```
1. E3 (дёшево, разблокирует V5 depth-механику)
   1.1 Проверить аккумулятор activations (есть/нет)
   1.2 Добавить call site apply_dream_update после Consolidation
   1.3 Очистка аккумулятора после применения
   1.4 Проверка: depth_store_len > 0 после DREAM

2. E1 путь А (большая работа, чинит корень)
   2.1 anchor_match.rs + decomposition_table.rs
   2.2 Начальная таблица: латиница, кириллица, цифры, базовые операторы
   2.3 compute_position_from_anchors (взвешенный центроид примитивов)
   2.4 Интеграция в TextPerceptor, fallback на FNV сохранён
   2.5 Проброс &AnchorSet в TextPerceptor если нужно
   2.6 Проверка: energy != 0, dominant != Unknown

3. Перемерять E2 conflict rate после E1
   3.1 Если упал — подтверждение связи E1→E2
   3.2 Если 100% — E2 самостоятелен (разбор Sonnet)

4. Повторный OBS-02 прогон
   4.1 Те же 8 текстов
   4.2 Проверить все 6 вопросов ROADMAP заново
   4.3 Особенно: вопрос 2 (SubsystemId) и вопрос 6 (пороги depth)
```

E3 и E1 параллелятся если есть ресурс. E3 быстрее, начать с него разумно.

## 5. Что НЕ трогать

- **Token 64 байта** — HARD, ничего не добавлять (E1/E3 не требуют изменений Token).
- **FNV fallback** — НЕ удалять, он остаётся для незнакомого ввода (путь А §2.3 A3).
- **Архитектуру over-domain модулей** — E1/E3 это интеграционные баги, не архитектурные. Координатор (CR-TD-01) — отдельная работа, идёт своим чередом.
- **DreamCycle логику** — E3 это пропущенный вызов снаружи цикла, сам цикл корректен.
- **Определение октанта в AxialEvaluator** — это область E2, вне этого документа.
