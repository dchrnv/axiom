## 1. 🔥 КРИТИЧЕСКИЕ ПРОБЛЕМЫ - БЛОКИРУЮТ РАЗРАБОТКУ

### 1.1 DomainConfig V1.0 спецификация - ВЫСШИЙ ПРИОРИТЕТ

**Где:** `runtime/src/domain.rs`, `docs/spec/DomainConfig V1.0.md`  
**Что отложено:** DomainConfig не соответствует спецификации V1.0  
**Почему:** Текущая реализация 184 байта, спецификация требует 128 байт  
**Когда планируется:** v0.2.1 - НЕМЕДЛЕННО  

**Текущий статус:** 
- **Текущий размер:** 184 байта
- **Требуемый размер:** 128 байт (спецификация V1.0)
- **Проблема:** Неполная реализация структуры из спецификации

**Что требует реализации:**
- **Физика поля (32 байта):** gravity_strength, friction_coeff, resonance_freq, temperature, pressure, viscosity, elasticity, quantum_noise, time_dilation
- **Семантические оси (16 байт):** axis_x_ref, axis_y_ref, axis_z_ref, axis_config
- **Мембрана (32 байта):** input_filter, output_filter, permeability, thresholds, gate_complexity, membrane_state
- **Метаданные (32 байта):** token_capacity, connection_capacity, energy_budget, complexity_score, processing_state

**Предустановленные конфигурации:**
- SUTRA (Домен 0) - Источник истины
- LOGIC (Домен 6) - Чистое вычисление  
- DREAM (Домен 7) - Фоновая оптимизация
- MAYA (Домен 10) - Интерфейс проекции

**Влияние:** КРИТИЧЕСКОЕ - DomainConfig является ядром системы и должен соответствовать спецификации

### 1.2 Остальные падающие тесты (низкий приоритет)

**Где:** `runtime/src/lib.rs`, `runtime/src/config/tests.rs`, `runtime/src/connection.rs`  
**Что отложено:** 6 тестов все еще падают из 41  
**Почему:** Не критично для функциональности, не блокируют разработку  
**Когда планируется:** v0.2.1 - после завершения основной функциональности  

**Падающие тесты:**
- `test_config_initialization` - проблема с параллельным выполнением тестов (конфликт рабочей директории)
- `test_domain_v1_3_validation` - проблема с валидацией доменов, не связана с конфигурацией
- `connection::tests::test_connection_from_preset` - падает только при параллельном выполнении
- `connection::tests::test_connection_presets_loading` - падает только при параллельном выполнении  
- `connection::tests::test_connection_topology_constraints` - падает только при параллельном выполнении

**Причина проблем:** Вероятно связано с параллельным выполнением тестов и изменением рабочей директории в domain тестах через `env::set_current_dir("/home/chrnv/Axiom").ok();`

**Влияние:** Минимальное - основная функциональность работает корректно, по отдельности все тесты проходят

### 1.2 Оптимизация размеров структур до спецификаций - ПЕРЕНЕСЕНО В v0.4.0

**Где:** `runtime/src/event.rs`, `runtime/src/upo.rs`, `runtime/src/domain.rs`  
**Что отложено:** Приведение размеров структур к целевым значениям из спецификаций  
**Почему:** Превышение размеров влияет на производительность и память  
**Когда планируется:** v0.4.0 - Performance Optimization  

**Текущий статус:**
- **Event:** 64 байта → 32 байта (-32 байта, -50%)
- **DynamicTrace:** 64 байта → 32 байта (-32 байта, -50%)  
- **DomainConfig:** 176 байт → 128 байт (-48 байт, -27%)

**Проблемы:**
- Выравнивание полей создает избыточный padding
- Несоответствие спецификациям размеров
- Потенциальные проблемы с кэшированием и SIMD

**Причина переноса:** Оптимизация размеров не блокирует текущую разработку модульной интеграции и может быть выполнена позже без ущерба для функциональности.

## 11. Configuration System - исправление тестов

**Где:** `runtime/src/config/tests.rs`  
**Что отложено:** Исправление путей загрузки конфигураций в тестах  
**Почему:** Тесты запускаются из `runtime/` директории, но пытаются загрузить файлы из `config/`  
**Когда планируется:** v0.2.0 - Исправление тестов  

### 🐛 **Проблема:**

**Текущие тесты:**
- Запускаются из `runtime/` директории
- Пытаются загрузить файлы по относительным путям `config/axiom.yaml`
- Рабочая директория тестов != рабочая директория приложения

**Ожидаемое поведение:**
- Тесты должны запускаться из корня проекта
- Пути к конфигурационным файлам должны быть корректными
- Файлы должны находиться в правильных местах

### 🔧 **Требуется исправить:**

#### **1. Настройка рабочей директории:**
```rust
// В тестах установить правильную рабочую директорию
#[cfg(test)]
fn setup_test_config() {
    // Установить рабочую директорию в корень проекта
    std::env::set_current_dir("/home/chrnv/Axiom");
}
```

#### **2. Абсолютные пути в тестах:**
```rust
// Использовать абсолютные пути для надежности
let config_path = Path::new("config/axiom.yaml");
let runtime_path = Path::new("config/runtime/runtime.yaml");
let schema_path = Path::new("config/schema/domain.yaml");
```

#### **3. Создание тестовых файлов:**
```rust
// Создавать тестовые файлы перед запуском тестов
fn create_test_files() {
    // Убедиться что файлы существуют
    fs::create_dir_all("config/").ok();
    fs::create_dir_all("config/runtime").ok();
    fs::create_dir_all("config/schema").ok();
}
```

#### **4. Использование временной директории:**
```rust
// Для тестов использовать временную директорию
#[cfg(test)]
const TEST_CONFIG_DIR: &str = "/tmp/axiom_test_config/";
```

### 🚀 **План исправления:**

**Phase 1: Быстрое исправление**
- Исправить пути в существующих тестах
- Добавить setup/teardown для тестовых файлов
- Использовать абсолютные пути

**Phase 2: Улучшение тестов**
- Добавить тесты для отсутствующих файлов
- Тесты валидации с ошибочными данными
- Тесты горячей перезагрузки

**Phase 3: Тестовая инфраструктура**
- Отдельная тестовая конфигурация
- Mock файлы для edge cases
- Интеграционные тесты с реальными модулями

### 📊 **Конечная цель:**

**Надежные тесты:**
- Все тесты проходят независимо от рабочей директории
- Правильная обработка отсутствующих файлов
- Валидация путей и конфигураций
- Изоляция тестовых данных

### 🎯 **Критерии успеха:**

1. **cargo test config** проходит из любой директории
2. **Тесты создают** свои конфигурационные файлы
3. **Пути корректны** для любой рабочей директории
4. **Очистка** после выполнения тестов
5. **Изоляция** тестовых данных от production
