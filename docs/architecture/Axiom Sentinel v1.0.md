# Спецификация Axiom Sentinel v1.1: Адаптивный Когнитивный Движок

## 1. Концепция: Axiom Hybrid

**Axiom Hybrid** — это архитектура, которая превращает движок из набора жестких алгоритмов в динамическую среду, способную к самопрофилированию и предсказательному моделированию. Главная цель: удержание «мыслительного акта» в пределах **1 ms** при любых изменениях среды.

### 1.1. Hardware-Aware Topology (Самодиагностика ресурсов)

При инициализации (`AxiomEngine::new`) система определяет свои физические границы:

- **Динамический Worker Pool:** Вместо констант используется `std::thread::available_parallelism()`. Резервируется 1 поток под ОС/Gateway, остальные распределяются как вычислительные `Workers`.
- **Кэш-лимиты:** Задачи (chunks) нарезаются под размер **L2 кэша** (512 KB для Ryzen 5 3500U), чтобы избежать деградации скорости при работе с большими объемами данных (например, >1M токенов).
- **Memory Pressure:** Лимит доступной RAM (~1.8 GiB) становится триггером для агрессивной очистки памяти.

## 2. Когнитивная архитектура (11 Доменов × 8 Слоев)

### 2.1. Семантическая Матрица «Плавающих задач»

Мы отказываемся от привязки «домен = поток» в пользу балансировки нагрузки:

- **Параллельное дробление:** Тяжелые операции, такие как `resonance_search` в домене `EXPERIENCE`, дробятся на чанки и распределяются между всеми свободными воркерами.
- **Волнообразная приоритезация:** 8 семантических слоев обрабатываются иерархически. Слои 1–3 (Рефлексы) имеют приоритет исполнения. Слои 4–8 могут быть отложены, если бюджет тика исчерпан на 80%.
- **Lock-free Parallelism:** Реализация через разделение данных на непересекающиеся куски, что исключает блокировки (Mutex) в горячих циклах.

### 2.2. Speculative Layer (Слой предсказания)

Для минимизации задержек система использует избыточные мощности для «заглядывания вперед»:

- **Параллельные гипотезы:** Пока `Arbiter` занят текущим тиком, свободные воркеры просчитывают 2–3 вероятных состояния `SpatialHashGrid` для следующего шага.
- **Zero-Cost Switch:** При подтверждении гипотезы сенсорами система выполняет мгновенный переход (`restore_from`) к предсчитанному состоянию (~9 µs вместо полных ~40 µs).

## 3. Жизненный цикл данных и Энергоэффективность

### 3.1. Динамическая дистилляция памяти

- **Активная фаза:** Новые трейсы хранятся в быстрых `HashMap` до порога в 1000 единиц (стабильный поиск ~10.6 µs).
- **Кристаллизация:** По достижении 5000 трейсов запускается `export_skills` (5.6 ns), сжимающий опыт в компактные «скиллы».
- **Автономный GC:** Процедуры `run_horizon_gc` и `snapshot_and_prune` (32–40 µs) активируются при критическом давлении на L3 кэш или RAM.

### 3.2. Variable Tick Rate (Эластичный биоритм)

Система адаптирует частоту пульсации под когнитивную нагрузку:

- **Low Power (60 Hz):** Режим ожидания при отсутствии внешних стимулов.
- **High Alert (1000 Hz):** Мгновенная мобилизация всех 11 доменов при обнаружении семантических конфликтов.

---

## 4. Адаптивный JIT и Интерфейсы

- **Direct Sensor Injection:** Прямая запись данных в `DomainState` (20 ns), аналогичная `FractalChain::inject_input`, устраняющая накладные расходы на аллокацию.
- **Векторный форсаж (SIMD):** При обнаружении **AVX2** (256-бит) расчеты `apply_gravity_batch` ускоряются до **80–120M токенов/сек**.
