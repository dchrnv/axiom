# AXIOM SPECIFICATION: Time Model V1.0

**Статус:** Конституционный документ (foundational)  
**Версия:** 1.0.0  
**Дата:** 2026-03-19  
**Назначение:** Определяет единственную модель времени для всей системы AXIOM  
**Связанные спеки:** COM V1.0, Event-Driven V1, Causal Frontier V1, Heartbeat V2.0

---

## 1. Проблема

Слово "время" в контексте AXIOM означает три разные вещи, которые часто путаются:

- Порядок событий (что раньше, что позже)
- Длительность процессов (затухание, гравитация, остывание)
- Связь с внешним миром (секунды, миллисекунды)

Эта спецификация разделяет их на три изолированных слоя и запрещает смешение.

---

## 2. Три слоя времени

### Слой 1: Причинный порядок (Causal Order)

**Где живёт:** Внутри ядра AXIOM  
**Механизм:** COM `event_id` (монотонный u64)  
**Спецификация:** COM V1.0

Причинный порядок — это единственная "ось времени" внутри системы. Одно событие порождает другое, каждое получает уникальный `event_id`. Порядок `event_id` полностью определяет эволюцию системы.

```
stateₙ₊₁ = F(stateₙ, eventₙ)
```

В этом слое **нет часов, секунд, тиков и delta_time**. Есть только последовательность причин и следствий.

### Слой 2: Причинный возраст (Causal Age)

**Где живёт:** Внутри ядра AXIOM  
**Механизм:** Разница `event_id`  
**Используется:** Для затухания, гравитации, остывания, любых процессов зависящих от "длительности"

Причинный возраст сущности определяется как:

```
causal_age = current_event_id - entity.last_event_id
```

Это **не время в секундах**. Это мера причинной активности, прошедшей с момента последнего обновления сущности. Все внутренние процессы, зависящие от "длительности", используют причинный возраст:

```rust
fn compute_decay(token: &Token, current_event_id: u64, config: &DomainConfig) -> f32 {
    let age = current_event_id - token.last_event_id;
    let decay_factor = (-age as f32 * config.decay_rate).exp();
    decay_factor
}
```

Причинный возраст полностью детерминирован и воспроизводим.

### Слой 3: Реальное время (Wall-Clock Time)

**Где живёт:** Только в адаптерах, за границей ядра  
**Механизм:** Системные часы операционной системы  
**Используется:** Для связи с внешним миром (LLM, сенсоры, пользователь, API)

Реальное время существует только на границе системы. Адаптер (REST, WebSocket, CLI, LLM-клиент) может приклеить `timestamp` к входящему сигналу. Но этот `timestamp` — метаданные для внешнего потребителя. Ядро его **не использует для вычислений**.

Цепочка:

```
Внешний сигнал (с timestamp)
    → Адаптер (отбрасывает/сохраняет timestamp как метаданные)
    → UCL Command (без timestamp)
    → Ядро (присваивает event_id)
    → Причинный порядок
```

## 3. Четыре правила

### Правило 1: Ядро не знает что такое секунды

Ни один модуль внутри ядра AXIOM (Token, Connection, Domain, COM, UPO, Causal Frontier) не может использовать, хранить или зависеть от wall-clock времени. Единственная мера порядка и длительности — `event_id`.

### Правило 2: Возраст сущности — это разница event_id

Любой процесс, зависящий от "прошедшего времени" (затухание, остывание, ослабление связей, гравитационная адаптация), вычисляется через `current_event_id - entity.last_event_id`. Это число детерминировано и одинаково при каждом воспроизведении.

### Правило 3: Heartbeat — легитимный тип причинности

"Прошло N событий" является достаточной причиной для генерации нового события. Heartbeat не возвращает время в систему. Он выражает факт: "причинная глубина системы увеличилась на N шагов, фоновые процессы могут потребовать обработки". Heartbeat добавляет сущности в Causal Frontier для ленивой обработки.

### Правило 4: Реальное время живёт только за границей

Wall-clock timestamp может существовать только в слое адаптеров (REST, WebSocket, CLI). Он может быть сохранён как метаданные события для внешних потребителей, но ядро никогда не использует его для принятия решений или вычислений.

## 4. Запрещённые практики

Следующие практики являются нарушением модели времени:

- Использование `std::time`, `SystemTime`, `Instant` внутри ядра
- Поля `timestamp_ms`, `created_at_unix`, `duration_seconds` в core-структурах
- Вычисление decay/gravity/cooling на основе реального времени
- Генерация event_id на основе системных часов
- Использование `sleep()`, `delay()`, таймеров внутри ядра
- Зависимость порядка обработки от wall-clock времени

---

## 5. Допустимые практики

- `event_id` как единственная мера порядка и возраста
- Причинный возраст (`current_event_id - last_event_id`) для всех "временных" процессов
- Heartbeat по счётчику событий для фоновых процессов
- Wall-clock timestamp в адаптерах для внешних потребителей
- `Option<u64>` для опционального timestamp в метаданных событий (не используется ядром)

---

## 7. Философский инвариант

Время в AXIOM — это не координата. Это глубина причинной истории.

Если нет событий — нет изменений — нет времени. Система в статическом состоянии не стареет.

Причинный порядок делает время **производной от изменений**, а не независимой осью. Это фундаментальное отличие от систем, построенных на глобальном тике.
