# ПРАВИЛА РЕАЛИЗАЦИИ API В AXIOM

## Назначение

Данный документ определяет правила реализации внешних интерфейсов системы AXIOM (REST, WebSocket, gRPC, CLI).

Целью правил является обеспечение архитектурной изоляции протоколов транспортного уровня от внутренней логики системы.  
API-слой должен выступать исключительно в роли границы трансляции между внешним миром и внутренней моделью AXIOM.

---

## Основной принцип

**Модули и ядро AXIOM не должны зависеть от транспортных протоколов.**

Внутренняя логика системы не должна напрямую использовать или зависеть от:

- HTTP
    
- WebSocket
    
- gRPC
    
- CLI-фреймворков
    
- форматов сериализации (JSON, protobuf и т.п.)
    

Вся транспортная логика должна быть изолирована в слое runtime-адаптеров.

---

## Архитектурная граница

Система разделяется на три уровня взаимодействия:

### 1. Модули (Modules)

Модули реализуют предметную логику системы.

Они:

- работают через команды, события и запросы
    
- реализуют поведение системы
    
- не должны импортировать или использовать транспортные библиотеки
    

---

### 2. Внутренние интерфейсы (Internal Interfaces)

Внутренние интерфейсы определяют язык взаимодействия компонентов системы.

Они описывают:

- команды
    
- события
    
- запросы состояния
    

Это канонический внутренний API AXIOM.

---

### 3. Runtime-адаптеры (Adapters)

Runtime-адаптеры реализуют взаимодействие с внешним миром.

Их задачи:

- обработка протоколов
    
- разбор входящих запросов
    
- сериализация и десериализация
    
- управление соединениями
    
- трансляция запросов во внутренние команды
    

---

## Схема взаимодействия

Любое внешнее взаимодействие должно проходить через следующую цепочку:

Внешний протокол → Адаптер → Внутренний интерфейс → Модуль

Прямой доступ протоколов к модулям запрещён.

---

## Модель взаимодействия: Команды, Запросы, События

Взаимодействие внутри AXIOM строится на трёх типах операций.

### Команды (Commands)

Команды инициируют изменение состояния системы.

Примеры:

- `StartSensor`
    
- `StopActuator`
    
- `ExecuteControlCommand`
    

Команды поступают от внешних клиентов и передаются через адаптеры.

---

### Запросы (Queries)

Запросы возвращают текущее состояние системы и не должны изменять его.

Примеры:

- `GetSensorState`
    
- `GetSystemStatus`
    

Запросы должны быть детерминированными и не иметь побочных эффектов.

---

### События (Events)

События представляют уведомления о произошедших изменениях или наблюдениях системы.

Примеры:

- `FrameDetected`
    
- `SensorStateChanged`
    
- `ControlLoopUpdate`
    

События могут транслироваться клиентам через потоковые протоколы (например WebSocket).

---

## Ответственность API-слоя

Адаптеры API отвечают исключительно за:

- разбор входящих запросов
    
- аутентификацию и авторизацию
    
- трансляцию протоколов
    
- сериализацию данных
    
- управление соединениями
    
- доставку потоковых событий
    

Адаптеры **не должны содержать предметную логику системы**.

---

## Требования к стабильности

Внутренние команды, события и запросы должны быть значительно стабильнее, чем внешние протоколы.

Протоколы могут изменяться, но внутренний язык взаимодействия системы должен оставаться консистентным.

---

## Правило разработки новой функциональности

При добавлении новой возможности необходимо соблюдать следующий порядок:

1. Определить внутреннюю команду, запрос или событие.
    
2. Реализовать поведение в соответствующем модуле.
    
3. Добавить поддержку во внешних адаптерах.
    

Проектирование внешнего API **не должно предшествовать определению внутренних интерфейсов**.

---

## Запрещённые практики

Следующие архитектурные практики считаются нарушением:

- импорт HTTP или WebSocket библиотек внутри модулей
    
- реализация предметной логики внутри обработчиков API
    
- использование структур сетевых сообщений как внутренних моделей
    
- перенос бизнес-логики в слой адаптеров
    

---

## Стратегия протоколов

В системе могут одновременно использоваться несколько протоколов.

Типичные роли:

- REST — конфигурация и получение состояния
    
- WebSocket — поток событий и обновления в реальном времени
    
- CLI — операционное управление системой
    
- gRPC — межсервисное взаимодействие (опционально)
    

Все протоколы должны транслироваться в единый внутренний язык команд, запросов и событий.

---

## Дополнительное правило: внутренний API системы

**Команды, события и запросы являются истинным API AXIOM.**

Внешние протоколы (REST, WebSocket, gRPC, CLI) рассматриваются лишь как представления или проекции этого API.

Изменение или добавление транспортного протокола не должно требовать изменения модулей или внутренней модели системы.

---

## Архитектурная цель

Ядро AXIOM должно оставаться полностью функциональным даже при полной замене слоя внешних протоколов.

Корректная реализация системы позволяет переписать runtime-слой без изменения модулей и внутренних интерфейсов.
