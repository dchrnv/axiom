
# UPO v2.1 — Полная спецификация для Axiom (Token + Connection + Causal Order)

**Версия:** 2.1  
**Статус:** УТВЕРЖДЕНА  
**Система:** Axiom  
**Язык реализации:** Rust (ядро)  
**Размер структуры DynamicTrace:** 32 байта (с выравниванием)  
**Модель времени:** Causal Order (причинный порядок событий, event_id)  
**Связанные спеки:** COM, Token V5.0, Domain V1.2  
**Последнее обновление:** 2026-03-01  

---

## 1. Обзор и назначение

**Universal Phase Operator (UPO)** — модуль, который наблюдает за динамикой в одном **домене** (семантическом пространстве) Axiom, используя данные из структур **Token**  и **Connection** . На основе их движения, напряжений и термодинамических параметров UPO вычисляет текущие динамические характеристики и проецирует их в трёхмерное пространство **экрана** в виде точки с весом.

Время в системе понимается как **причинный порядок событий** (Causal Order Model). Глобальной временной оси нет; эволюция определяется последовательностью событий с монотонно возрастающими идентификаторами `event_id`. UPO использует `event_id` для маркировки следов и организации затухания экрана.

Экран является общим для всех доменов, но разделён на октанты знаками координат. Точки на экране — **не токены**, а **следы динамики** (узоры). Они накапливаются, затухают по событийной шкале, но **никогда не достигают нуля** (минимальный предел). Это гарантирует вечную память системы. Экран текущего уровня может быть заморожен и передан как библиотека следующему уровню иерархии.

UPO **не интерпретирует** семантику, **не ищет паттерны** — это задача других модулей (IntuitionEngine, PatternRecognizer). Его единственная задача — преобразовывать динамику домена в точки на экране.

```
Token'ы и Connection'ы в домене D
    ↓ (чтение полей)
[UPO] — вычисляет метрики и сворачивает в (x, y, z, weight) + берёт текущий event_id
    ↓ (запись)
Экран — трёхмерное пространство, затухание по event_id, min_intensity > 0, октанты по знакам координат
    ↓ (чтение и анализ)
IntuitionEngine и другие модули — ищут паттерны, интерпретируют
    ↓ (опционально)
Заморозка экрана → библиотека следующего уровня
```

## 2. Структуры Token и Connection (Axiom)

### 2.1 Token – 64 байта, выравнивание 64

```rust
#[repr(C, align(64))]
pub struct Token {
    // --- ИДЕНТИФИКАЦИЯ (8 Байт) ---
    pub sutra_id: u32,       // Глобальный ID смысла
    pub domain_id: u16,      // Текущий домен
    pub type_flags: u16,     // Тип сущности

    // --- ЛОКАЛЬНАЯ ФИЗИКА ПОЛЯ (16 Байт) ---
    pub position: [i16; 3],  // X, Y, Z
    pub velocity: [i16; 3],  // Вектор скорости
    pub target: [i16; 3],    // Целевая точка (аттрактор)
    pub reserved_phys: u16,

    // --- ТЕРМОДИНАМИКА (4 Байта) ---
    pub valence: i8,         // Эмоциональный заряд (-127..+127)
    pub mass: u8,            // Инерционная масса (1..255)
    pub temperature: u8,     // Пластичность (0..255)
    pub state: u8,           // Active, Sleeping, Locked...

    // --- ФРАКТАЛЬНАЯ НАВИГАЦИЯ (36 Байт) ---
    pub lineage_hash: u64,
    pub context_payload: [u8; 28],
}
```

### 2.2 Connection (связь) – 64 байта, выравнивание 64

```rust
#[repr(C, align(64))]
pub struct Connection {
    // --- ТОПОЛОГИЯ (16 Байт) ---

    pub domain_id: u16,

    pub flags: u32,

    // --- ДИНАМИКА (16 Байт) ---

    // --- МЕТАДАННЫЕ (16 Байт) ---
    pub created_at: u64,     // event_id создания

}
```

## 3. Модель времени: Causal Order

В системе отсутствует глобальное физическое время. Эволюция определяется строго монотонной последовательностью событий.

```rust
#[repr(C)]
pub struct Event {
    pub event_id: u64,        // Глобальный причинный индекс
    pub domain_id: u16,       // Домен, инициировавший событие
    pub event_type: u16,      // Тип (UpdateToken, UpdateConnection, Tick, ...)
    pub payload_hash: u64,    // Хеш для детерминизма
    pub _reserved: [u8; 8],
}
```

**Инварианты:**
- `event_id` генерируется атомарно и строго возрастает.
- Все изменения состояния применяются в порядке возрастания `event_id`.
- Нет понятия одновременности — любые параллельные изменения сериализуются через присвоение ID.

**Генерация event_id:**
```rust
static COUNTER: AtomicU64 = AtomicU64::new(0);
let event_id = COUNTER.fetch_add(1, Ordering::SeqCst);
```

**Снапшоты:**
```rust
pub struct Snapshot {
    pub snapshot_id: u64,     // Последний применённый event_id
    pub state_hash: u64,
}
```

## 4. Входные данные UPO

UPO для каждого домена `d` (domain_id) активируется при наступлении события в этом домене (или по запросу). На момент вызова он получает:

- **Текущий `event_id`** (из обработчика событий).
- **Срез активных Token'ов** в домене `d` (state == Active).
- **Срез активных Connection'ов** в домене `d` (flags & ACTIVE != 0).

### 4.1 Вычисление агрегированных метрик домена

#### По Token'ам:
- Средняя скорость (вектор):
  ```
  avg_vel = (sum(velocity_i * mass_i) / sum(mass_i))   // покомпонентно
  ```
- Средняя норма скорости: `V = |avg_vel|` (нормируется адаптивно).
- Средняя кривизна траекторий: храним предыдущий `avg_vel` и вычисляем угол между ними → `C` (нормируется на π).
- Средняя знаковая температура:
  ```
  signed_temp_i = (valence_i.sign() as f32) * (temperature_i / 255.0)
  avg_temp = sum(signed_temp_i * mass_i) / sum(mass_i)
  ```
- Средняя масса: `avg_mass = sum(mass_i) / N` (нормируется /255).

#### По Connection'ам (опционально, включается флагом `use_connections`):
- Средний стресс: `avg_stress = sum(current_stress_i * strength_i) / sum(strength_i)`.
- Суммарная энергия связей: `total_stress_energy = sum(current_stress_i * strength_i)`.

#### Дополнительные метрики (для будущего):
- Разброс скоростей (температура скорости).
- Плотность Token'ов в пространстве.

## 5. Преобразование в координаты экрана (x, y, z)

Оси экрана нормированы в диапазон `[-1, 1]`. Предлагаемые формулы (коэффициенты настраиваются):

- **x (напряжение/активность):** комбинация скорости и кривизны  
  `x = tanh(α₁ * V_norm + α₂ * C_norm)`
- **y (стабильность):** инверсия скорости с учётом массы  
  `y = tanh(β * (1 - V_norm) * (1 + avg_mass_norm))`  
  или упрощённо: `y = 1 - V_norm` (с насыщением через tanh)
- **z (валентность/температура):** средняя знаковая температура  
  `z = tanh(γ * avg_temp)`

Если используются Connection'ы, можно модифицировать x, добавив вклад стресса:
```
x = tanh(α₁*V_norm + α₂*C_norm + α₃*stress_norm)
```

**Нормировка:**
- `V_norm = V / V_max_adaptive` (адаптивный максимум скорости)
- `C_norm = C / π`
- `stress_norm = avg_stress / STRESS_MAX` (константа или адаптивная)
- `avg_mass_norm = avg_mass / 255.0`

Все величины обрезаются до [0,1] перед подстановкой в tanh.

---

## 6. Вычисление веса (weight)

Вес определяет вклад текущей точки в накопленную интенсивность на экране. Он знаковый.

```
weight = avg_mass_norm * avg_temp * confidence
```

где `confidence` может зависеть от количества Token'ов (например, `confidence = min(1.0, N / N_min)`).  
Если используются Connection'ы, можно добавить энергию связей:
```
weight = (avg_mass_norm * avg_temp + δ * total_stress_norm) * confidence
```

## 7. Структура DynamicTrace (запись на экран)

Каждая запись имеет фиксированный размер **32 байта** с резервом.

```rust
#[repr(C)]
struct DynamicTrace {
    x: f32,          // 0-3   напряжение/активность
    y: f32,          // 4-7   стабильность
    z: f32,          // 8-11  валентность/температура
    weight: f32,     // 12-15 знаковый вклад
    event_id: u64,   // 16-23 причинный индекс (последнее событие, вызвавшее запись)
    _reserved: [u8; 8], // 24-31 зарезервировано
}
```

**Гарантии:**
- `x, y, z ∈ [-1, 1]`
- `weight ∈ [-1, 1]` (на практике может выходить за эти пределы, но экран аккумулирует, поэтому не критично)
- `event_id` соответствует моменту обработки событий, предшествующему записи.

## 8. Экран: организация, запись, затухание

### 8.1 Структура экрана

Экран — трёхмерный массив ячеек, каждая хранит текущую интенсивность (`f32`) и, опционально, `last_event_id` для ленивого затухания.

```rust
struct Screen {
    grid: Box<[[[f32; 256]; 256]; 256]>,   // интенсивность
    last_event: Box<[[[u64; 256]; 256]; 256]>, // event_id последнего обновления ячейки (опционально)
    decay: f32,                              // коэффициент затухания (например, 0.99)
    min_intensity: f32,                      // минимальная интенсивность (например, 0.001)
    current_event: u64,                       // последний обработанный event_id
}
```

Размерность 256³ выбрана как компромисс: 16.7 млн ячеек × 4 байта = 64 МБ. При необходимости масштабируется.

### 8.2 Запись на экран

При получении `DynamicTrace`:

```rust
fn write(screen: &mut Screen, trace: &DynamicTrace) {
    let ix = ((trace.x + 1.0) * 0.5 * 255.999) as usize;
    let iy = ((trace.y + 1.0) * 0.5 * 255.999) as usize;
    let iz = ((trace.z + 1.0) * 0.5 * 255.999) as usize;

    // Применить ленивое затухание к этой ячейке перед записью
    if screen.last_event[ix][iy][iz] < screen.current_event {
        let steps = screen.current_event - screen.last_event[ix][iy][iz];
        let decay_factor = screen.decay.powi(steps as i32);
        screen.grid[ix][iy][iz] *= decay_factor;
        if screen.grid[ix][iy][iz].abs() < screen.min_intensity {
            screen.grid[ix][iy][iz] = screen.min_intensity * screen.grid[ix][iy][iz].signum();
        }
    }

    // Добавить новый вклад
    screen.grid[ix][iy][iz] += trace.weight;
    screen.last_event[ix][iy][iz] = screen.current_event;
}
```

### 8.3 Затухание (ленивое)

Затухание применяется не глобально, а при каждом обращении к ячейке (как показано выше). Это эффективно, так как большинство ячеек могут долго не использоваться. При необходимости можно также запускать фоновый процесс, применяющий затухание ко всем ячейкам раз в N событий, но ленивый подход обычно достаточен.

**Важно:** даже после многократного затухания ячейка не становится нулевой — удерживается `min_intensity` с тем же знаком. Так система сохраняет память обо всех когда-либо произошедших состояниях.

### 8.4 Чтение с экрана

Другие модули (IntuitionEngine) могут сканировать экран, искать кластеры, анализировать распределение интенсивности. Для ускорения можно строить индексы (octree), но это выходит за рамки UPO.

## 9. Параметры конфигурации UPO

Каждый экземпляр UPO (на один домен) настраивается через структуру:

```rust
struct UPOConfig {
    domain_id: u16,
    update_mode: UpdateMode,        // OnEvent, Periodic(u64), OnDemand
    event_threshold: f32,            // мин. изменение для срабатывания (если OnEvent)
    smoothing: f32,                  // коэффициент сглаживания (EMA)
    alpha1: f32, alpha2: f32, alpha3: f32,  // веса для x
    beta: f32,                        // вес для y
    gamma: f32,                       // вес для z
    delta: f32,                        // вес для stress в weight
    use_connections: bool,
    min_tokens: usize,                 // мин. число Token'ов для confidence=1
    // ... другие параметры
}
```

## 10. Производительность и память

- **Вычисление метрик:** O(N + M), где N — активные Token'ы, M — активные Connection'ы.
- **Запись на экран:** O(1) с ленивым затуханием (несколько умножений).
- **Память экрана:** 64 МБ для 256³ (интенсивность) + опционально 64 МБ для `last_event` (если используется u64). Можно уменьшить, храня `last_event` как u32, если event_id умещается в 32 бита.
- **Снапшоты экрана:** копирование 64 МБ (редко).

## 11. Тестирование и валидация

### 11.1 Модульные тесты
- Фильтрация по domain_id и state.
- Вычисление средних скоростей и кривизны на синтетических данных.
- Преобразование координат (i16 → f32 → индекс).
- Затухание: проверка, что после многих шагов значение не падает ниже min_intensity.

### 11.2 Интеграционные тесты
- Подписка на события домена, проверка записи на экран.
- Воспроизведение последовательности событий из лога — детерминизм.
- Чтение экрана другим модулем, проверка наличия ожидаемых кластеров.

### 11.3 Граничные случаи
- Пустой домен (нет активных Token'ов) — UPO не пишет (или пишет с weight=0).
- Все Token'ы с нулевой скоростью — x=0, y=1, z по температуре.
- Максимальная скорость — насыщение x=1, y=-1.

## 12. Связь с другими модулями

### 12.1 IntuitionEngine
Анализирует экран, ищет паттерны (кластеры, повторяющиеся последовательности), сопоставляет их с архетипическими сюжетами (4 базовых, 7 Букер, и т.д.).

### 12.2 Библиотека следующего уровня
Экран может быть заморожен (прекращено затухание) и передан как входной массив для вышестоящего октаидера. В этом случае его ячейки интерпретируются как особого рода «токены» — узоры, которые могут сами участвовать в динамике.

### 12.3 CDNA (ограничения)
UPO не хранит персональные данные, не интерпретирует семантику, работает детерминированно. Все настройки должны быть доступны для аудита.

UPO v2.1 объединяет:
- Полную совместимость со структурами Token и Connection.
