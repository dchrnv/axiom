# Документация по модулям NeuroGraph

Этот документ представляет собой высокоуровневый обзор модулей NeuroGraph, их реализаций и ключевых методов. Он предназначен для помощи в миграции системы на новую версию.

## Общая архитектура

Система NeuroGraph — это гибридное приложение на Python/Rust, разработанное для высокопроизводительных пространственных вычислений на основе "токенов". Архитектуру можно кратко описать следующим образом:

- **Ядро (`core_rust`)**: Сердце системы — это библиотека на Rust (`core_rust`), реализующая сложную когнитивную архитектуру. Это ядро отвечает за все тяжелые вычисления, включая хранение данных, операции с графами и реализацию основных концепций, таких как Токены, Соединения и движок политик ADNA. Оно разработано для высокой производительности и доступно в Python через привязки PyO3.
- **Шлюз (`gateway`)**: Модуль на Python, который действует как основной сенсорный интерфейс к ядру. Он получает внешние сигналы (текст, аудио, системные метрики и т.д.), предварительно их обрабатывает, кодирует в 8D-семантическое векторное пространство и передает в ядро Rust.
- **API (`api`)**: Приложение на Python FastAPI, предоставляющее REST API для взаимодействия с системой NeuroGraph. Оно управляет аутентификацией, маршрутизацией и валидацией данных. Оно использует `gateway` для обработки входящих данных и взаимодействует с библиотекой `core_rust` (вероятно, через абстракцию хранилища) для управления базовыми данными.
- **Ядро Python (`core/token`)**: Реализация основной структуры данных `Token` на Python. Вероятно, используется для удобства и прототипирования в контекстах, где используется только Python, в то время как реализация на Rust используется для критически важных с точки зрения производительности операций.

## Детали модулей

### 1. `core_rust`

Это центральный и самый сложный модуль, написанный на Rust для максимальной производительности.

- **Расположение**: `src/core_rust/`
- **Точка входа**: `src/core_rust/src/lib.rs`
- **Ключевые концепции**:
    - `Token V2.0`: 64-байтовая атомарная единица информации с 8D-системой координат. Это фундаментальная структура данных.
    - `Connection V3.0`: 64-байтовая связь между токенами, позволяющая формировать графы.
    - `ADNA v3.0`: 256-байтовый "движок политик", который, вероятно, управляет поведением системы.
    - `ExperienceStream v2.1`: Событийная система памяти.
    - `Grid` и `Graph`: Структуры данных для организации и соединения токенов.
    - `Guardian`: Компонент для валидации данных и обработки событий.
- **Ключевые методы (доступные в Python)**:
    - Файл `lib.rs` реэкспортирует большое количество структур и функций из своих подмодулей. Привязки Python (`pub mod python;`) определяют точный API, доступный в Python. Судя по файлу, ключевые предоставляемые функции будут вращаться вокруг:
        - Создания, чтения и манипулирования объектами `Token` и `Connection`.
        - Построения и выполнения запросов к `Graph` и `Grid`.
        - Взаимодействия с движком политик `ADNA`.
        - Отправки событий в `ExperienceStream`.
        - Использования `Gateway` для обработки сигналов.

### 2. `core.token`

Реализация структуры данных `Token` на Python.

- **Расположение**: `src/core/token/`
- **Файл**: `token_v2.py`
- **Ключевой класс**: `Token`
- **Ключевые методы**:
    - `__init__(...)`: Создает новый экземпляр `Token`.
    - `set_coordinates(level, x, y, z)`: Устанавливает координаты для одного из 8 семантических уровней.
    - `get_coordinates(level)`: Получает координаты для заданного уровня.
    - `set_flag(flag)`, `clear_flag(flag)`, `has_flag(flag)`: Управляет 16-битными флагами токена.
    - `pack()`: Сериализует токен в 64-байтное бинарное представление.
    - `unpack(data)`: Десериализует токен из 64 байт.
    - `to_dict()`: Экспортирует данные токена в словарь Python.
- **Вспомогательные функции**:
    - `create_token_id(...)`, `extract_local_id(token_id)` и т.д.: для работы с 32-битным ID токена.

### 3. `gateway`

Модуль на Python, отвечающий за обработку входящих данных (сигналов) перед их отправкой в ядро Rust.

- **Расположение**: `src/gateway/`
- **Файл**: `gateway.py`
- **Ключевой класс**: `SignalGateway`
- **Ключевые методы**:
    - `__init__(core_system=None)`: Инициализирует шлюз, опционально подключая его к `core_system` (ядру Rust).
    - `initialize()`: Инициализирует шлюз и регистрирует встроенные сенсоры.
    - `push_text(text, ...)`: Основная точка входа для текстовых сигналов.
    - `push_audio(audio_data, ...)`: Основная точка входа для аудиосигналов.
    - `push_vision(image_data, ...)`: Основная точка входа для сигналов изображений.
    - `push_system(metric_name, ...)`: Основная точка входа для системных метрик.
- **Внутренние методы обработки**:
    - `_push_signal(...)`: Основной внутренний метод, который организует конвейер обработки.
    - `_normalize_data(...)`: Применяет предобработчики к сырым данным.
    - `_encode_to_vector(...)`: Кодирует данные в 8D-вектор с использованием настроенного кодировщика.
    - `_build_event(...)`: Создает конечный объект `SignalEvent`.
    - `_send_to_core(event)`: Метод-заполнитель для отправки события в ядро Rust.

### 4. `api`

Приложение FastAPI, которое предоставляет доступ к системе NeuroGraph через REST API.

- **Расположение**: `src/api/`
- **Точка входа**: `main.py`
- **Ключевые файлы**:
    - `main.py`: Настраивает приложение FastAPI, промежуточное ПО и маршрутизаторы.
    - `routers/tokens.py`: Определяет конечные точки CRUD для токенов.
    - Другие файлы маршрутизаторов (`grid.py`, `cdna.py`, `query.py` и т.д.) для других функций.
- **Ключевые конечные точки (из `tokens.py`)**:
    - `POST /api/v1/tokens`: Создает новый токен.
    - `GET /api/v1/tokens`: Перечисляет существующие токены.
    - `GET /api/v1/tokens/{token_id}`: Получает конкретный токен.
    - `PUT /api/v1/tokens/{token_id}`: Обновляет токен.
    - `DELETE /api/v1/tokens/{token_id}`: Удаляет токен.
- **Зависимости**:
    - API использует систему внедрения зависимостей (`Depends(...)`) для получения доступа к бэкендам хранилищ (`get_token_storage`) и для обработки аутентификации (`get_current_active_user`). Бэкенд хранилища является вероятной точкой интеграции с библиотекой `core_rust`.

## API-интерфейсы

Система NeuroGraph предоставляет два основных способа взаимодействия для внешних клиентов: многофункциональный **REST API** для выполнения запросов и управления, и высокопроизводительный **WebSocket API** для получения событий в реальном времени.

### REST API

REST API построен на базе FastAPI и предоставляет стандартизированный интерфейс для управления всеми основными ресурсами системы. Все эндпоинты, как правило, находятся под префиксом `/api/v1`.

#### Аутентификация

Большинство эндпоинтов требуют аутентификации. Доступ контролируется с помощью API-ключей и системы разрешений (scopes), например, `tokens:read` или `cdna:write`.

#### Основные ресурсы

**Токены (`/tokens`)**
-   `POST /tokens`: Создание нового токена.
-   `GET /tokens`: Получение списка токенов с пагинацией.
-   `GET /tokens/{token_id}`: Получение детальной информации о конкретном токене.
-   `PUT /tokens/{token_id}`: Обновление существующего токена.
-   `DELETE /tokens/{token_id}`: Удаление токена.
-   `DELETE /tokens/admin/clear`: (Только для администраторов) Удаление всех токенов.

**Пространственная сетка (`/grid`)**
-   `GET /grid/status`: Проверка доступности модуля Grid.
-   `POST /grid/create`: Создание нового экземпляра сетки для пространственного индексирования.
-   `POST /grid/{grid_id}/tokens/{token_id}`: Добавление токена в сетку.
-   `GET /grid/{grid_id}/neighbors/{token_id}`: Поиск "соседей" токена в заданном радиусе и координатном пространстве.
-   `GET /grid/{grid_id}/range`: Поиск всех токенов в определенной области.
-   `DELETE /grid/{grid_id}`: Удаление экземпляра сетки.

**Конституция (`/cdna`)**
-   `GET /cdna/status`: Получение текущего состояния CDNA, включая активный профиль и статус карантина.
-   `GET /cdna/profiles`: Получение списка всех доступных профилей (`Explorer`, `Creative` и т.д.).
-   `POST /cdna/profiles/{profile_id}/switch`: Переключение на другой профиль CDNA.
-   `PUT /cdna/config`: Обновление конфигурации CDNA, например, установка пользовательских весов для измерений.
-   `POST /cdna/quarantine/start`: Запуск режима "карантина" для безопасного тестирования новой конфигурации.
-   `POST /cdna/quarantine/stop`: Завершение режима "карантина" с применением или откатом изменений.
-   `GET /cdna/history`: Получение истории изменений конфигурации.

**Запросы (`/query`)**
-   `POST /query`: Выполнение семантического запроса к системе для поиска связанных токенов.

**Системные эндпоинты**
-   `/health`: Проверка работоспособности сервиса.
-   `/metrics`: Экспорт метрик в формате Prometheus.
-   `/status`: Получение общего статуса системы и ее компонентов.
-   `/api/v1/websocket/status`: Получение статуса WebSocket-сервера и количества подключенных клиентов.

### WebSocket API

WebSocket API предназначен для получения потоковых данных и событий от системы в реальном времени. Он использует модель подписок на каналы.

**Подключение**
-   Клиент устанавливает соединение с эндпоинтом `/ws`.

**Модель взаимодействия**
1.  После подключения клиент отправляет JSON-сообщение для подписки на интересующие его каналы.
    ```json
    {
      "type": "subscribe",
      "channels": ["metrics", "logs"]
    }
    ```
2.  Сервер начинает транслировать все события, происходящие в этих каналах, данному клиенту.
3.  Клиент может в любой момент отписаться от канала, отправив сообщение:
    ```json
    {
      "type": "unsubscribe",
      "channels": ["logs"]
    }
    ```

**Доступные каналы**
-   `metrics`: Трансляция системных метрик производительности (CPU, память, количество событий в секунду).
-   `signals`: События от `SignalSystem` в реальном времени, отражающие обработку входящих сигналов.
-   `actions`: Уведомления о выполненных системой действиях и их результатах.
-   `logs`: Системные логи для отладки и мониторинга.
-   `status`: Изменения в общем статусе системы (запуск, остановка, ошибки).
-   `connections`: Мета-канал, транслирующий события о подключении и отключении WebSocket-клиентов.

## Конституционная система: CDNA, Guardian и ADNA

Для обеспечения стабильности, безопасности и адаптивности в NeuroGraph реализована трехуровневая система управления, которую можно сравнить с государственным устройством. Она состоит из Конституции (CDNA), Судебной и Исполнительной власти (Guardian) и Правительства (ADNA).

### Уровень 1: `CDNA` (Конституция / Физика мира)

**CDNA (Cognitive DNA)** — это фундаментальный, редко изменяемый свод законов системы. Он определяет саму "физику" 8D-пространства и базовые правила поведения его обитателей (токенов и связей).

-   **Расположение**: `src/core_rust/src/cdna.rs`, `docs/adr/005-cdna-profiles.md`

-   **Ключевые концепции**:
    -   **Профили (Profiles)**: CDNA — это не один статичный документ, а система переключаемых "профилей" (`Explorer`, `Creative`, `Conservative`). Каждый профиль задает уникальный набор параметров. Администратор может сменить профиль в реальном времени через API, мгновенно изменяя "характер" всей системы.
    -   **Масштабирование измерений (Dimensional Scaling)**: Главная задача CDNA — определять веса (коэффициенты) для каждого из 8 измерений. Это меняет геометрию пространства: например, в профиле `Creative` больший вес придается абстрактным измерениям, поощряя неожиданные связи, а в профиле для пространственной навигации — первым трем (физическим) измерениям.
    -   **Конституционные параметры**: Профили CDNA также содержат такие параметры, как `plasticity` (пластичность) и `evolution_rate` (скорость эволюции), которые напрямую влияют на скорость обучения и адаптации ADNA.
    -   **Безопасность**: Для тестирования новых, потенциально нестабильных профилей CDNA предусмотрен **"Режим Карантина"**. В этом режиме система работает с новой конфигурацией ограниченное время, и если ключевые метрики (рост памяти, количество ошибок) выходят за пределы нормы, конфигурация автоматически откатывается.

### Уровень 2: `Guardian` (Верховный Суд и Полиция)

**Guardian** — это модуль-исполнитель, который обеспечивает соблюдение законов, установленных в CDNA. Он является центральным координатором и валидатором, гарантируя целостность и безопасность системы.

-   **Расположение**: `src/core_rust/src/guardian.rs`

-   **Ключевые функции**:
    -   **Хранитель Конституции**: `Guardian` владеет активным экземпляром `CDNA`, являясь единственным источником истины о текущих законах системы.
    -   **Полная Валидация ("Верховный Суд")**: При создании или изменении токенов и связей `Guardian` выполняет полную и строгую проверку по всем правилам из активного профиля CDNA (например, `min_token_weight`, `allowed_connection_types`).
    -   **Быстрая Валидация ("Полиция")**: Для "быстрого пути" принятия решений `Guardian` использует облегченный метод `validate_reflex`. Он не сверяется с CDNA (так как это медленно), а выполняет лишь несколько критически важных проверок безопасности (например, уверенность > 50%, связь не является гипотезой). Это прагматичная оптимизация для обеспечения сверхбыстрой реакции.
    -   **Контроль Ресурсов**: `Guardian` следит за системными ресурсами, применяя квоты на максимальное количество токенов и используемую память. Это конституционная функция, предотвращающая отказ системы из-за переполнения памяти (OOM).
    -   **История и Откаты**: `Guardian` ведет историю изменений CDNA и предоставляет функцию `rollback_cdna()` для отката к предыдущей стабильной версии конституции.

### Уровень 3: `ADNA` (Правительство)

**ADNA (Adaptive DNA)** — это адаптивный слой политик и эвристик. Он представляет собой "правительство", которое учится, развивается и принимает оперативные решения, но всегда действует в рамках законов, установленных "конституцией" (CDNA) и контролируемых "полицией" (Guardian).

-   **Расположение**: `src/core_rust/src/adna.rs`

-   **Роль в системе**:
    -   **Движок Рассуждений**: ADNA является ядром "медленного пути" принятия решений. Когда `ActionController` не находит подходящего рефлекса, он обращается к ADNA за `ActionPolicy` (политикой действий).
    -   **Обучение и Адаптация**: `IntuitionEngine` постоянно анализирует опыт системы и генерирует `Proposals` (предложения) по улучшению политик ADNA. Эти предложения, в свою очередь, также проходят валидацию у `Guardian` перед применением, гарантируя, что новые "законы" не нарушают "конституцию".

### Итог: Взаимодействие и Баланс

Трёхуровневая система создаёт надёжный баланс между скоростью, стабильностью и гибкостью:
1.  **CDNA** устанавливает фундаментальные, но настраиваемые "законы физики".
2.  **Guardian** неукоснительно следит за соблюдением этих законов, обеспечивая безопасность и стабильность всей системы.
3.  **ADNA** динамически адаптируется и обучается, находя оптимальные стратегии поведения *внутри* установленных `Guardian`-ом рамок.

## Система опыта (Experience System)

Система опыта — это не просто система логирования, а высокопроизводительная **in-memory база данных временных рядов**, которая служит кратковременной и среднесрочной памятью для всего NeuroGraph. Она является краеугольным камнем для механизмов обучения и адаптации, записывая все значимые события и предоставляя мощные инструменты для их анализа.

-   **Расположение**: `src/core_rust/src/experience_stream.rs`

### Ключевые компоненты

#### `ExperienceEvent`
Это фундаментальная единица записи, "воспоминание". Каждое событие представляет собой строго типизированную 128-байтную структуру, оптимизированную для размещения в кеше процессора. Она содержит исчерпывающую информацию о произошедшем:
-   ID события и временную метку.
-   Тип события (`TokenCreated`, `ActionCompleted` и т.д.).
-   8D-вектор `state` (состояние системы *до* события).
-   8D-вектор `action` (предпринятое действие).
-   Раздельные поля для наград: `reward_homeostasis`, `reward_curiosity` и др.

#### `HotBuffer` (Горячий буфер)
Сердце хранилища. Это **кольцевой буфер** в памяти, который хранит огромное количество `ExperienceEvent` (по умолчанию 1 миллион, что занимает ~128 МБ). Использование кольцевого буфера — это высокопроизводительный паттерн, который исключает дорогостоящие операции по перераспределению памяти: когда буфер заполняется, новые события просто начинают записываться поверх самых старых.

#### `ExperienceStream`
Главный управляющий компонент, который объединяет `HotBuffer` с системой трансляции событий и предоставляет другим модулям удобные интерфейсы для взаимодействия.

### Инструменты и процессы

Система опыта предоставляет набор "инструментов" (программных интерфейсов) для других модулей, которые позволяют ей быть центром цикла обучения.

#### 1. Запись событий (`ExperienceWriter`)
Любой модуль в системе (например, `ActionController`) может записывать события в `ExperienceStream`. Это делается через типаж `ExperienceWriter`. Когда `ActionController` выполняет действие, он записывает два события: `ActionStarted` и `ActionCompleted`, сохраняя состояние системы и результат действия.

#### 2. Асинхронная оценка наград
Уникальная особенность архитектуры — награды за событие могут выставляться *после* того, как оно произошло.
-   Событие записывается с нулевыми наградами.
-   Специализированные модули-"оценщики" (`Appraisers`) получают это событие, асинхронно анализируют его и вызывают метод `set_appraiser_reward()`, чтобы обновить соответствующее поле награды (`reward_homeostasis`, `reward_curiosity`) в уже существующей записи в буфере.
-   Это безопасно с точки зрения конкуренции, так как каждый "оценщик" пишет в свое собственное, выделенное поле в 128-байтной структуре `ExperienceEvent`.

#### 3. Интеллектуальная выборка (`sample_batch`)
Это главный инструмент, используемый `IntuitionEngine` для обучения. Вместо того чтобы считывать миллионы событий подряд, `IntuitionEngine` запраширует "выборку" наиболее интересных из них с помощью метода `sample_batch`, указывая одну из стратегий:
-   **`SamplingStrategy::Uniform`**: Случайная равномерная выборка.
-   **`SamplingStrategy::PrioritizedByReward`**: **Приоритетная выборка на основе награды**. Это ключевая стратегия для обучения. Она с большей вероятностью выберет события, которые привели к очень хорошим или очень плохим результатам, позволяя системе быстрее учиться на своих успехах и ошибках.
-   **`SamplingStrategy::RecencyWeighted`**: Предпочитает более "свежие" события.

Этот механизм интеллектуальной выборки позволяет модулю обучения эффективно фокусироваться на самых значимых данных, а не тратить ресурсы на анализ рутинных, неинформативных событий.

#### 4. Потоковая трансляция
Благодаря использованию `tokio::broadcast`, `ExperienceStream` может транслировать каждое записанное событие всем подписанным компонентам в реальном времени. Это позволяет, например, дашборду для мониторинга отображать "живую" ленту событий, происходящих внутри системы.

### Заключение
Система Опыта — это больше, чем лог. Это высокопроизводительный механизм памяти, который записывает, распространяет и, самое главное, **предоставляет инструменты для интеллектуального анализа** этого опыта, что делает его центральным элементом адаптивности и обучения всей системы NeuroGraph.

## Движок интуиции (IntuitionEngine)

**Движок интуиции** — это, возможно, самый важный компонент в ядре `core_rust`, отвечающий за обучение и принятие решений. Его можно рассматривать как мозг системы, который преобразует сырой опыт в интеллект. `IntuitionEngine` реализует гибридную двухпроцессную модель, сочетая быстрые, интуитивные реакции с медленным, но глубоким аналитическим обучением.

-   **Расположение**: `src/core_rust/src/intuition_engine.rs`

### Двухпроцессная модель: Система 1 и Система 2

`IntuitionEngine` является прямой реализацией двухпроцессной теории мышления.

#### Система 1: Быстрый путь (Рефлексы)
Это путь мгновенных, почти бессознательных реакций. Когда `ActionController` запрашивает действие, он сначала обращается к "быстрому пути" `IntuitionEngine`.
-   **Механизм**: `IntuitionEngine` берет текущее состояние системы (8D-вектор), вычисляет для него **пространственный хеш** и мгновенно ищет этот хеш в своей `AssociativeMemory` (ассоциативной памяти).
-   **Результат**: Если совпадение найдено, он немедленно возвращает связанное с ним действие (`ConnectionV3`) в виде готового рефлекса.
-   **Производительность**: Весь процесс занимает **~30-50 наносекунд**, что позволяет системе реагировать на знакомые ситуации практически в реальном времени.

#### Система 2: Медленный путь (Анализ и Обучение)
Это путь обдуманного, фонового анализа, который работает не для сиюминутной реакции, а для извлечения знаний из прошлого опыта. Он выполняется в виде фонового процесса (`run_analysis_cycle`).

### Процесс обучения на "медленном" пути

"Медленный путь" — это непрерывный цикл анализа данных из `ExperienceStream` для выявления новых закономерностей.

1.  **Интеллектуальная выборка**: Периодически `IntuitionEngine` запрашивает у `ExperienceStream` выборку "интересных" событий, используя стратегию `PrioritizedByReward`, чтобы сфокусироваться на событиях с самыми значимыми последствиями (как положительными, так и отрицательными).

2.  **Квантование и анализ**: Непрерывные 8D-состояния из выборки квантуются (группируются) в дискретные "контейнеры" (state bins). Затем движок анализирует, какие действия (`action`) в пределах одного и того же "контейнера" состояний (`state`) приводят к лучшим наградам (`reward`).

3.  **Поиск паттернов**: С помощью статистического анализа система ищет закономерности вида: "В ситуациях типа **А** действие **Х** стабильно приводит к лучшим результатам, чем действие **Y**".

4.  **Генерация предложений (`Proposal`)**: Для каждой найденной и статистически подтвержденной закономерности `IntuitionEngine` генерирует `Proposal` — формальное предложение об изменении политики поведения системы. Это предложение отправляется в `EvolutionManager` для возможного применения к `ADNA`.

### Механизм консолидации рефлексов

Ключевая задача `IntuitionEngine` — замыкать цикл обучения, превращая знания, полученные "медленным путем", в быстрые реакции.

-   **Механизм**: Когда в результате анализа `ADNA` или других процессов формируется надежная и проверенная связь "состояние-действие", вызывается метод `consolidate_reflex()`.
-   **Действие**: Этот метод берет данную связь (`ConnectionV3`) и добавляет ее в `AssociativeMemory` "быстрого пути".
-   **Эффект**: В следующий раз, когда система столкнется с аналогичной ситуацией, ей больше не нужно будет тратить миллисекунды на медленные рассуждения `ADNA`. `IntuitionEngine` мгновенно найдет эту связь как готовый рефлекс и отреагирует за наносекунды. Метод `try_auto_consolidate` позволяет делать это автоматически для связей, которые достигли высокого уровня уверенности и были проверены `Guardian`.

### Взаимодействие с другими модулями

-   **`ActionController`**: Является основным "клиентом" `IntuitionEngine`. При каждом запросе на действие он сначала вызывает `try_fast_path()`, чтобы получить быстрый интуитивный ответ.
-   **`ExperienceStream`**: Служит "источником пищи" для `IntuitionEngine`, поставляя данные для анализа и обучения.
-   **`ADNA` / `EvolutionManager`**: Являются "получателями" знаний, которые генерирует `IntuitionEngine` в виде `Proposals`.
-   **`Guardian`**: Используется для проверки безопасности и надежности связей перед их "консолидацией" в рефлексы.

## Система принятия решений: Рефлексы и Рассуждение (Двухпроцессная модель)

Архитектура NeuroGraph v3.0 включает в себя сложную двухпроцессную систему принятия решений, которая имитирует Систему 1 (быстрое, интуитивное мышление) и Систему 2 (медленное, аналитическое мышление) человеческого познания. Это разделение позволяет системе реагировать на знакомые ситуации почти мгновенно (рефлекторно), при этом сохраняя способность к глубокому анализу для новых или сложных сценариев.

### Ключевые компоненты

-   **`ActionController` (Арбитр)**: Расположенный в `src/core_rust/src/action_controller.rs`, этот модуль является центральным координатором принятия решений. Он получает текущее состояние системы и решает, какой путь использовать: быстрый (рефлекс) или медленный (рассуждение).

-   **`IntuitionEngine` (Движок интуиции)**: Находится в `src/core_rust/src/intuition_engine.rs`. Этот компонент реализует как быстрый, так и медленный пути.
    -   **Быстрый путь (Рефлекторный уровень)**: Использует пространственное хеширование (`AssociativeMemory`) для сверхбыстрого поиска (30-50 нс) заранее "консолидированного" действия (рефлекса), соответствующего текущему состоянию.
    -   **Медленный путь (Аналитический уровень)**: В фоновом режиме анализирует прошлый опыт (`ExperienceStream`), находит статистически значимые закономерности и предлагает новые правила или действия.

-   **`Guardian` (Страж)**: Компонент, отвечающий за безопасность и соблюдение правил. Прежде чем выполнить рефлекторное действие, `ActionController` запрашивает у `Guardian` валидацию (`validate_reflex`). Это предотвращает выполнение потенциально опасных или нелогичных быстрых решений.

-   **`ADNA` (Аналитическое ядро)**: Представляет собой базу знаний и движок рассуждений. Когда быстрый путь недоступен или нецелесообразен, `ActionController` обращается к `ADNA` для получения взвешенной политики действий, основанной на глубоком анализе.

### Процесс принятия решения

Поток принятия решений, управляемый методом `act()` в `ActionController`, выглядит следующим образом:

1.  **Получение состояния**: `ActionController` получает на вход текущее состояние системы в виде 8D-вектора.

2.  **Попытка быстрого пути**:
    -   Контроллер немедленно обращается к `IntuitionEngine` с запросом `try_fast_path()`.
    -   `IntuitionEngine` вычисляет хеш состояния и ищет в своей ассоциативной памяти (`associative_memory`) соответствующую `ConnectionV3` (заранее сохраненный рефлекс).

3.  **Проверка и валидация рефлекса**:
    -   **Если рефлекс найден**, `ActionController` выполняет две проверки:
        1.  **Проверка уверенности**: Уверенность (`confidence`) рефлекса сравнивается с пороговым значением `reflex_confidence_threshold` (настраивается в `ArbiterConfig`, по умолчанию ~78%).
        2.  **Проверка Стражем**: Если уверенность достаточна, `Guardian` проверяет рефлекс на соответствие внутренним правилам и ограничениям (`validate_reflex`).
    -   **Если обе проверки пройдены**, `ActionController` принимает решение **использовать рефлекс**. Формируется `ActionIntent` (намерение к действию) на основе данных из рефлекса, и процесс завершается. Это **рефлекторный путь**.

4.  **Переход к медленному пути**:
    -   Если рефлекс не найден, его уверенность слишком низка или `Guardian` его отклонил, `ActionController` переходит к **пути рассуждения**.
    -   Он вызывает метод `act_slow_path()`, который обращается к `ADNA`.
    -   `ADNA` анализирует состояние и возвращает `ActionPolicy` — взвешенный список возможных действий.
    -   `ActionController` выбирает лучшее действие из политики и формирует `ActionIntent`. Это занимает значительно больше времени (~1-10 мс), но позволяет принять решение в новой ситуации.

5.  **Отказоустойчивость**: Если оба пути по какой-либо причине терпят неудачу, система возвращает безопасное действие по умолчанию (например, "сохранить состояние").

### Цикл обучения: от рассуждения к рефлексу

Ключевой особенностью архитектуры является постоянное самосовершенствование. Медленный путь не только решает текущую задачу, но и обучает быстрый путь:

-   `IntuitionEngine` в фоновом режиме (`run_analysis_cycle`) анализирует поток опыта, выявляя, какие действия в каких состояниях стабильно приводят к хорошим результатам.
-   Когда обнаруживается сильная закономерность, `IntuitionEngine` использует метод `consolidate_reflex()`.
-   Этот метод создает новую `ConnectionV3`, представляющую успешную пару "состояние-действие", и добавляет ее в ассоциативную память быстрого пути.

Таким образом, решение, на которое раньше требовались миллисекунды аналитической работы `ADNA`, становится рефлексом, доступным за наносекунды. Это классический цикл "осознанная некомпетентность -> осознанная компетентность -> неосознанная компетентность", реализованный в архитектуре системы.

## Эффективное сжатие: трехуровневая стратегия

Одной из ключевых архитектурных особенностей NeuroGraph является многоуровневый подход к "сжатию". Это не единый алгоритм, а комплексная стратегия, направленная на повышение производительности, уменьшение потребления памяти и оптимизацию сетевого трафика. Стратегия включает три различных уровня сжатия.

### Уровень 1: Семантическое сжатие (в Gateway)

На этом уровне происходит "сжатие смысла". Его основная задача — преобразовать сырые, высокоразмерные и разнородные входные данные (например, длинный текст, аудиопоток, системные метрики) в компактное, но осмысленное представление — плотный 8D-вектор.

-   **Компонент**: `Gateway` (`src/gateway/`)
-   **Механизм**: Различные **Кодеры (`Encoders`)** из `src/gateway/encoders/`.
    -   `TextTfidfEncoder`: Преобразует текст в вектор на основе частоты слов (TF-IDF).
    -   `SentimentSimpleEncoder`: Анализирует тональность текста и представляет ее в виде вектора.
    -   `NumericDirectEncoder`: Нормализует и отображает числовые значения в 8D-пространство.
-   **Тип сжатия**: С потерями. Исходные данные невозможно восстановить из 8D-вектора, но вектор сохраняет их ключевые семантические характеристики, необходимые для дальнейшей работы системы.
-   **Цель**: Унификация всех входящих данных в единый формат (`Token`) и радикальное уменьшение их размерности для эффективной обработки ядром на Rust.

### Уровень 2: Структурное сжатие (в `core_rust`)

На этом уровне происходит сжатие представления самих данных в памяти. Фундаментальные единицы системы, `Token` и `Connection`, имеют строго фиксированный размер в **64 байта**. Это достигается за счет низкоуровневой упаковки битов.

-   **Компонент**: Ядро (`core_rust`) и его Python-представление (`src/core/token/token_v2.py`).
-   **Механизм**: Использование техник побитовой упаковки, аналогичных `struct.pack` в Python. В файле `token_v2.py` это наглядно демонстрируется форматной строкой `'<24h I H f 2B I'`, которая описывает, как множество полей (координаты, ID, флаги, вес и т.д.) плотно упаковываются в 64-байтную последовательность.
-   **Тип сжатия**: Без потерь (относительно структуры). Все поля сохраняют свои точные значения.
-   **Цель**: Максимально эффективное использование памяти и повышение производительности CPU. Фиксированный и предсказуемый размер структур позволяет:
    -   Уменьшить общий объем занимаемой памяти.
    -   Эффективно использовать кеш процессора (cache locality).
    -   Выполнять быстрые, пакетные операции над памятью.

### Уровень 3: Сжатие данных при передаче (в API)

Этот уровень отвечает за оптимизацию сетевого трафика, в основном при обмене данными через WebSockets.

-   **Компонент**: WebSocket менеджер (`src/api/websocket/compression.py`).
-   **Механизм**: Применение стандартных алгоритмов сжатия данных, таких как **GZIP**, **ZLIB** и **DEFLATE**, к сообщениям перед их отправкой по сети. Система также достаточно умна, чтобы не сжимать маленькие сообщения, для которых накладные расходы на сжатие могут превысить выгоду (используется `min_size_threshold`).
-   **Тип сжатия**: Без потерь. Клиент получает и распаковывает данные в их исходном виде.
-   **Цель**: Уменьшение объема передаваемого трафика, что особенно важно для клиентов с медленным интернет-соединением или при передаче больших объемов данных (например, при трансляции событий из `Grid`).

### Итог

Эти три уровня сжатия работают синергетически, делая систему эффективной на всех этапах: от первоначальной обработки сырых данных до их хранения в памяти и передачи клиентам. Это позволяет NeuroGraph обрабатывать большие объемы информации, сохраняя при этом высокую производительность и низкие требования к ресурсам.

## Оптимизация производительности на стыке Rust и Python

Создание гибридных систем на Python и Rust требует особого внимания к производительности, так как вызовы между двумя языками (через Foreign Function Interface, FFI) могут быть дорогостоящими. В NeuroGraph применяется продуманная стратегия для минимизации этих накладных расходов, позволяя использовать сильные стороны каждого языка: гибкость и простоту Python для высокоуровневой логики и непревзойденную скорость Rust для вычислений.

Эта стратегия основана на четырех ключевых принципах.

### Принцип 1: Четкое разделение труда (Python-дирижер, Rust-оркестр)

Вместо того чтобы смешивать логику, в NeuroGraph принято строгое архитектурное разделение:
-   **Python** используется для задач высокого уровня, где скорость не так критична: управление API-сервером (FastAPI), оркестрация процессов, конфигурация и бизнес-логика.
-   **Rust** (`core_rust`) используется как "вычислительный движок". Вся работа, требующая высокой производительности — операции с графами, пространственное хеширование, работа с памятью, выполнение алгоритмов (`IntuitionEngine`) — полностью делегируется ядру на Rust.

Python выступает в роли "дирижера", который отдает команды, а Rust — в роли "оркестра", который эффективно их исполняет.

### Принцип 2: "Крупные" вызовы вместо "болтливых" (Chunky vs. Chatty FFI)

Каждый вызов из Python в Rust несет в себе накладные расходы на преобразование типов и пересечение границы FFI. Множество мелких, "болтливых" вызовов в цикле могут быстро свести на нет все преимущества Rust.

Архитектура NeuroGraph спроектирована так, чтобы минимизировать количество таких переходов, предпочитая единичные, но "крупные" вызовы.

**Яркий пример — `Token.create_batch()`:**
-   **Плохой ("болтливый") подход**: `tokens = [Token(i) for i in range(1_000_000)]`. Здесь происходит миллион отдельных вызовов конструктора `Token` через FFI.
-   **Хороший ("крупный") подход**: `tokens = Token.create_batch(1_000_000)`. Здесь происходит **один** вызов. Python просит Rust "создай миллион токенов", и вся работа по выделению памяти и инициализации происходит на стороне Rust, в одном эффективном цикле.

Как указано в комментариях к коду, такой подход обеспечивает **4-кратное ускорение** для пакетных операций.

### Принцип 3: Освобождение Global Interpreter Lock (GIL)

Global Interpreter Lock (GIL) в CPython — это механизм, который позволяет только одному потоку выполнять Python-байткод в один момент времени, что является серьезным ограничением для параллельных вычислений.

Для решения этой проблемы в `core_rust` повсеместно используется механизм `PyO3` для освобождения GIL во время длительных операций.

-   **Механизм**: Ключевые ресурсоемкие функции в Rust-коде, доступные из Python, обернуты в `py.allow_threads(|| { ... })`.
-   **Примеры**: `bootstrap()`, `query()`, `create_batch()`.
-   **Эффект**: Как только управление передается в Rust, он "отпускает" GIL. Это позволяет другим Python-потокам (например, event loop веб-сервера или фоновым задачам) продолжать работу, пока Rust выполняет тяжелые вычисления. Благодаря этому приложение в целом остается отзывчивым и не "замирает" во время обработки запросов.

### Принцип 4: Минимизация передачи данных

Проектировщики API на стыке языков избегали ненужного копирования больших объемов данных. Вместо того чтобы Python читал большой файл с эмбеддингами в память и затем передавал его в Rust, метод `bootstrap()` просто получает путь к файлу, а всю работу по чтению и парсингу выполняет на стороне Rust.

Аналогично, при выполнении запроса (`query()`) Python передает лишь короткую строку, а обратно получает только небольшой, финальный набор результатов, а не все промежуточные данные.

### Заключение

Сочетание этих четырех принципов — **разделение труда, крупные вызовы, освобождение GIL и минимизация передачи данных** — позволяет NeuroGraph эффективно использовать мощь Rust для вычислений, не жертвуя при этом удобством и скоростью разработки на Python для высокоуровневой логики. Это создает отзывчивую и высокопроизводительную систему, где каждый язык используется для тех задач, в которых он силен.

## Потенциальные узкие места и будущие направления (предположения)

Этот раздел содержит предположения о потенциальных узких местах, основанные на анализе архитектуры. Они могут служить направлениями для будущих исследований и оптимизаций.

-   **"Медленный путь" как ограничитель пиковой производительности.** Хотя "медленный путь" принятия решений (1-10 мс) является мощным аналитическим инструментом, при большом потоке принципиально новых для системы событий общая производительность может быть ограничена именно его скоростью. Масштабируемость и производительность самого ядра `ADNA` под пиковой нагрузкой требуют дополнительного исследования.

-   **Централизованный `Guardian` (Страж).** `Guardian` выполняет роль центрального валидатора, что обеспечивает целостность системы. Однако в архитектуре с очень высокой степенью параллелизма централизованная точка контроля теоретически может стать узким местом. Возможно, в будущем потребуются механизмы шардирования или распределенной валидации.

-   **Фиксированный размер `ExperienceStream`.** Кольцевой буфер `HotBuffer` эффективно решает проблему роста памяти. Однако его фиксированный размер (~128 МБ) означает, что система может "забывать" старые события. Это может помешать обучению на долгосрочных паттернах, выходящих за пределы объема буфера. Определение оптимального размера буфера или внедрение многоуровневой системы памяти (горячей/холодной) является важным направлением для развития.

-   **Производительность `Gateway` (Шлюза).** Компонент `Gateway`, работающий на Python, выполняет "семантическое сжатие" всех входящих данных. Сложность алгоритмов кодирования может сделать его узким местом по CPU при высокой интенсивности входящих сигналов. Профилирование производительности шлюза под разными типами нагрузки поможет выявить потенциальные проблемы.

-   **Конкуренция за ресурсы фонового процесса обучения.** Процесс обучения "медленного пути" (`run_analysis_cycle`) работает в фоновом режиме. Если он окажется слишком ресурсоемким, он может конкурировать за ресурсы CPU с "горячим" путем обработки запросов. Внедрение механизмов управления приоритетами или троттлинга для фоновых задач может быть необходимо для гарантии стабильной отзывчивости системы.
