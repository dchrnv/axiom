# AXIOM — Performance Notes

Внутренний журнал: причины регрессий, диагностические находки, открытые вопросы.
Не публичный. Дополнять по мере появления новых данных.

---

## Известные источники шума в бенчмарках

**Машина:** AMD Ryzen 5 3500U, 5.7 GiB RAM, Linux (личная машина, нет изоляции).
Дельты между версиями ненадёжны — замеряют "погоду" системы так же как и код.
Доверять только: стабильным серийным замерам (1M тиков), изолированным бенчам (engine снаружи closure).

**iter_batched с AxiomEngine::new** — паттерн который искажает всё что внутри него.
`AxiomEngine::new` стоит ~1.34 ms (v9) — если он в setup closure, он доминирует над
любой измеряемой операцией быстрее ~100 µs.
Затронутые бенчи: `run_adaptation`, `snapshot_and_prune`, `horizon_gc`, `run_horizon_gc`.

## Открытые регрессии

### ~~[P1]~~ AxiomEngine::new — исправлено

**Было (v9):** 1.34 ms (+205% от v6 baseline ~440 µs)

Две причины, обе исправлены:
1. `rayon::ThreadPool` — вынесен в `OnceLock` (`get_shared_pool`), создаётся один раз.
2. `SemanticContributionTable::default_ashti_core()` — теперь возвращает `&'static Self`
   через `OnceLock`; `Domain` хранит `&'static SemanticContributionTable` (не копию 2KB).

**Следствие для бенчей:** `iter_batched(|| engine_with_experience(...), ...)` теперь не
искажает результаты — `AxiomEngine::new` больше не доминирует над быстрыми операциями.
Реальные данные после исправления не замерялись — нужен прогон.

### [P2] compute_shell — 8x в shell_bench v9

**Симптом:** v7: 197 ns → v9: 1.55 µs (100 связей).

**Установленная причина:** предположительно cache miss из-за порядка прогонов.
В v9 `domain_detail_snapshot` bench (новый) запускается раньше shell_bench и вызывает
`compute_shell` с реальными данными — вытесняет горячие данные из L1/L2.
При следующем замере shell_bench начинается с холодного cache.

**Код не изменился:** `compute_shell` в shell_bench и в `from_token_with_connections` —
одна и та же функция, path не менялся.

**Результат изолированного прогона (v9):**

| | v7 full | v9 full | v9 isolated |
|--|--|--|--|
| connections/100 | 197 ns | 1.55 µs | 558 ns (386–742) |

- Full → isolated: 1.55 µs → 558 ns — cache thrashing подтверждён (~3x фактор)
- Isolated vs v7: 197 ns → 558 ns — остаточная регрессия, необъяснённая
- Высокая дисперсия [386, 742] ns — вероятно шум машины (личная, без изоляции)

**Выводы:**
1. Cache thrashing реален — порядок бенчей влияет на результат
2. Реальная стоимость `compute_shell(100 conn)` ≈ 400–560 ns, не 197 ns и не 1.55 µs
3. v7 197 ns мог быть артефактом горячего кэша или иной загрузки системы

**Статус:** подтверждено частично. Достоверный baseline: ~400–560 ns для 100 связей.

## axiom-neural — что это и как устроено (2026-06-12)

### Зачем

Советники NeuralAdvisor (их 5 штук) сейчас работают по жёстким правилам.
Мы заменяем их маленькими нейронными сетями которые сами учатся на истории AXIOM.
Пилот — советник ReactivationDepth (следит за тем какие октанты надо реактивировать).

### Как устроен крейт axiom-neural

Файлы в `crates/axiom-neural/src/`:
- `model.rs` — общие типы: AdvisorInput (что подаём на вход), AdvisorOutput (что получаем),
  трейт Model (любая модель советника обязана его реализовать)
- `layers.rs` — слои сети: Conv1D (свёртка), Linear (полносвязный), ReLU/Sigmoid (активации),
  GlobalAvgPool (схлопывает временную ось в один вектор)
- `fft.rs` — FFT: берёт историю активности системы (16/64/256 прошлых состояний)
  и превращает в частотный спектр — "какие ритмы есть у системы"
- `normalize.rs` — нормализация входных данных (Z-score)
- `calibration.rs` — калибратор уверенности: учится понимать насколько модель
  реально права когда говорит "я уверен на 90%"
- `weights.rs` — загрузка/сохранение весов из .bin файла (bincode)
- `reactivation_depth.rs` — сама пилотная модель

### Как работает пилотная модель (ReactivationDepth)

Вход: история активности 9 подсистем за 16/64/256 прошлых тиков
→ FFT каждого кольца → частотные признаки (171 число на подсистему, итого 1539)
→ Z-score нормализация
→ Conv1D (32 канала, ядро 3) → ReLU
→ Conv1D (64 канала, ядро 5) → ReLU
→ GlobalAvgPool → 64 числа
→ Linear(64→32) → ReLU
→ Linear(32→8) = depth[8] (насколько каждый октант нуждается в реактивации)
→ Linear(32→1) + Sigmoid = уверенность (0..1)

Параметры: ~13K (очень маленькая модель, помещается в L1-кэш).
Работает только раз в 11 тиков (t%11). Если тормозит — fallback на старые правила.

### Что сейчас НЕ сделано (следующие шаги)

1. **Конфиг архитектуры** — сейчас CH1=32, CH2=64 захардкожены. Надо добавить
   в genome.yaml секцию neural_advisor.reactivation_depth.arch чтобы можно было
   менять размер модели без перекомпиляции.

2. **Генерация данных** — чтобы обучить модель, нужны примеры. Запускаем OBS,
   каждые N тиков сохраняем срез Sensorium + что ответил rule-based советник.
   Это и есть тренировочная выборка (дистилляция: нейронка учится делать то же что правила).

3. **Тренировка** — offline (вне AXIOM), сохраняем веса в .bin.

4. **Подключение** — когда веса обучены, NeuralAdvisor грузит модель и запускает
   параллельно с правилами. Сначала в режиме Ignore (только смотрим), потом
   RequireConfirmation (просит подтверждения у chrnv), потом AutoApply.

### Охранная цифра производительности

TickForward горячий путь = 24.8 µs. После подключения нейронки НЕЛЬЗЯ его поднять.
Проверять через `cargo bench -p axiom-bench --bench engine_bench`.
