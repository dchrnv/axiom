## 1. Что такое примитив

### 1.1 Рабочее определение

Примитив — это единица которую система **не может разложить дальше** в текущей конфигурации своих перцепторов.

Не "абсолютный атом бытия". А **дно данного канала восприятия**.

```
VisionPerceptor (базовый):   "stroke" — примитив
VisionPerceptor (глубокий):  "stroke" разлагается на [direction, length, curvature]
                              → "stroke" стал C1_Atom, direction/length/curvature — новые C0
```

Это означает: примитивность **относительна** и **может меняться** при углублении разрешения перцептора.

### 1.2 Три признака примитива в AXIOM

1. **Нет composition bonds** — `composed_of` пуст или отсутствует
2. **SutraDepth = 65535** во всех октантах при загрузке из yaml
3. **STATE_LOCKED** — Guardian запрещает модификацию

Если хотя бы один признак не выполнен — это не C0_Primitive, это минимум C1_Atom.

### 1.3 Примитив не означает "простой"

Примитив "CAUSE_PRECEDES" концептуально сложнее чем примитив "visual_stroke_horizontal". Это не мешает обоим быть C0.

Сложность — свойство семантики. Атомарность — свойство структуры хранения.

## 2. Принцип относительного дна

### 2.1 Дно определяется перцептором

Каждый Perceptor задаёт свой минимальный гранулярный уровень:

| Perceptor | Текущее L0 | Возможное глубже |
|-----------|-----------|-----------------|
| TextPerceptor | word-level tokens | morpheme, phoneme, grapheme |
| VisionPerceptor (план) | edge, blob, corner | oriented_gradient, frequency_band |
| AudioPerceptor (план) | onset, tone, rhythm | spectral_bin, micro_timing |
| BodyPerceptor (план) | joint_state, effort | muscle_activation, pressure_point |

### 2.2 Следствие: универсальный дизайн примитивов

При проектировании L0 yaml нужно спрашивать: **"что возвращает Perceptor, а не что понятно человеку"**.

`writing_symbol` — плохой L0 (это уже интерпретация).  
`visual_closed_curve` — лучше (это то что детектирует перцептор).  
`visual_stroke` — ещё лучше.

## 3. Пять видов примитивов

### 3.1 Сенсорные примитивы

Прямо связаны с каналом восприятия. Возникают на выходе Perceptor-а.

```yaml
# Примеры
- visual_stroke_horizontal
- visual_wedge_45deg
- audio_onset
- audio_frequency_high
- body_joint_flexion
- body_contact_pressure
```

**Ключевое свойство:** существуют только пока есть входной сигнал. Без активации Perceptor-а — не возникают.

### 3.2 Реляционные примитивы

Описывают **отношение между вещами**, а не сами вещи. Не сигнал — связь между сигналами.

```yaml
- spatial_above         # A находится над B
- spatial_left_of       # A левее B
- spatial_inside        # A внутри B
- temporal_before       # A предшествует B
- temporal_simultaneous # A и B одновременны
- similarity_same       # A и B одинаковы
- similarity_different  # A и B различны
```

**Архитектурный нюанс:** реляционный примитив всегда требует **двух аргументов**. В SUTRA он живёт как токен, но семантически указывает на пару. Это создаёт особую проблему хранения — см. §6.

### 3.3 Каузальные примитивы

Особый подвид реляционных — про **причину и следствие во времени**.

```yaml
- causal_precedes       # A случается → B случается после
- causal_contact        # физический контакт → передача состояния
- causal_force_applied  # сила → изменение движения
- object_persists       # объект продолжает существовать
- object_disappears     # нарушение персистентности
- state_changes         # состояние объекта изменилось
```

Каузальные примитивы — основа **интуитивной физики**. Без них система не может строить предсказания о мире.

### 3.4 Оценочные примитивы

Субъективная окраска опыта. Не о мире — о **том как мир ощущается**.

```yaml
- valence_pleasant      # приятно
- valence_painful       # болезненно
- valence_dangerous     # опасность
- valence_safe          # безопасность
- urgency_high          # срочно
- urgency_low           # не срочно
```

**Нюанс:** оценочные примитивы — это то что связывает восприятие с мотивацией. Без них система наблюдает мир, но не реагирует на него. Они — мост к подсистемам Values и Ethics.

### 3.5 Модальные примитивы

О **состояниях ума** — своего или чужого. Основа прагматики и социального понимания.

```yaml
- modal_believes        # агент считает что X истинно
- modal_intends         # агент намерен сделать X
- modal_knows           # агент знает X
- modal_desires         # агент хочет X
- modal_perceives       # агент воспринимает X
```

**Нюанс:** модальные примитивы появляются **позже всех** в развитии системы — они требуют что система уже умеет моделировать агента отдельно от мира.

## 4. Универсальные примитивы

### 4.1 Пять кандидатов на абсолютное дно

Эти примитивы встречаются в каждом языке мира, в каждой теории когниции, в каждом когнитивном слое. Они кандидаты на то чтобы быть **поперёк всех модальностей** — не принадлежать ни одному Perceptor-у, но использоваться всеми.

| Примитив | Смысл | Присутствует в |
|----------|-------|----------------|
| `SAME / DIFFERENT` | различение | любое сравнение |
| `MORE / LESS` | величина | числа, интенсивность, размер |
| `BEFORE / AFTER` | порядок | время, последовательность, причина |
| `PART / WHOLE` | состав | любая структура |
| `CAUSE / EFFECT` | каузальность | физика, логика, нарратив |

### 4.2 Особый статус в SUTRA

Универсальные примитивы живут **глубже всех остальных**:

```
SutraDepth = 65535  →  обычные примитивы из yaml
SutraDepth = 65535  +  special_flag: universal = true  →  универсальные примитивы
```

Они никогда не становятся C1. Они не могут быть разложены даже при понижении дна (§7). Они — терминальный уровень.

### 4.3 Они живут в axiom-experience, не в yaml

Не загружаются из файлов. Инициализируются кодом при старте системы. Неизменяемы на уровне Guardian: `Permission::Forbidden` для любых изменений.

```rust
// axiom-experience/src/universal_primitives.rs
pub const UNIVERSAL_SAME:   SutraId = SutraId(0xFFFF_0001);
pub const UNIVERSAL_DIFF:   SutraId = SutraId(0xFFFF_0002);
pub const UNIVERSAL_MORE:   SutraId = SutraId(0xFFFF_0003);
pub const UNIVERSAL_LESS:   SutraId = SutraId(0xFFFF_0004);
pub const UNIVERSAL_BEFORE: SutraId = SutraId(0xFFFF_0005);
pub const UNIVERSAL_AFTER:  SutraId = SutraId(0xFFFF_0006);
pub const UNIVERSAL_PART:   SutraId = SutraId(0xFFFF_0007);
pub const UNIVERSAL_WHOLE:  SutraId = SutraId(0xFFFF_0008);
pub const UNIVERSAL_CAUSE:  SutraId = SutraId(0xFFFF_0009);
pub const UNIVERSAL_EFFECT: SutraId = SutraId(0xFFFF_000A);
```

## 5. Иерархия уровней абстракции

### 5.1 FrameComposition как реальная физическая метка

`FrameComposition` сейчас пассивные метаданные. В свете UGS — это **реальный уровень** в иерархии, определяемый глубиной composition bonds от C0.

```
C0_Primitive    depth(composition_tree) = 0   →  прямо из Perceptor или universal
C1_Atom         depth = 1               →  собран из C0
C2_Molecule     depth = 2               →  собран из C1
C3_Structure    depth = 3               →  собран из C2
C4_Composition  depth = 4               →  собран из C3
C5_Plus         depth ≥ 5               →  высшие абстракции
```

Глубина **вычисляется** из composition bonds, не задаётся вручную.

### 5.2 Примеры для разных когнитивных слоёв

**Язык:**
```
C0  visual_stroke, visual_curve             (перцептивный атом)
C1  grapheme (буква: собрана из штрихов)
C2  morpheme (корень: собран из букв)
C3  word     (слово: собрано из морфем)
C4  phrase   (фраза)
C5  sentence, meaning
```

**Пространство:**
```
C0  spatial_above, spatial_left_of         (реляционный атом)
C1  landmark   (точка ориентира)
C2  local_layout (расположение нескольких объектов)
C3  route      (путь между точками)
C4  cognitive_map
C5  spatial_principle ("в этом городе всё симметрично")
```

**Каузальность:**
```
C0  causal_precedes, causal_contact        (каузальный атом)
C1  causal_pair (A → B)
C2  causal_chain (A → B → C)
C3  physical_scene (набор причинно связанных событий)
C4  causal_model (понимание механизма)
C5  physical_principle ("предметы падают вниз")
```

## 6. Реляционные примитивы — нюанс хранения

### 6.1 Проблема

Реляционный примитив `spatial_above` сам по себе бессмысленен. Он означает "А выше чем Б". Но где хранить "А" и "Б"?

Варианты:

**Вариант A:** Реляционный примитив — это тип Connection, не Token.
- `spatial_above` = тип связи между двумя токенами
- Хранится в Connection.link_type
- Минус: link_type уже занят (0x0800 синтаксические связи и т.д.)

**Вариант B:** Реляционный примитив — Token-якорь, аргументы — отдельные Tokens, связи между ними.
- `[above]` ← TokenAnchor
- `[cup]` → `[above]` → `[table]` через CONNECTION
- Плюс: вписывается в существующую архитектуру

**Вариант C:** Реляционный примитив — это Shell-профиль Connection.
- `spatial_above` закодирован в shell[8] Connection-а, не отдельный токен

Пока **решение не зафиксировано**. Требует проработки при реализации.

## 7. Понижение дна — Floor Lowering

### 7.1 Главная идея

Система обнаруживает что предполагаемый примитив ведёт себя **несогласованно** — значит он не атом, в нём есть структура.

### 7.2 Сигнал несогласованности

```
primitive_X  в контексте A  →  TransitionMatrix паттерн {B:0.8, C:0.1}
primitive_X  в контексте B  →  TransitionMatrix паттерн {D:0.7, E:0.2}
```

Если X действительно примитив — паттерны должны быть одинаковы (одинаковый атом → одинаковое поведение). Если нет — X содержит скрытую структуру.

### 7.3 Процедура понижения

Происходит **только в DREAM Phase**, не на горячем пути.

```
1. DREAM Phase: scan для несогласованных C0
   Критерий: variance(transition_patterns_by_context) > INCONSISTENCY_THRESHOLD

2. Для каждого несогласованного примитива X:
   - Запросить Perceptor: "проанализируй X на уровень глубже"
   - Perceptor возвращает: sub_primitives = [X_a, X_b, X_c]

3. FrameWeaver:
   - X переводится из C0 → C1_Atom
   - X_a, X_b, X_c регистрируются как новые C0
   - composition_of(X) = [X_a, X_b, X_c]
   - Все Frame которые использовали X: получают updated composition bonds

4. NotifyFloorLowered(original: X, sub_primitives: [X_a, X_b, X_c])
   → chrnv может одобрить или отклонить
```

### 7.4 Ограничения

| Правило | Значение |
|---------|----------|
| Максимум понижений за цикл | **3 уровня** (не глубже) |
| Только в DREAMING | Guardian запрещает вне DREAM Phase |
| Требует одобрения chrnv | через Workstation |
| Универсальные примитивы | никогда не понижаются |
| Минимальный возраст примитива | `min_age_events` перед рассмотрением |
| Cooldown | между понижениями одного примитива |

### 7.5 Зачем 1–3 порога, не больше

Каждый уровень снижения умножает число примитивов (один X → три X_*). Три уровня: 1 → 3 → 9 → 27 примитивов. Это управляемо. Четыре уровня: 1 → 81 — уже может взорвать пространство. Плюс: три уровня покрывают все практические случаи — редко нужно идти глубже.

## 8. Жизненный цикл примитива

```
proposed    →  system обнаружила кандидата (из co-activation или floor lowering)
candidate   →  DREAM Phase подтвердил стабильность
in_review   →  chrnv смотрит в Workstation
active      →  загружен, используется системой; depth = по реактивациям
mature      →  стабилен, высокий reactivation_count
deprecated  →  вытесняется более точными sub-primitives (floor lowering)
archived    →  не загружается, но composition bonds сохранены
```

Примитивы из стартового yaml начинают сразу с `active` и depth = 65535.

## 9. Абстракции — как строятся из примитивов

### 9.1 Единственный механизм: co-activation + FrameWeaver

Абстракция возникает когда одни и те же примитивы (или C1/C2 Frame) **регулярно активируются вместе**. FrameWeaver кристаллизует их co-activation в новый Frame — следующего уровня в иерархии.

```
visual_stroke_horizontal + visual_stroke_vertical + visual_corner
  → co-activation 50+ раз
  → FrameWeaver: кристаллизует Frame "прямой угол" (C1_Atom)
  → composition_of: [stroke_h, stroke_v, corner]
```

Это **единственный** путь возникновения абстракций. Нет другого механизма.

### 9.2 Насколько абстрактна абстракция

Степень абстракции = **глубина дерева composition bonds от C0**.

`прямой_угол`:  depth=1 (из C0 strokes)  
`буква_Л`:     depth=2 (из C1 прямых углов)  
`слово`:       depth=3 (из C2 букв)  
`математика`:  depth=5+ (из множества путей)

### 9.3 Обратное направление: абстракция → примитив

Высокий SutraDepth Frame-а → высокая масса → сильная гравитация → он **притягивает** близкие токены.

Это top-down влияние: концепция "окружность" заставляет систему замечать круглые объекты быстрее. Встроено в физику SPACE — не нужен отдельный механизм.

## 10. Нюансы

### 10.1 Одно и то же, разные уровни

`writing_symbol` (L1, семантический) и `visual_stroke` (L0, перцептивный) — разные примитивы на разных уровнях. Они связаны через composition bonds, но каждый существует самостоятельно.

Не путать: L0/L1 — это про происхождение (перцептор vs человек). C0/C1 — это про позицию в иерархии (атом vs составной).

L0-примитив всегда C0. L1-примитив обычно C0, но может быть C1 если он составной (редко).

### 10.2 Примитивы без Perceptor-а

Универсальные примитивы (§4) и стартовые L1 yaml-примитивы существуют даже когда ни один Perceptor не активен. Они — постоянная часть SUTRA.

L0-примитивы без активного Perceptor-а в SUTRA присутствуют как токены (загружены при старте), но не получают новых активаций.

### 10.3 Конфликтующие примитивы

Два примитива могут быть в одном регионе SUTRA-пространства с похожей shell но разными composition bonds. Это **не ошибка** — это семантическая близость. AxialEvaluator разбирает их по октантам.

### 10.4 Emergent примитив ≠ primitive из yaml

Emergent primitive (обнаруженный NeuralAdvisor/DepthThresholdDetector) начинает как C1 или C2 — он crystallized из co-activations. Только при одобрении chrnv и регистрации в SutraDepthStore он получает `depth = high` и начинает вести себя как примитив. До этого — кандидат.

### 10.5 Примитивы и SutraDepth

У yaml-примитивов depth = 65535 **статически**. Это не "они часто использовались" — это конституционная метка.

У emergent primitive depth **растёт** с реактивациями. Он может стать "глубоким" как yaml-примитив, но только эмпирически, со временем.

## 11. Практика: как добавлять примитивы

### Добавление L0-примитива (yaml)

```yaml
# config/anchors/perceptual/visual_primitives.yaml
- id: "visual_stroke_horizontal"
  layer: L0
  position: [X, Y, Z]    # позиция в SUTRA-пространстве
  mass: 50                # небольшой (L0 не должны доминировать)
  shell: [0,0,0,0,0,0,0,0]  # нейтральная shell
  universal: false
```

Правила:
- `mass` L0-примитивов меньше чем L1 (они атомы, не доминанты)
- `shell` нейтральная или слабо выраженная
- `position` в специально выделенной области SUTRA для L0

### Добавление L1-примитива (yaml, существующая практика)

Без изменений — как сейчас в `config/anchors/writing/`, `config/anchors/mathematics/`.

### Добавление универсального примитива

**Только через код**, не через yaml. Через `GUARDIAN` с явным `Permission::Forbidden` на модификацию.
