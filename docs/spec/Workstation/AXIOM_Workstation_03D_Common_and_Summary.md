## 10. Общие элементы интерфейса

### 10.1 Структура каждого окна

Каждое окно Workstation имеет одинаковую внешнюю структуру:

```
┌─────────────────────────────────────────────────────────────────────┐
│  AXIOM Workstation                            ● Connected (14:32)   │  ← Шапка окна
│  ┌─────┬──────┬─────────┬───────────┬─────────────┬───┬─────┬─────┐ │  ← Tabs
│  │ Map │Field │Patterns │ DreamState│Conversation │..│Files│Bench│  │
│  └─────┴──────┴─────────┴───────────┴─────────────┴───┴─────┴─────┘  │
│                                                                     │
│  [           Содержимое активной вкладки                          ] │  ← Контент окна
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

Три уровня:

1. **Шапка** — название "AXIOM Workstation" слева (можно убрать в системную title bar) + индикатор подключения справа
2. **Tabs** — навигация между окнами
3. **Контент** — описано в Частях A, B, C

В **detached** окне:

```
┌─────────────────────────────────────────────────────────────────────┐
│  System Map (detached)                        ● Connected (14:32)   │  ← Только индикатор
│                                                                     │
│  [           Содержимое окна                                      ] │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

В detached режиме нет табов (это окно посвящено одной вкладке) и название отражает имя вкладки.

### 10.2 Индикатор подключения

Постоянный элемент в правом верхнем углу. Имеет три состояния:

```
● Connected (engine v2.1.0, since 14:32)         [зелёный шалфей]
● Reconnecting (attempt 3, next in 5s)           [тёплый янтарный]
● Disconnected — Engine not running              [тёплый красный]
```

**Состояния:**

- **Connected** — circle + текст с версией Engine и временем подключения
- **Reconnecting** — circle с лёгкой пульсацией + текст про попытку и следующую retry
- **Disconnected** — circle + причина (engine not running, engine crashed, network error, etc.)

**Поведение:**

- Клик на индикатор → раскрывается **диагностический popup** с детальной информацией:
  - Точное время подключения
  - Версии Engine и Workstation
  - Адрес engine
  - Статистика последних минут (сообщений получено/отправлено)
  - Кнопки: Disconnect, Reconnect now, Configure connection

- Двойной клик → быстрая команда reconnect

При смене состояния — короткая анимация перехода (~ 500ms).

### 10.3 Цвет индикатора и общий тон окна

Индикатор подключения **не влияет** на общий тон окна. То есть фон Workstation не становится красным когда Disconnected. Только сам индикатор окрашивается. Это держит интерфейс **спокойным**.

Тёплый красный (внимание) появляется в окнах **только при необходимости**:
- В Live Field overlay при потере связи
- В Conversation при ошибке отправки
- На карточках в Patterns при GUARDIAN veto
- В индикаторе подключения

### 10.4 Главное меню

В верхней части окна (под title bar или в title bar — зависит от ОС) — **главное меню**:

```
File   Engine   View   Configuration   Help
```

**File:**
- New Workstation Window (открывает копию для других детачей) — V2.0
- Quit

**Engine:**
- Connect / Disconnect
- Spawn local engine...
- Shutdown engine... (с подтверждением)
- Force shutdown... (с подтверждением)
- Settings → Configuration tab

**View:**
- Tab list (System Map, Live Field, Patterns, ...) — переход к вкладке
- Detach current tab
- Re-attach detached tab → submenu со списком detached
- Reset window topology (закрыть все detached, вернуть к default)

**Configuration:**
- Open Configuration tab
- Apply all pending changes
- Discard all pending changes
- Reset all to defaults (с подтверждением)

**Help:**
- About AXIOM Workstation
- Show welcome screen (одноразово, без изменения config)
- Documentation (открывает README или web)

Это **минимальное** меню. Никаких 50 пунктов как в IDE.

### 10.5 Контекстное меню — нет в V1.0

Правый клик не открывает контекстных меню в V1.0. Все действия — через основной UI или главное меню.

В V2.0 возможно добавление контекстного меню для специфических действий (правый клик на Frame в Patterns → Open in Live Field, и т.д.).

## 11. Диалоги подтверждения

### 11.1 Когда нужны

Диалог подтверждения появляется только для **значимых необратимых действий**:

- Shutdown engine
- Force shutdown engine
- Cancel running import (с уже добавленными токенами)
- Cancel running benchmark
- Reset configuration to defaults

### 11.2 Композиция диалога

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│   Cancel import?                                     │
│                                                      │
│   Tokens added so far (1,234) will remain in the     │
│   system. Cancellation cannot be undone.             │
│                                                      │
│                                                      │
│              [ Cancel ]    [ Yes, cancel ]           │
│                                                      │
└──────────────────────────────────────────────────────┘
```

Структура:

- **Заголовок** — короткий вопрос ("Cancel import?", "Shutdown engine?", "Reset configuration?")
- **Описание** — что произойдёт, обратимо ли. Спокойным тоном, без устрашения.
- **Две кнопки**: отмена и подтверждение
- **Default action** на отмену — Esc возвращает к работе без изменений
- **Подтверждение** на правой стороне, отмена слева (стандартная конвенция)

### 11.3 Что НЕ должно быть в диалогах

- **Не блокировать всё приложение.** Диалог модальный для своего окна, но не блокирует другие detached окна.
- **Никаких изображений/иконок предупреждения.** "⚠️ Warning!" с рамками — нет.
- **Никаких трёх кнопок.** Только две: yes/no. Если кажется, что нужна третья — это плохо спроектированное действие.
- **Никаких длинных описаний.** Максимум 2 строки.

### 11.4 Принцип проектирования диалога

**Диалог должен помогать принять решение, не пугать.** Тон — спокойный. Описание — фактическое. Никаких "Are you sure???".

## 12. Сообщения об ошибках

### 12.1 Типология ошибок

В Workstation возможны три типа ошибок:

**1. Connection errors** — потеря связи с Engine.
- Не блокируют интерфейс
- Показываются через индикатор подключения
- Фоновая retry логика (см. Документ 2 раздел 2.4)

**2. Command errors** — Engine отверг команду (validation, GUARDIAN veto, etc).
- Показываются inline там, где была подана команда
- Например, в Conversation — в ленте как сообщение системы с тёплым красным акцентом
- В Configuration — на поле, где было невалидное значение

**3. Workstation errors** — внутренние сбои Workstation (cannot read config file, render error, etc).
- Логируются в файл (Документ 2 раздел 10.2)
- Показываются как Alert в углу окна, не блокируя работу
- Если критично — graceful degradation (одно окно может перестать работать, остальные продолжают)

### 12.2 Что НЕ должно быть в обработке ошибок

- **Никаких всплывающих модальных окон с ошибками.** Ошибка показывается в контексте, не как блокирующее уведомление.
- **Никаких stack traces в UI.** Они в логах. В UI — человеческое описание.
- **Никаких "Что-то пошло не так".** Сообщение должно быть конкретным.
- **Никаких звуков ошибок.** Звука нет вообще в V1.0.

### 12.3 Тон сообщений об ошибках

Ошибки в системных сообщениях — **технически точные, без эмоций**:

✓ "Could not process text: domain LOGIC is at capacity (50000/50000 tokens)."
✓ "Connection lost — engine not responding to ping. Retrying in 5s."
✓ "Configuration value out of range: scan_interval_ticks must be between 1 and 1000."

✗ "Oh no! Something went wrong with the engine 😢"
✗ "ERROR! ERROR! Connection failed!"
✗ "Houston, we have a problem."

## 13. Адаптация к размеру окна

### 13.1 Минимальные размеры

Workstation предполагает размер монитора **не меньше 1280x720** для main window. Это разумный минимум для современного оборудования.

При попытке уменьшить окно ниже минимума — окно не уменьшается дальше (jest горизонтальная или вертикальная ось может остановиться раньше).

### 13.2 Адаптация в больших окнах

Когда окно растёт (полноэкранный режим на 4K мониторе) — содержимое **не тянется бесконечно**. Каждое окно имеет логичную **максимальную ширину контента**:

- **System Map** — до ~ 1600px по ширине, потом контент центрируется и появляются поля справа/слева
- **Patterns, Dream State, Conversation, Configuration, Files, Benchmarks** — аналогично, до 1400-1600px
- **Live Field** — занимает максимум доступного пространства (визуальное окно выигрывает от размера)

Это **классическая школа Cupertino**: контент имеет свой размер, белое пространство — полноценный участник композиции.

### 13.3 Адаптация в маленьких окнах

При размере меньше комфортного минимума окна:

- **Многоколоночные layout-ы** становятся одноколоночными (например, в Configuration — Sections панель сворачивается в dropdown)
- **Panels** могут стать табами внутри одного окна (например, в Files — Available adapters / New import / Active imports)

В V1.0 — поддерживаем это упрощённо. Полноценная responsive design — V2.0.

### 13.4 Detached окна

Detached окна **могут быть меньше**, чем main window — там одна вкладка, ей нужно меньше пространства.

Минимальный размер detached окна — около 600x400. Меньше уже неудобно.

## 14. Шорткаты клавиатуры

Минимальный набор шорткатов в V1.0:

**Глобальные (работают в любом окне):**

- `Ctrl+1` ... `Ctrl+8` — переключение между вкладками (в порядке: Map, Field, Patterns, Dream, Conversation, Files, Config, Bench)
- `Ctrl+,` — переход в Configuration
- `Ctrl+/` — открыть help/about
- `Esc` — закрыть текущий диалог
- `F11` — fullscreen toggle (если ОС поддерживает)

**В Conversation:**

- `Ctrl+Enter` (или `Cmd+Enter`) — Submit text
- `Enter` — новая строка в поле ввода

**В Configuration:**

- `Ctrl+S` — Apply changes (текущей категории)
- `Ctrl+Z` — Discard changes (текущей категории)

**В Live Field:**

- `R` — reset camera to default position
- `Click+drag` — поворот камеры
- `Scroll` — приближение/отдаление

**В Benchmarks:**

- `Enter` на строке бенчмарка — Run

Это минимальный набор. Никаких 50 шорткатов как в IDE. Только то, что часто используется.

## 15. Анимации — общие принципы

Из Документа 1 (раздел 5.2): анимации длиной 200-400 мс, плавные, ease-in-out.

Здесь — конкретизация для разных типов:

### 15.1 Анимации появления

- **Появление окна (Welcome → System Map):** ~ 800ms fade
- **Переключение между табами:** ~ 200ms fade-cross
- **Появление новой карточки в Patterns:** ~ 300ms fade-in + slide
- **Появление сообщения в Conversation:** ~ 200ms fade-in
- **Появление диалога подтверждения:** ~ 250ms fade + scale (от 95% до 100%)

### 15.2 Анимации исчезновения

- **Закрытие диалога:** ~ 200ms fade
- **Удаление карточки (например, completed import после 24h):** ~ 400ms fade-out + collapse

### 15.3 Анимации состояния

- **Изменение fatigue в Dream State:** sparkline плавно обновляется по 1 точке в секунду
- **Прогресс импорта в Files:** progress bar плавно растёт, не скачет
- **Пульсация мандалы в System Map:** период 3-5 секунд (см. Часть A раздел 2.4)
- **Подсветка линии потока в System Map при активности:** ~ 500ms вспышка с fade-out

### 15.4 Анимации, которые мы НЕ делаем

- **Никаких bounce/elastic эффектов.** Школа Cupertino, не Material Design.
- **Никаких rotation 360.** Crystallization не "вращается".
- **Никаких parallax эффектов.**
- **Никаких эффектов "shake" при ошибках.** Ошибка показывается просто, без атаки внимания.

### 15.5 Performance

Если FPS UI падает ниже 30 — анимации **упрощаются автоматически** (без длительного fade, без плавных переходов). Это deferred к этапу реализации, но архитектурно учитывается.

## 16. Доступность (accessibility)

### 16.1 Что делаем в V1.0

- **Все элементы доступны через клавиатуру.** Tab порядок логичный.
- **Контрастность палитры.** Тёмно-графитовый текст на off-white фоне даёт хороший контраст. Цвета состояний выбраны так, чтобы быть различимыми и при дальтонизме (синий vs янтарный vs красный — три разные оси).
- **Размер текста.** Дефолтный размер достаточно большой для чтения. Можно увеличить через ОС настройки.

### 16.2 Что не делаем в V1.0

- **Высокий контраст mode.** V2.0.
- **Screen reader support.** Iced ограничивает это в текущих версиях. V2.0 в зависимости от состояния iced.
- **Локализация.** В V1.0 — английский интерфейс с возможным русским в некоторых местах (chrnv лексика). Полная локализация — V2.0.

## 17. Сводка всех окон

Финальная таблица всех окон Workstation V1.0:

| #  | Окно             | Назначение                                      | Detach | Категория       | Часть |
|----|------------------|-------------------------------------------------|--------|-----------------|-------|
| 1  | Welcome          | Первое впечатление, запуск engine               | Нет    | Стартовое       | A     |
| 2  | System Map       | Главное окно, карта системы с мандалой          | Да     | Наблюдение      | A     |
| 3  | Live Field       | 3D-визуализация полей доменов                   | Да     | Наблюдение      | B     |
| 4  | Patterns         | Узоры в реальном времени (FrameWeaver events)   | Да     | Наблюдение      | B     |
| 5  | Dream State      | Состояние сна, fatigue, DreamCycle, history     | Да     | Наблюдение      | B     |
| 6  | Conversation     | Утилитарный чат для подачи текста               | Да     | Взаимодействие  | C     |
| 7  | Files            | Knowledge import через External Adapters        | Да*    | Взаимодействие  | C     |
| 8  | Configuration    | Настройки Engine и Workstation                  | Нет    | Взаимодействие  | C     |
| 9  | Benchmarks       | Запуск бенчмарков, история результатов          | Да     | Взаимодействие  | C     |

(*) Files технически может, но обычно служебное.

### 17.1 Логика разделения

- **Welcome** — отдельная категория, специальное стартовое окно.
- **Окна наблюдения** (System Map, Live Field, Patterns, Dream State) — главные. **Все могут быть detached** и часто — на второй монитор.
- **Окна взаимодействия** (Conversation, Files, Configuration, Benchmarks) — служебные. Открываются по необходимости, обычно работают в main window.
- **Detach всех ключевых окон** — даёт пользователю возможность настроить рабочее место под свои привычки.

### 17.2 Сценарии использования

**Сценарий 1: Разработка**
- Main window: Configuration или Conversation
- Detached на втором мониторе: System Map (постоянно работает)
- Иногда переключение между Patterns и Dream State

**Сценарий 2: Импорт большого корпуса**
- Main window: Files
- Detached: System Map (видеть общее состояние) + Patterns (видеть кристаллизации) + Live Field на EXPERIENCE (видеть, как заполняется поле)

**Сценарий 3: Бенчмарки**
- Main window: Benchmarks
- Detached: System Map (видеть, что bench-instance работает отдельно от основного engine)

**Сценарий 4: Долгое наблюдение**
- Main window: спокойное окно — Conversation для случайных тестов
- Detached: System Map постоянно
- Workstation остаётся открытым весь день

## 18. Связи между окнами — навигация

Некоторые окна могут "переводить" пользователя в другие через клик:

**System Map:**
- Клик на блок домена → не переход (только подсветка), domain detail можно увидеть в Live Field
- Клик на over-domain точку (GUARDIAN, FrameWeaver, DREAM Phase) → переход в соответствующий tab (Patterns или Dream State)

**Patterns:**
- Клик на карточку Frame → попап деталей с кнопкой "View in Live Field" (открывает Live Field на нужном домене с подсветкой анкера)

**Dream State:**
- Клик на DreamReport → попап деталей с кнопкой "View promoted frames" (открывает Patterns с фильтром на эти Frame)

**Files:**
- После завершения импорта — потенциальный быстрый jump в Live Field на target domain. Не делаем в V1.0, но архитектурно возможно.

Эти **cross-window navigation** связи делают окна не изолированными, но **связанной системой**. Главное — они **минимальны** и **очевидны**.

## 19. Что НЕ описано в Документе 3 (но нужно знать)

Этот документ описывает **дизайн UI**, не **дизайн визуальный** (точные пиксели, шрифты, оттенки). После согласования Документа 3:

- **Этап визуального дизайна** — Figma/Sketch макеты с точной типографикой, цветами, иконками, иллюстрациями (мандала)

Документ 3 даёт **что** и **как должно работать**. Этап дизайна даёт **как это выглядит конкретно**. Реализация — **код**.

## 20. Открытые вопросы по общему дизайну

### 20.1 Tabs visual style

Я описал tabs как простые табы в верхней части. Но **визуально** они могут быть:
- Как закладки (с лёгким радиусом сверху, активная вкладка имеет фон контента)
- Как pills (отдельно стоящие кнопки)
- Как простой text-нав (только текст, активная подчёркнута)

Я склоняюсь к **закладкам** — самый традиционный и понятный паттерн. Решается на этапе визуального дизайна.

### 20.2 Title bar — собственный или системный

Workstation может рисовать свой title bar (с логотипом и интегрированным меню) или использовать системный. iced поддерживает оба варианта.

Свой title bar даёт больше контроля над эстетикой. Системный — лучше интеграция с ОС.

Я склоняюсь к **системному** в V1.0, своему — в V2.0 если будет нужно.

### 20.3 Иконка приложения

В V1.0 нужна иконка для bundle/installer. Это **этап визуального дизайна**, не часть этого документа. Заглушка — простой геометрический символ (возможно, упрощённая мандала с одним кругом).

### 20.4 Темы — light/dark

В V1.0 — **только light theme** (off-white фон).

Dark theme — V2.0. Для разработчика, который работает ночью — может быть нужна. Но в V1.0 не делаем.

### 20.5 Плотность интерфейса

Я везде проектировал в **средней плотности** — спокойно, с воздухом. Это не "compact mode" как в Slack, не "spacious" как в Apple Notes — посередине.

Если в реальной работе окажется что плотность не подходит — пользователь может настроить через Configuration → Display → Density (compact / normal / spacious). Это deferred в V1.0.

## 21. Финальное резюме Документа 3

### 21.1 Девять окон

Workstation V1.0 имеет девять окон:

1. **Welcome** — спокойное стартовое окно
2. **System Map** — главное окно с мандалой и архитектурой системы
3. **Live Field** — 3D-визуализация полей доменов
4. **Patterns** — узоры FrameWeaver в реальном времени
5. **Dream State** — состояние сна и DreamCycle
6. **Conversation** — утилитарный чат
7. **Files** — knowledge import через External Adapters
8. **Configuration** — настройки Engine и Workstation
9. **Benchmarks** — запуск и просмотр бенчмарков

### 21.2 Общие свойства

- Все окна следуют школе Cupertino: минимализм, скруглённые формы, ограниченная палитра, спокойствие
- Все используют семантику цвета (5 состояний)
- Все имеют верхнюю панель с индикатором подключения и табами
- Большинство (7 из 9) могут быть detached на отдельные системные окна
- Никаких pop-up уведомлений, звуков, прогрессбаров кроме Dream State

### 21.3 Что главное

System Map — **центр**. Мандала на нём — символическое присутствие AXIOM в Workstation. Остальные окна разворачивают конкретные аспекты системы.

Все окна **спроектированы для расширения**: каждое имеет явный список "что НЕ делается в V1.0", который служит roadmap для будущих версий.

### 21.4 Что дальше

После согласования всех частей Документа 3 — переходим к **Документу 4: Implementation Phases**.

Документ 4 — это **порядок реализации**. Не дизайн, не архитектура. План: что Sonnet делает первым, вторым, третьим. С этапами проверки после каждого шага.

Без Документа 4 проект не может перейти от спецификации к коду.

## Приложение: язык глобальных элементов

Чтобы все четыре документа использовали одни и те же слова:

- **Шапка окна** — верхняя часть с названием и индикатором подключения
- **Tabs** — навигация между окнами в main application
- **Контент окна** — основная область с описанным в Частях A/B/C
- **Detached** — вкладка, вытащенная в отдельное системное окно
- **Главное меню** — File / Engine / View / Configuration / Help
- **Индикатор подключения** — circle с состоянием в правом углу шапки
- **Диалог подтверждения** — модальное окно для значимых действий
- **Inline ошибка** — ошибка в контексте окна, не модальная
- **Cross-window navigation** — переходы между окнами по клику на элементы
