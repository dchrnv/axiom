
# Axiom Core Specification

Version: 1.0 (Canonical)

## 1 Назначение

**Axiom Core** определяет фундаментальные инварианты системы.
Любая реализация, модуль или расширение обязаны соблюдать эти инварианты.
Нарушение инвариантов означает несовместимость с Axiom Core.

## 2 Термины

### Базовые сущности
**Token** — голографическая частица смысла (64 байта), существующая в Domain с позицией, импульсом и термодинамикой.
**Connection** — направленная связка между двумя Token (64 байта), реализующая пружинную динамику с шлюзами.
**Domain** — изолированный вычислительный контейнер ("клетка") с уникальной физикой поля и мембраной.
**COM (Causal Order Model)** — модель времени как причинного порядка событий, заменяющая wall-clock время.

### Временная модель
**Event** — изменение состояния системы с монотонным event_id (32 байта).
**Timeline** — последовательность событий, определяющая эволюцию системы.
**event_id** — монотонно возрастающий идентификатор, единственная мера времени в системе.

### Архитектура высокого уровня
**Ashti_Core** — фрактальный уровень сознания из 10 Доменов: SUTRA(0), ASHTI(1-8), MAYA(10).
**SUTRA** — Домен 0, единственный источник истины, генератор Token.
**ASHTI** — Домены 1-8, активные процессоры с уникальной физикой поля.
**MAYA** — Домен 10, интерфейс проекции результата.
**CODEX** — живой закон, эволюционирующий из базовых аксиом.
**GUARDIAN** — над-доменный слой, ингибирующий нарушения CODEX.

### Конфигурация и физика
**DomainConfig** — 128 байт конфигурация Домена, определяющая его природу и поведение.
**Field** — активное 3D пространство Домена с уникальной физикой.
**Membrane** — фильтрующий слой Домена, контролирующий вход/выход.
**Anchor** — точка ноля (0,0,0), гравитационный центр Домена.

### Визуализация и наблюдение
**UPO (Universal Phase Operator)** — модуль наблюдения за динамикой Домена.
**DynamicTrace** — след динамики на экране (32 байта).
**Screen** — 3D пространство следов с затуханием по event_id.

### Протоколы и взаимодействие
**10 -> 0** — протокол передачи результата от MAYA к SUTRA следующего уровня.
**inject_pattern** — впрыск паттерна в Домен.
**harvest_result** — сбор результата из Домена.
**peek_state** — сканирование состояния Домена GUARDIAN'ом.

## 3 Структура пространства

3.1. **Фундаментальная единица** — Домен (Domain), изолированный суверенный контейнер.
3.2. **Физика Домена** — уникальная для каждого DomainConfig, определяет поведение Token.
3.3. **Фрактальная модель** — Домены могут рекурсивно создавать под-Домены.
3.4. **Уровни сознания** — Ashti_Core из 10 Доменов образует фрактальный уровень.
3.5. **Протокол 10 -> 0** — результаты MAYA становятся входом SUTRA следующего уровня.

## 4 Token (64 байта)

4.1. **Структура** — sutra_id, domain_id, position, velocity, momentum, термодинамика.
4.2. **Существование** — Token всегда принадлежит одному Domain.
4.3. **Движение** — определяется физикой поля Domain и COM событиями.
4.4. **Время** — синхронизация через last_event_id из COM.

## 5 Connection (64 байта)

5.1. **Направленность** — source_id → target_id в рамках одного Domain.
5.2. **Динамика** — пружинная модель (strength, stress, elasticity, ideal_dist).
5.3. **Шлюзы** — фильтрация по mass, temperature через gates.
5.4. **Время** — created_at хранит COM event_id.

## 6 COM (Causal Order Model)

6.1. **Монотонность** — event_id строго возрастает.
6.2. **Детерминизм** — одинаковые payload_hash → одинаковые изменения.
6.3. **Причинность** — parent_event_id < event_id.
6.4. **Доменная изоляция** — события разных доменов независимы.

## 7 Domain (128 байт конфигурация)

7.1. **Суверенитет** — каждый Domain имеет уникальную DomainConfig.
7.2. **Изоляция** — Token не может существовать вне Domain.
7.3. **Мембрана** — фильтрует вход/выход по threshold'ам.
7.4. **Физика** — gravity, friction, temperature уникальны для Domain.

## 8 Ashti_Core Архитектура

8.1. **SUTRA (0)** — единственный источник истины, генератор Token.
8.2. **ASHTI (1-8)** — восемь процессоров с уникальной физикой поля.
8.3. **MAYA (10)** — интерфейс проекции результата.
8.4. **CODEX + GUARDIAN** — над-доменный контроль соблюдения законов.

## 9 Фундаментальные инварианты

9.1. **Размеры структур** — Token (64), Connection (64), Event (32), DomainConfig (128).
9.2. **COM модель времени** — нет wall-clock времени, только event_id.
9.3. **Изоляция Доменов** — прямая коммуникация запрещена, только через шины.
9.4. **Единый источник истины** — только SUTRA создает новые Token.
9.5. **Вечная память** — min_intensity > 0 на Screen UPO.

## 10 Расширения

10.1. **Нарушение инвариантов** — запрещено под угрозой несовместимости.
10.2. **Новые DomainConfig** — допустимы в рамках Domain V1.3.
10.3. **Дополнительные EventType** — допустимы в COM V1.0.
10.4. **Новые StructuralRole** — допустимы в Ashti_Core.

## 11 Версионирование

11.1. **Мажорная версия** — при изменении инвариантов или размеров структур.
11.2. **Минорная версия** — при добавлении полей с сохранением размеров.
11.3. **Патч-версия** — при исправлении ошибок без изменения API.
