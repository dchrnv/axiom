## 1. Domain JSON API и конфигурация - КРИТИЧЕСКИЙ

**Где:** `runtime/src/domain.rs`, `runtime/src/domain_config.rs`  
**Что отложено:** JSON сериализация/десериализация DomainConfig, API для создания/модификации  
**Почему:** Нужно с самого начала для гибкости и экспериментов  
**Когда планируется:** Немедленно - блокирует дальнейшую разработку  

**Требуется:**
- JSON serde derive для DomainConfig
- Factory методы для стандартных доменов (Logic, Dream, Math...)
- from_file() для загрузки YAML/JSON конфигов
- Builder pattern для кастомизации доменов
- Валидация конфигов при загрузке
- Примеры конфигов для Ashti_Core доменов

**Приоритет:** КРИТИЧЕСКИЙ - фундамент для всей архитектуры

## 3. Оптимизация DomainConfig до 128 байт

**Где:** `runtime/src/domain.rs`  
**Что отложено:** Оптимизация структуры DomainConfig с 184 до 128 байт  
**Почему:** Проблема с выравниванием полей в Rust (padding)  
**Когда планируется:** После реализации JSON API  

**Текущий статус:** 184 байта вместо 128  
**Проблема:** 
- Выравнивание полей создает padding (~56 байт)
- Спецификация DomainConfig V1.0 содержит больше полей чем Domain V1.3
- Нужно выбрать правильную спецификацию

**Возможные решения:** 
- Использовать спецификацию Domain V1.3 (меньше полей)
- Пересмотреть порядок полей для минимизации padding
- Использовать `#[repr(packed)]` с осторожностью
- Обновить спецификацию DomainConfig V1.0 до 128 байт

## 11. Configuration System - исправление тестов

**Где:** `runtime/src/config/tests.rs`  
**Что отложено:** Исправление путей загрузки конфигураций в тестах  
**Почему:** Тесты запускаются из `runtime/` директории, но пытаются загрузить файлы из `config/`  
**Когда планируется:** v0.2.0 - Исправление тестов  

### 🐛 **Проблема:**

**Текущие тесты:**
- Запускаются из `runtime/` директории
- Пытаются загрузить файлы по относительным путям `config/axiom.yaml`
- Рабочая директория тестов != рабочая директория приложения

**Ожидаемое поведение:**
- Тесты должны запускаться из корня проекта
- Пути к конфигурационным файлам должны быть корректными
- Файлы должны находиться в правильных местах

### 🔧 **Требуется исправить:**

#### **1. Настройка рабочей директории:**
```rust
// В тестах установить правильную рабочую директорию
#[cfg(test)]
fn setup_test_config() {
    // Установить рабочую директорию в корень проекта
    std::env::set_current_dir("/home/chrnv/Axiom");
}
```

#### **2. Абсолютные пути в тестах:**
```rust
// Использовать абсолютные пути для надежности
let config_path = Path::new("config/axiom.yaml");
let runtime_path = Path::new("config/runtime/runtime.yaml");
let schema_path = Path::new("config/schema/domain.yaml");
```

#### **3. Создание тестовых файлов:**
```rust
// Создавать тестовые файлы перед запуском тестов
fn create_test_files() {
    // Убедиться что файлы существуют
    fs::create_dir_all("config/").ok();
    fs::create_dir_all("config/runtime").ok();
    fs::create_dir_all("config/schema").ok();
}
```

#### **4. Использование временной директории:**
```rust
// Для тестов использовать временную директорию
#[cfg(test)]
const TEST_CONFIG_DIR: &str = "/tmp/axiom_test_config/";
```

### 🚀 **План исправления:**

**Phase 1: Быстрое исправление**
- Исправить пути в существующих тестах
- Добавить setup/teardown для тестовых файлов
- Использовать абсолютные пути

**Phase 2: Улучшение тестов**
- Добавить тесты для отсутствующих файлов
- Тесты валидации с ошибочными данными
- Тесты горячей перезагрузки

**Phase 3: Тестовая инфраструктура**
- Отдельная тестовая конфигурация
- Mock файлы для edge cases
- Интеграционные тесты с реальными модулями

### 📊 **Конечная цель:**

**Надежные тесты:**
- Все тесты проходят независимо от рабочей директории
- Правильная обработка отсутствующих файлов
- Валидация путей и конфигураций
- Изоляция тестовых данных

### 🎯 **Критерии успеха:**

1. **cargo test config** проходит из любой директории
2. **Тесты создают** свои конфигурационные файлы
3. **Пути корректны** для любой рабочей директории
4. **Очистка** после выполнения тестов
5. **Изоляция** тестовых данных от production
