# AXIOM MODULE SPECIFICATION: Shell V3.0

**Статус:** Актуальная спецификация (core)  
**Версия:** 3.0.0  
**Дата:** 2026-03-20  
**Назначение:** Лёгкий семантический кэш, вычисляемый из связей токена  
**Формат записи:** 8 байт на токен (`[u8; 8]`)  
**Модель времени:** COM `event_id` (причинный порядок, u64)  
**Связанные спеки:** Token V5.2, Connection V5.0, DomainConfig V2.1, Heartbeat V2.0, Configuration System V1.0

---

## 1. Назначение

**Shell** — семантический профиль токена, описывающий *чем* является сущность в восьми измерениях восприятия. Shell не хранится внутри Token (Token V5.2 остаётся 64 байта). Shell — это внешний кэш, вычисляемый из совокупности активных Connection токена.

Shell отвечает на вопрос: «Что собой представляет эта сущность?»  
Token отвечает на вопрос: «Где она находится и как себя ведёт?»  
Connection отвечает на вопрос: «С кем она связана?»

Shell — эмерджентное свойство. Токен без связей имеет нулевой Shell. Токен с богатой сетью связей приобретает выразительный семантический профиль. Смысл возникает из отношений, а не из свойств.

---

## 2. Семантические слои (L1–L8)

Восемь слоёв описывают восемь ортогональных измерений восприятия. Они не совпадают с октантами ASHTI (доменами 1-8) и не зависят от них.

Октанты — это *среды обработки* (где и как обрабатывается паттерн).  
Слои — это *измерения существования* (чем является сущность).

Один и тот же токен может быть сильно физическим (L1 высокий) и одновременно обрабатываться в домене LOGIC(6). Это ортогональные координаты.

### 2.1 Таблица слоёв

| Слой | Название | Описание | Пример высокого значения |
|------|----------|----------|--------------------------|
| **L1** | Physical | Материальность, телесность, вещественность | Камень, рука, дождь |
| **L2** | Sensory | Ощущения, восприятие, качества | Сладкий, яркий, громкий |
| **L3** | Motor | Действие, движение, процесс | Бежать, строить, разрушать |
| **L4** | Emotional | Эмоция, аффект, переживание | Радость, страх, гнев |
| **L5** | Cognitive | Мышление, знание, структура | Теорема, алгоритм, категория |
| **L6** | Social | Отношения, роли, нормы | Друг, закон, иерархия |
| **L7** | Temporal | Время, последовательность, ритм | Вчера, цикл, эпоха |
| **L8** | Abstract | Абстракция, символ, метафора | Бесконечность, свобода, число |

### 2.2 Формат значений

Каждый слой — `u8`, диапазон 0..255.

- **0** — отсутствие проявления на данном уровне.
- **255** — максимальная насыщенность.
- Отрицательных значений нет. Слои описывают *степень проявления*, а не полярность.

## 3. Структура кэша

### 3.1 Запись Shell

```rust
pub type ShellProfile = [u8; 8];  // L1, L2, L3, L4, L5, L6, L7, L8
```

8 байт. Одна кэш-линия процессора (64 байта) вмещает 8 профилей Shell.

### 3.2 Хранение

Shell хранится в отдельном массиве, параллельном массиву токенов домена:

```rust
pub struct DomainShellCache {
    profiles: Vec<ShellProfile>,    // Индексируется по позиции токена в домене
    dirty_flags: BitVec,            // Помечает токены, чьи Shell требуют пересчёта
    generation: u64,                // Монотонный счётчик reconciliation
}
```

Индексация: `profiles[token_index]` соответствует токену `domain_state.tokens[token_index]`.

Каждый домен имеет собственный DomainShellCache. Shell одного и того же sutra_id в разных доменах может различаться, потому что связи в каждом домене — свои.

### 3.3 Связь с Token V5.2

Token V5.2 **не изменяется**. Shell — внешняя структура. Токен не знает о своём Shell. Любой модуль, которому нужен семантический профиль токена, обращается к DomainShellCache.

## 4. Справочник семантических вкладов

### 4.1 Назначение

Справочник определяет, какой вклад в семантические слои вносит каждый тип связи. Это таблица отображения:

```
link_type → ShellContribution [L1, L2, L3, L4, L5, L6, L7, L8]
```

где каждое значение — `u8` (0..255), определяющее базовый вклад данного типа связи в соответствующий слой.

### 4.2 Двухуровневая организация

Поле `link_type` в Connection V5.0 — это `u16`, допускающее организацию 256 категорий × 256 типов внутри категории.

Справочник организован в два уровня:

**Уровень 1 — Категория** (старший байт `link_type`, биты 15..8).

Каждая категория имеет базовый профиль вклада. Это грубая семантическая характеристика целого класса связей.

**Уровень 2 — Конкретный тип** (полный `link_type`, оба байта).

Если для конкретного `link_type` задано переопределение — оно используется вместо базового профиля категории. Если не задано — наследуется профиль категории.

### 4.3 Пример

```yaml
# Schema Configuration: semantic_contributions.yaml

categories:
  0x01:  # Structural (Part_Of, Contains, Member_Of...)
    name: "Structural"
    base_profile: [20, 5, 0, 0, 5, 0, 0, 0]
    #               L1  L2  L3  L4  L5  L6  L7  L8
    
  0x02:  # Semantic (Synonym, Antonym, Hypernym...)
    name: "Semantic"
    base_profile: [0, 0, 0, 0, 15, 0, 0, 10]
    
  0x03:  # Causal (Cause, Effect, Enables...)
    name: "Causal"
    base_profile: [0, 0, 5, 0, 15, 0, 10, 8]
    
  0x04:  # Experiential (Feels_Like, Tastes_Like, Sounds_Like...)
    name: "Experiential"
    base_profile: [5, 20, 0, 15, 0, 0, 0, 0]
    
  0x05:  # Social (Friend_Of, Reports_To, Belongs_To_Group...)
    name: "Social"
    base_profile: [0, 0, 0, 5, 0, 25, 0, 0]
    
  0x06:  # Temporal (Precedes, Follows, During...)
    name: "Temporal"
    base_profile: [0, 0, 0, 0, 5, 0, 25, 0]
    
  0x07:  # Motor (Used_For, Performed_By, Requires_Action...)
    name: "Motor"
    base_profile: [10, 0, 25, 0, 5, 0, 0, 0]

# Переопределения конкретных типов
overrides:
  0x0302:  # Emotional_Cause (категория Causal, но с эмоциональным вкладом)
    name: "Emotional_Cause"
    profile: [0, 0, 0, 20, 10, 0, 10, 5]
    
  0x0303:  # Physical_Cause
    name: "Physical_Cause"
    profile: [15, 5, 10, 0, 10, 0, 8, 0]
    
  0x0406:  # Aesthetic_Feel (категория Experiential, но с абстрактным оттенком)
    name: "Aesthetic_Feel"
    profile: [0, 15, 0, 10, 0, 0, 0, 15]
```

### 4.4 Загрузка и хранение

Справочник — часть Schema Configuration (Configuration System V1.0). Загружается из YAML при старте системы. Не изменяется в рантайме.

```rust
pub struct SemanticContributionTable {
    categories: [ShellProfile; 256],        // Базовые профили категорий
    overrides: HashMap<u16, ShellProfile>,   // Переопределения конкретных типов
}

impl SemanticContributionTable {
    pub fn get(&self, link_type: u16) -> &ShellProfile {
        // Сначала проверяем переопределение для конкретного типа
        if let Some(profile) = self.overrides.get(&link_type) {
            return profile;
        }
        // Иначе возвращаем базовый профиль категории
        let category = (link_type >> 8) as usize;
        &self.categories[category]
    }
}
```

### 4.5 Эволюция справочника

Двухуровневая организация позволяет расти без ограничений:

- Новые категории добавляются правкой YAML. Категории без записи имеют нулевой профиль (связь не вносит семантического вклада).
- Новые переопределения добавляются для конкретных `link_type`, когда базовый профиль категории недостаточно точен.
- В будущем DREAM(7) или EXPERIENCE(9) могут предлагать корректировки профилей на основе накопленного опыта. Это потребует отдельной спецификации адаптивного справочника.

## 5. Алгоритм вычисления Shell

### 5.1 Полный пересчёт (Full Recompute)

Полный пересчёт Shell для токена T в домене D:

```
1. Найти все активные Connection, где source_id = T.sutra_id 
   или target_id = T.sutra_id, и domain_id = D.

2. Для каждой Connection:
   a. Получить contribution = SemanticContributionTable.get(link_type)
   b. Получить weight = connection.strength (f32, нормализованный в 0.0..1.0)
   c. Для каждого слоя i (0..7):
      accumulator[i] += contribution[i] * weight

3. Нормализовать accumulator:
   a. Найти max_value = максимум среди accumulator[0..7]
   b. Если max_value > 255:
      factor = 255.0 / max_value
      Для каждого слоя i: accumulator[i] *= factor
   c. Округлить каждый accumulator[i] до u8.

4. Записать результат в profiles[token_index].
```

### 5.2 Нормализация

Пропорциональное сжатие сохраняет *профиль* — соотношение между слоями. Токен, у которого L1 в два раза сильнее L5, сохранит это соотношение независимо от количества связей.

**Открытый вопрос (помечен для будущих версий):** нормализация стирает абсолютную величину. Токен с 3 связями и токен с 300 связями могут получить одинаковый Shell, если пропорции совпадают. В будущем может быть добавлено поле `density: u8`, отражающее суммарный вес до нормализации. Это расширит Shell до 9 байт.

### 5.3 Сложность

Полный пересчёт одного Shell — O(K), где K — количество связей токена. Для типичного токена (5-20 связей) это тривиальная операция. Для хаб-токенов (сотни связей) — заметная, но всё равно линейная.

## 6. Инкрементальное обновление

Полный пересчёт каждого Shell при каждом событии — расточительно. Большинство событий затрагивают одну-две связи, а не всю сеть. Инкрементальное обновление пересчитывает Shell только для затронутых токенов.

### 6.1 Триггеры обновления

| Событие | Затронутые токены | Действие |
|---------|-------------------|----------|
| Connection создана | source, target | Полный пересчёт Shell обоих |
| Connection удалена | source, target | Полный пересчёт Shell обоих |
| Connection.strength изменился | source, target | Полный пересчёт Shell обоих |
| Connection.link_type изменился | source, target | Полный пересчёт Shell обоих |
| Connection.flags: ACTIVE снят | source, target | Полный пересчёт Shell обоих |
| Token создан (без связей) | новый токен | Shell = [0, 0, 0, 0, 0, 0, 0, 0] |
| Token удалён | — | Запись Shell обнуляется |

### 6.2 Механизм

При наступлении триггерного события:

```
1. Определить затронутые токены (source_id, target_id из Connection).
2. Пометить их в dirty_flags.
3. Выполнить полный пересчёт Shell для помеченных токенов.
4. Снять dirty_flags.
```

Почему полный пересчёт, а не дельта-обновление: дельта-обновление (прибавить/вычесть вклад одной связи) накапливает ошибки округления, потому что Shell хранится в u8. После нескольких десятков дельт профиль может "уплыть" от истинного значения. Полный пересчёт для одного токена (5-20 связей) достаточно дешёв, чтобы не экономить на точности.

### 6.3 Интеграция с Causal Frontier

Инкрементальное обновление Shell естественно вписывается в цикл Causal Frontier:

```
Connection событие (создание, удаление, изменение)
    ↓
apply_event() — обновляет Connection
    ↓
collect_neighbors() — добавляет source и target в frontier
    ↓
evaluate_local_rules(token) — среди прочего, пересчитывает Shell
    ↓
обновлённый Shell записывается в DomainShellCache
```

Shell не генерирует собственных событий. Это побочный эффект обработки Connection-событий. Изменение Shell не фиксируется в COM — это кэш, а не состояние.

## 7. Reconciliation через Heartbeat

### 7.1 Проблема

Инкрементальное обновление корректно в теории, но на практике возможны ситуации, когда Shell рассинхронизируется с реальностью:

- Баг в обработчике пропустил обновление.
- Восстановление из snapshot не включало Shell-кэш.
- Новая версия SemanticContributionTable загружена, но Shell не пересчитаны.

Reconciliation — периодическая проверка и коррекция Shell-кэша.

### 7.2 Механизм

Reconciliation встраивается в существующий механизм Heartbeat V2.0. Heartbeat уже добавляет токены в Causal Frontier батчами (раздел 6 Heartbeat V2.0). При обработке токена из Heartbeat-батча, помимо decay и термодинамики, выполняется пересчёт Shell:

```
Heartbeat → frontier.push_token(token_idx)
    ↓
Frontier.pop() → evaluate_local_rules(token)
    ↓
    ├── decay check (существующее)
    ├── gravity check (существующее)
    ├── thermodynamics check (существующее)
    └── shell reconciliation (новое)
```

Shell reconciliation:

```
1. Вычислить Shell заново (полный пересчёт).
2. Сравнить с текущим значением в кэше.
3. Если различаются — записать новое значение.
```

### 7.3 Гарантия полного покрытия

Heartbeat V2.0, раздел 6.2:

```
token_index = (pulse_number * batch_size + offset) % total_tokens
```

За `ceil(total_tokens / batch_size)` пульсов каждый токен будет проверен хотя бы один раз. Shell reconciliation наследует эту гарантию.

### 7.4 Конфигурация

В HeartbeatConfig добавляется один флаг:

```rust
pub struct HeartbeatConfig {
    // ... существующие поля из Heartbeat V2.0 ...
    pub enable_shell_reconciliation: bool,  // Активировать проверку Shell на Heartbeat
}
```

По умолчанию: `true`. Может быть отключён на слабом оборудовании, если инкрементальное обновление достаточно надёжно.

## 8. Потребители Shell

Shell — read-only кэш. Его читают, но никогда не пишут напрямую (только через вычисление из связей).

### 8.1 EXPERIENCE (Домен 9)

Резонансный поиск в домене 9 может использовать Shell для быстрого сравнения входящего паттерна с существующими следами. Два токена с похожими Shell-профилями — кандидаты на ассоциацию.

### 8.2 ASHTI (Домены 1-8)

Домены-процессоры могут учитывать Shell при обработке паттернов. Например, LOGIC(6) может отдавать приоритет токенам с высоким L5 (Cognitive), а DREAM(7) — токенам с высоким L8 (Abstract).

### 8.3 UPO (визуализация)

UPO может отображать Shell как цветовой профиль токена. Каждый слой — компонент цвета или иное визуальное свойство.

### 8.4 Arbiter

Arbiter может использовать Shell при формировании подсказок для доменов 1-8: фильтровать ассоциации по семантической близости к контексту.

## 9. Инварианты

1. **Token V5.2 не изменяется.** Shell — внешний кэш, не встроенный в токен.
2. **Shell вычисляется, не назначается.** Единственный способ изменить Shell — изменить связи токена.
3. **Нормализация сохраняет профиль.** Соотношение между слоями неизменно при любом количестве связей.
4. **Нулевой Shell — корректное состояние.** Токен без активных связей имеет Shell `[0, 0, 0, 0, 0, 0, 0, 0]`.
5. **Домен-локальность.** Shell вычисляется из связей внутри одного домена. Shell одного sutra_id в разных доменах — независимы.
6. **Детерминизм.** При одинаковом наборе Connection и одинаковом SemanticContributionTable, Shell вычисляется одинаково.
7. **Shell — не причинность.** Изменение Shell не генерирует COM-события. Это производная величина, кэш.

## 10. Ограничения и компромиссы

### 10.1 Потеря абсолютной величины

Нормализация стирает информацию о "богатстве" связей. Токен с 3 связями и токен с 300 могут иметь одинаковый профиль. Помечено для решения в Shell V3.1 (поле `density`).

### 10.2 Зависимость от справочника

Качество Shell целиком определяется качеством SemanticContributionTable. Плохо настроенный справочник даёт бессмысленные профили. Справочник требует итеративной калибровки.

### 10.3 Только активные связи

Shell учитывает только связи с флагом `ACTIVE`. Ингибированные или временно отключённые связи не вносят вклад. Это может быть как преимуществом (Shell отражает текущее состояние), так и ограничением (история неактивных связей невидима).

### 10.4 Один домен — один Shell

Shell токена в домене EXECUTION(1) может отличаться от его Shell в LOGIC(6), потому что связи в этих доменах — разные. Это корректно архитектурно (один и тот же смысл проявляется по-разному в разных средах), но усложняет кросс-доменное сравнение.

## 12. Взаимодействия

### 12.1 С Token V5.2

Shell — параллельная структура. Token не знает о Shell. Потребители читают Token для позиции/массы/температуры и Shell для семантического профиля.

### 12.2 С Connection V5.0

Connection — единственный источник данных для вычисления Shell. Поля `link_type`, `strength`, `flags` (ACTIVE) используются напрямую.

### 12.3 С DomainConfig V2.1

DomainConfig не содержит Shell-специфичных полей. Shell определяется связями, а не конфигурацией домена. Домен влияет на Shell косвенно — через набор связей, которые в нём существуют.

### 12.4 С Heartbeat V2.0

Heartbeat обеспечивает reconciliation Shell. В HeartbeatConfig добавляется флаг `enable_shell_reconciliation`.

### 12.5 С Configuration System V1.0

SemanticContributionTable загружается как часть Schema Configuration при старте системы.

### 12.6 С Causal Frontier V1

Инкрементальное обновление Shell встраивается в цикл обработки Frontier. Shell пересчитывается при обработке токена, попавшего во Frontier из-за Connection-события.

## 13. Жизненный цикл Shell

1. **Инициализация.** При создании домена DomainShellCache инициализируется нулевыми профилями.
2. **Первичное вычисление.** При загрузке состояния (из snapshot или при импорте) — полный пересчёт Shell для всех токенов домена.
3. **Инкрементальное обновление.** При каждом Connection-событии — пересчёт Shell затронутых токенов.
4. **Reconciliation.** На каждом Heartbeat — проверка и коррекция батча токенов.
5. **Snapshot.** Shell-кэш может быть сериализован в snapshot, но не обязан: он может быть полностью пересчитан из Connection при восстановлении. Сериализация — оптимизация, ускоряющая загрузку.

## 14. История изменений

- **V3.0**: Полная переработка концепции. Shell — эмерджентный кэш, вычисляемый из Connection, а не фиксированное свойство токена. Восемь слоёв L1-L8 (Physical, Sensory, Motor, Emotional, Cognitive, Social, Temporal, Abstract). Двухуровневый справочник семантических вкладов. Инкрементальное обновление. Reconciliation через Heartbeat.
- **V2.0 (SemanticShell)**: Предыдущая концепция. Семантические слои как свойства токена. Заменена V3.0.
- **V1.x**: Ранние версии с фиксированными оболочками в NeuroGraph.
