# AXIOM MODULE SPECIFICATION: SPACE V6.1

**Статус:** Актуальная спецификация (core)  
**Версия:** 6.1.0  
**Дата:** 2026-05-12  
**Назначение:** Пространственная модель доменов: координаты, индексация, вычисление расстояний, обнаружение соседей  
**Модель времени:** COM `event_id` (причинный порядок, u64)  
**Связанные спеки:** Token V5.2, Connection V5.0, DomainConfig V2.1, Domain V1.3, Event-Driven V1, Causal Frontier V1, Heartbeat V2.0, COM V1.0, Shell V3.0

---

## 1. Назначение

**SPACE** определяет, как Token существуют в пространстве Domain: систему координат, пространственный индекс для поиска соседей, вычисление расстояний и механику пространственных взаимодействий.

SPACE — не отдельный модуль, а набор правил и структур данных, работающих внутри каждого Domain. Каждый домен имеет своё изолированное пространство с собственной конфигурацией (field_size, gravity_strength и т.д. из DomainConfig V2.1).

SPACE не управляет временем. Все пространственные изменения происходят через COM-события в рамках Event-Driven модели.

---

## 2. Фундаментальные принципы

**Разреженность.** Пространственный индекс содержит только активные токены. Пустые области не потребляют ресурсов.

**Локальность.** Взаимодействия обнаруживаются только через пространственный хэш. Глобальные проходы по всем токенам запрещены (Causal Frontier V1, раздел 2.2).

**Детерминизм.** Целочисленная арифметика, фиксированный порядок обхода. Бит-в-бит одинаковый результат на любой платформе.

**Zero-alloc.** Все структуры данных предварительно выделены. Во время обработки событий аллокаций не происходит.

**Event-Driven.** Пространство не обновляется "каждый тик". Позиции токенов меняются только через COM-события (ShellMoved, ShellMerged и т.п.). Пространственный индекс перестраивается после применения пространственных событий.

## 3. Система координат

### 3.1 Формат координат

Token V5.2 определяет:

```rust
pub position: [i16; 3],  // Координаты в текущем Field (X, Y, Z)
pub velocity: [i16; 3],  // Вектор скорости (инерция движения)
pub target: [i16; 3],    // Целевая точка (аттрактор)
```

Координаты — знаковые 16-битные целые числа. Диапазон каждой оси: -32768..+32767.

Это целочисленное квантованное пространство. Каждая единица — один квант. Конкретная "метрика" кванта определяется масштабом Field домена.

### 3.2 Привязка к DomainConfig

DomainConfig V2.1 определяет физические размеры поля:

```rust
pub field_size: [f32; 3],  // Размеры поля (X, Y, Z) в абстрактных единицах
```

Отношение `field_size` к диапазону i16 определяет масштаб кванта:

```
quant_size_x = field_size[0] / 65536.0
quant_size_y = field_size[1] / 65536.0
quant_size_z = field_size[2] / 65536.0
```

Это отношение используется только в адаптерах (визуализация, ввод-вывод). Внутренняя физика работает в квантах (i16), не в абстрактных единицах.

### 3.3 Начало координат

Точка `(0, 0, 0)` — Anchor (сутра-центр) домена (Domain V1.3, раздел 2.1). Это гравитационный центр, к которому стремятся токены в доменах с положительной гравитацией.

### 3.4 Целочисленная арифметика

Все пространственные вычисления выполняются в целых числах. Промежуточные результаты используют i32 или i64 для предотвращения переполнения:

```rust
// Разность координат: i16 - i16 может выйти за i16, используем i32
let dx: i32 = a.position[0] as i32 - b.position[0] as i32;
```

Операции с плавающей точкой запрещены в пространственных вычислениях ядра. Это гарантирует детерминизм на всех платформах.

## 4. Пространственный хэш (Spatial Hash Grid)

### 4.1 Назначение

Пространственный хэш — структура данных для быстрого поиска соседей токена. Это реализация spatial index, упомянутого в Causal Frontier V1 (раздел 7).

Вместо перебора всех токенов домена (O(N²)), хэш позволяет найти токены в окрестности за O(1) в среднем.

### 4.2 Параметры

```rust
/// Размер ячейки в квантах (степень двойки для битового сдвига)
/// При CELL_SHIFT = 8: ячейка = 256 квантов, пространство делится на 256 × 256 × 256 ячеек
const CELL_SHIFT: u32 = 8;
const CELL_SIZE: i16 = 1 << CELL_SHIFT;  // 256 квантов

/// Количество корзин (степень двойки для битовой маски)
const BUCKET_COUNT: usize = 1 << 16;  // 65536 корзин
const BUCKET_MASK: u32 = (BUCKET_COUNT - 1) as u32;
```

`CELL_SHIFT` определяет гранулярность: при значении 8, пространство i16 (-32768..+32767) делится на ячейки по 256 квантов, что даёт 256 ячеек по каждой оси. Значение настраивается через Runtime Configuration.

### 4.3 Хэш-функция

```rust
fn cell_key(position: [i16; 3]) -> u32 {
    let ix = (position[0] as i32) >> CELL_SHIFT;
    let iy = (position[1] as i32) >> CELL_SHIFT;
    let iz = (position[2] as i32) >> CELL_SHIFT;
    
    let h = (ix.wrapping_mul(73856093))
          ^ (iy.wrapping_mul(19349663))
          ^ (iz.wrapping_mul(83492791));
    
    (h as u32) & BUCKET_MASK
}
```

Три простых числа обеспечивают равномерное распределение. Битовая маска вместо остатка от деления — быстрее на всех платформах.

### 4.4 Структура (плоская, zero-alloc)

```rust
pub struct SpatialHashGrid {
    bucket_heads: Vec<u32>,     // Размер BUCKET_COUNT, начальное значение u32::MAX (пусто)
    entries: Vec<CellEntry>,    // Пул записей, предвыделен
    entry_count: u32,           // Текущее количество записей
}

struct CellEntry {
    token_index: u32,           // Индекс токена в массиве домена
    next: u32,                  // Следующая запись в корзине (u32::MAX = конец)
}
```

Все структуры предвыделены при инициализации домена. Размер пула entries равен `token_capacity` из DomainConfig V2.1.

### 4.5 Перестройка индекса

Пространственный хэш перестраивается после применения пространственных событий. Перестройка — O(N), где N — количество активных токенов:

```
1. Обнулить bucket_heads (заполнить u32::MAX).
2. Сбросить entry_count в 0.
3. Для каждого активного токена:
   a. Вычислить bucket = cell_key(token.position)
   b. Взять запись из пула: entries[entry_count]
   c. Заполнить: token_index, next = bucket_heads[bucket]
   d. bucket_heads[bucket] = entry_count
   e. entry_count += 1
```

Перестройка выполняется не при каждом событии, а батчами — после обработки группы событий в Causal Frontier. Частота перестройки — параметр конфигурации.

### 4.6 Поиск соседей

```rust
fn find_neighbors(
    grid: &SpatialHashGrid,
    position: [i16; 3],
    radius: i16,
    tokens: &[Token],
    result: &mut Vec<u32>,  // Индексы найденных токенов
) {
    let radius_cells = (radius as i32 >> CELL_SHIFT) + 1;
    let center_ix = (position[0] as i32) >> CELL_SHIFT;
    let center_iy = (position[1] as i32) >> CELL_SHIFT;
    let center_iz = (position[2] as i32) >> CELL_SHIFT;
    
    let radius2 = (radius as i32) * (radius as i32);
    
    // Обход ячеек в окрестности
    for dix in -radius_cells..=radius_cells {
        for diy in -radius_cells..=radius_cells {
            for diz in -radius_cells..=radius_cells {
                let cell_pos = [
                    ((center_ix + dix) << CELL_SHIFT) as i16,
                    ((center_iy + diy) << CELL_SHIFT) as i16,
                    ((center_iz + diz) << CELL_SHIFT) as i16,
                ];
                let bucket = cell_key(cell_pos);
                
                let mut idx = grid.bucket_heads[bucket as usize];
                while idx != u32::MAX {
                    let entry = &grid.entries[idx as usize];
                    let t = &tokens[entry.token_index as usize];
                    
                    if distance2(position, t.position) <= radius2 {
                        result.push(entry.token_index);
                    }
                    
                    idx = entry.next;
                }
            }
        }
    }
}
```

### 4.7 Каждый домен — свой индекс

Пространственный хэш — компонент DomainState. Каждый домен имеет собственный SpatialHashGrid. Токены одного домена не видят токены другого через пространственный хэш.

## 5. Вычисление расстояний

### 5.1 Квадрат расстояния (без корня)

```rust
fn distance2(a: [i16; 3], b: [i16; 3]) -> i32 {
    let dx = a[0] as i32 - b[0] as i32;
    let dy = a[1] as i32 - b[1] as i32;
    let dz = a[2] as i32 - b[2] as i32;
    dx * dx + dy * dy + dz * dz
}
```

Результат — i32 (максимум: 3 × 32768² = 3 221 225 472, вмещается в i32 со знаком: нет, максимум i32 = 2 147 483 647). Корректировка: максимальное значение превышает i32. Используем u32 или i64:

```rust
fn distance2(a: [i16; 3], b: [i16; 3]) -> i64 {
    let dx = a[0] as i64 - b[0] as i64;
    let dy = a[1] as i64 - b[1] as i64;
    let dz = a[2] as i64 - b[2] as i64;
    dx * dx + dy * dy + dz * dz
}
```

Результат — i64. Максимум: 3 × 32768² = 3 221 225 472, вмещается.

### 5.2 Сравнение расстояний

Все пороги (радиус столкновения, радиус взаимодействия) хранятся как квадрат радиуса в i64. Сравнение:

```rust
if distance2(a.position, b.position) < collision_radius2 {
    // Столкновение
}
```

Извлечение квадратного корня не требуется ни для каких пространственных проверок.

## 6. Пространственные события

SPACE не генерирует события самостоятельно. Пространственные события генерируются логикой Causal Frontier при обработке токенов:

### 6.1 Типы пространственных событий

**TokenMoved** — позиция токена изменилась (гравитация, инерция, внешняя сила).

```rust
TokenMoved {
    token_index: u32,
    old_position: [i16; 3],
    new_position: [i16; 3],
}
```

**TokenCollision** — два токена оказались в пределах радиуса столкновения.

```rust
TokenCollision {
    token_a: u32,
    token_b: u32,
    distance2: i64,
}
```

**TokenEnteredCell** — токен перешёл в новую ячейку пространственного хэша (триггер для перестройки индекса).

Все пространственные события проходят через COM и получают `event_id`.

### 6.2 Интеграция с Causal Frontier

Пространственные проверки — часть `evaluate_local_rules()` в цикле Causal Frontier:

```
Frontier.pop() → token
    ↓
evaluate_local_rules(token):
    ├── decay check
    ├── gravity check → может сгенерировать TokenMoved
    ├── neighbor check → может сгенерировать TokenCollision
    ├── thermodynamics check
    └── shell reconciliation (Shell V3.0)
    ↓
Если TokenMoved → apply_event → обновить position → 
    → добавить соседей нового положения в frontier
```

### 6.3 Интеграция с Heartbeat

Heartbeat V2.0 периодически добавляет токены в Causal Frontier. Это обеспечивает, что даже токены, не затронутые внешними событиями, проверяются на пространственные изменения (гравитационный дрейф, инерционное движение).

## 7. Гравитация

### 7.1 Модель

Гравитация в домене определяется `gravity_strength` из DomainConfig V2.1. Токены притягиваются к Anchor (0, 0, 0) с силой, пропорциональной `gravity_strength` и `mass` токена.

### 7.2 Вычисление (целочисленное)

```rust
fn compute_gravity(
    token: &Token,
    gravity_strength: f32,  // Из DomainConfig, конвертируется в i32 при загрузке
    gravity_i: i32,          // gravity_strength, масштабированный в целочисленное
) -> [i16; 3] {
    // Вектор к Anchor (0,0,0)
    let dx = -(token.position[0] as i32);
    let dy = -(token.position[1] as i32);
    let dz = -(token.position[2] as i32);
    
    // Сила пропорциональна расстоянию и массе
    let mass = token.mass as i32;
    
    // Ускорение = gravity * mass * direction / |direction|
    // Упрощение: линейная гравитация (сила пропорциональна расстоянию)
    let ax = ((dx as i64 * gravity_i as i64 * mass as i64) >> 24) as i16;
    let ay = ((dy as i64 * gravity_i as i64 * mass as i64) >> 24) as i16;
    let az = ((dz as i64 * gravity_i as i64 * mass as i64) >> 24) as i16;
    
    [ax, ay, az]
}
```

Конкретная формула гравитации (линейная, обратноквадратичная) — параметр конфигурации домена. Выше показан простейший вариант (линейная). Shift `>> 24` — нормализация, подбирается при калибровке.

### 7.3 Применение

Гравитация вычисляется при обработке токена в Causal Frontier. Если результирующее ускорение ненулевое, генерируется событие TokenMoved.

## 8. Инерция и движение

### 8.1 Модель

Token V5.2 хранит `velocity: [i16; 3]` и `momentum: [i32; 3]`.

- `velocity` — текущая скорость (изменение position за шаг).
- `momentum` — накопленный импульс (для более точного интегрирования).

### 8.2 Обновление позиции

При событии движения:

```rust
// Применить ускорение к скорости
new_velocity[i] = clamp_i16(
    token.velocity[i] as i32 + acceleration[i] as i32
);

// Применить трение (из DomainConfig)
// friction_coeff: u8 (0..255 → 0.0..1.0)
// Чем выше трение, тем сильнее замедление
let friction = config.friction_coeff as i32;  // 0..255
new_velocity[i] = ((new_velocity[i] as i32 * (255 - friction)) >> 8) as i16;

// Применить скорость к позиции
new_position[i] = clamp_i16(
    token.position[i] as i32 + new_velocity[i] as i32
);
```

Функция `clamp_i16` ограничивает результат диапазоном i16, предотвращая переполнение.

### 8.3 Движение к цели (аттрактор)

Token V5.2 хранит `target: [i16; 3]`. Если target задан (не (0,0,0) или специальный маркер), токен испытывает дополнительную силу притяжения к target. Это используется для направленного движения при обработке паттернов.

## 9. Столкновения

### 9.1 Обнаружение

При обработке токена в Causal Frontier, пространственный хэш используется для поиска соседей в радиусе столкновения. Если `distance2(a, b) < collision_radius2` — генерируется событие TokenCollision.

Радиус столкновения может быть глобальным параметром домена или зависеть от mass токенов.

### 9.2 Обработка

Обработчик столкновения — plugin в Domain. Возможные стратегии:

- **Отталкивание.** Токены получают скорость, направленную друг от друга.
- **Слияние.** Два токена объединяются (один поглощает другой). Новая позиция — центр масс. Связи обоих токенов переназначаются.
- **Резонанс.** Если link_type допускает — создаётся Connection между столкнувшимися токенами.
- **Игнорирование.** Столкновение зафиксировано, но не обработано (для статистики).

Конкретная стратегия определяется конфигурацией домена. Это не часть SPACE — SPACE только обнаруживает пространственные отношения.

## 10. Фрактальная иерархия доменов

### 10.1 Принцип

Каждый домен имеет собственное пространство с собственными координатами. Ashti_Core V2.0 определяет 11 доменов, каждый со своим Field. Координаты токена в EXECUTION(1) не имеют отношения к координатам токена в DREAM(7).

### 10.2 Междоменные переходы

Когда токен (точнее, ссылка на sutra_id) передаётся из одного домена в другой (через шины данных, раздел 2.6 Ashti_Core V2.0), он получает новую позицию в пространстве целевого домена. Исходная позиция не копируется — она не имеет смысла в другом пространстве.

Начальная позиция в целевом домене определяется мембраной домена (точка входа) или механизмом инъекции паттерна.

### 10.3 Масштабирование (будущее)

В будущем возможна иерархия доменов (родительский-дочерний), где координаты преобразуются между уровнями. Коэффициент масштабирования задаётся как степень двойки для оптимизации:

```rust
child_abs_pos = parent_abs_pos + (child_local_pos >> SCALE_SHIFT);
```

Это направление развития, не реализуемое в V6.0.

## 11. Конфигурация

### 11.1 Runtime Configuration

```yaml
space:
  cell_shift: 8               # Размер ячейки = 2^cell_shift квантов (256)
  bucket_count_log2: 16        # Количество корзин = 2^16 (65536)
  rebuild_frequency: 10        # Перестройка хэша каждые N событий
  collision_radius: 100        # Радиус столкновения в квантах (i16)
  gravity_model: "linear"      # linear | inverse_square
  gravity_scale_shift: 24      # Нормализация гравитационной формулы
```

### 11.2 Примеры для разного оборудования

Слабое оборудование (цель: минимальная нагрузка):

```yaml
space:
  cell_shift: 10              # Крупные ячейки (1024 кванта) — меньше корзин
  bucket_count_log2: 12       # 4096 корзин
  rebuild_frequency: 50       # Редкая перестройка
```

Среднее оборудование:

```yaml
space:
  cell_shift: 8
  bucket_count_log2: 16
  rebuild_frequency: 10
```

Мощное оборудование:

```yaml
space:
  cell_shift: 6               # Мелкие ячейки (64 кванта) — точный поиск
  bucket_count_log2: 20       # 1M корзин
  rebuild_frequency: 1        # Перестройка после каждого события
```

## 12. Инварианты

1. **Целочисленная арифметика.** Все пространственные вычисления выполняются в целых числах (i16, i32, i64). Операции с плавающей точкой запрещены в ядре пространственных вычислений.
2. **Детерминизм.** Одинаковые входные данные (позиции токенов, конфигурация) дают бит-в-бит одинаковый результат на любой платформе.
3. **Zero-alloc.** Все структуры предвыделены при инициализации домена. Во время обработки событий аллокаций не происходит.
4. **Локальность.** Поиск соседей — только через пространственный хэш. Глобальные проходы запрещены.
5. **Доменная изоляция.** Каждый домен имеет собственный SpatialHashGrid. Пространства доменов не пересекаются.
6. **COM совместимость.** Все пространственные изменения генерируют COM-события с event_id.
7. **Event-Driven.** Позиции обновляются только через события. Нет "тиков" или "кадров".

## 13. Взаимодействия

### 13.1 С Token V5.2

Token хранит position, velocity, target, mass, momentum. SPACE читает эти поля для вычислений и записывает обновлённые значения через COM-события.

### 13.2 С DomainConfig V2.1

field_size определяет масштаб пространства. gravity_strength, friction_coeff, viscosity, elasticity, temperature — параметры физики. Все читаются как read-only.

### 13.3 С Connection V5.0

Connection хранит ideal_dist — целевое расстояние между связанными токенами. SPACE может использовать это для генерации сил притяжения/отталкивания между связанными токенами ("пружинная" динамика).

### 13.4 С Shell V3.0

Shell не участвует в пространственных вычислениях напрямую. Но результаты пространственных взаимодействий (новые Connection после столкновения) могут вызвать пересчёт Shell затронутых токенов.

### 13.5 С Causal Frontier V1

SpatialHashGrid — реализация spatial index из Causal Frontier V1 (раздел 7). Frontier использует пространственный хэш для поиска соседей при `collect_neighbors()`.

### 13.6 С Heartbeat V2.0

Heartbeat обеспечивает периодическую проверку токенов. При обработке Heartbeat-батча, пространственные проверки (гравитация, инерция) выполняются для токенов, не затронутых недавними событиями.

### 13.7 С COM V1.0

Все пространственные изменения (движение, столкновение) генерируют COM-события, обеспечивая детерминизм и воспроизводимость.

## 14. Ограничения и компромиссы

### 14.1 Диапазон координат

i16 даёт диапазон -32768..+32767 = 65536 квантов по оси. Это меньше, чем i32 Q16.16 в предыдущих версиях (SPACE V5.0). Компромисс ради компактного 64-байтного Token. Для большинства задач 65K квантов достаточно. При необходимости большего пространства — использовать иерархию доменов (раздел 10.3).

### 14.2 Коллизии хэша

При большом количестве токенов в одной ячейке поиск соседей деградирует до O(K) для каждой корзины, где K — токенов в корзине. Решение: уменьшить CELL_SHIFT (мельче ячейки) или увеличить bucket_count. Параметры подбираются при профилировании.

### 14.3 Перестройка индекса

Полная перестройка хэша — O(N). При большом количестве токенов и частых событиях это может стать узким местом. В будущем возможна инкрементальная перестройка (обновление только затронутых ячеек).

## 15. SIMD и батчевая физика (Axiom Sentinel V1.1)

### 15.1 apply_gravity_batch

```rust
pub fn apply_gravity_batch(
    positions: &[[i16; 3]],
    masses: &[u8],
    gravity_scale_shift: u8,
    model: GravityModel,
) -> GravityBatchResult
```

Файл: `crates/axiom-space/src/simd.rs`

Батчевое вычисление гравитационных сил для массива токенов. Обрабатывает весь слайс за один проход. При `target-cpu=native` (S4) компилятор авто-векторизует до AVX2 (256-бит), достигая ~80–120M токенов/сек.

Ограничение: безопасно для любого N, но при N > `L2_CHUNK_TOKENS` весь входной массив не помещается в L2-кэш, что вызывает деградацию производительности.

### 15.2 apply_gravity_batch_chunked (S3)

```rust
pub const L2_CHUNK_TOKENS: usize = 65536;  // 512 KB / 8 bytes per token

pub fn apply_gravity_batch_chunked(
    positions: &[[i16; 3]],
    masses: &[u8],
    gravity_scale_shift: u8,
    model: GravityModel,
) -> GravityBatchResult
```

Файл: `crates/axiom-space/src/simd.rs`; реэкспортируется из `crates/axiom-space/src/lib.rs`.

Обёртка над `apply_gravity_batch` с нарезкой входа на чанки размером `L2_CHUNK_TOKENS`. Каждый чанк обрабатывается в пределах L2-кэша Ryzen 5 3500U (512 KB), предотвращая деградацию скорости при N > 1M.

Результаты чанков накапливаются в единый `GravityBatchResult`. При N ≤ `L2_CHUNK_TOKENS` вызов делегируется прямо в `apply_gravity_batch` без оверхеда.

**Целевой показатель:** 1M токенов за ~8–10 ms (вместо ~25 ms без чанкинга).

## 16. Направления развития

1. **Инкрементальное обновление хэша.** Вместо полной перестройки — обновление только затронутых ячеек при TokenMoved.
2. **Пружинная динамика.** Систематическое использование ideal_dist из Connection для генерации сил между связанными токенами.
3. **Кластеризация.** Автоматическое обнаружение плотных групп токенов и замена взаимодействий внутри кластера на лидерские (оптимизация для >100K токенов).
4. **Иерархия доменов.** Преобразование координат между родительским и дочерним доменом для фрактальных структур.
5. **Явные AVX2 intrinsics (SENT-S4b).** Если `apply_gravity_batch` с авто-векторизацией не достигает цели 8–10 ms — явные intrinsics через `std::arch::x86_64` в `#[cfg(feature = "simd")]`. Scalar fallback остаётся. Статус: DEFERRED до верификации бенчей.
