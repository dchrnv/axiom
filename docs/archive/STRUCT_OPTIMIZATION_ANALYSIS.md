# Анализ оптимизации структур Event и DynamicTrace

## Дата: 2026-03-20

## Текущее состояние

### Event - 56 байт (цель: 32 байт)
**Где:** [event.rs:111](runtime/src/event.rs#L111)

**Layout:**
```rust
pub struct Event {
    // ИДЕНТИФИКАТОР (8 байт)
    pub event_id: u64,        // 8b - Монотонный причинный индекс (COM V1.0)
    pub domain_id: u16,       // 2b - Домен события
    pub event_type: u16,      // 2b - Тип события
    pub priority: u8,         // 1b - Приоритет (0..255)
    pub flags: u8,            // 1b - Флаги
    pub _reserved: [u8; 2],   // 2b - Резерв

    // СОДЕРЖАНИЕ (16 байт)
    pub payload_hash: u64,    // 8b - Хеш содержимого
    pub target_id: u32,       // 4b - ID целевого объекта
    pub source_id: u32,       // 4b - ID источника
    pub payload_size: u32,    // 4b - Размер данных

    // МЕТАДАННЫЕ (16 байт)
    pub parent_event_id: u64, // 8b - Предыдущее событие
    pub pulse_id: u64,        // 8b - Номер пульса Heartbeat
}
```

**Итого:** 8 + 2 + 2 + 1 + 1 + 2 + 8 + 4 + 4 + 4 + 8 + 8 = **40 байт контента**
**Реально:** 56 байт (padding из-за alignment)
**Нужно убрать:** 24 байта

### DynamicTrace - 56 байт (цель: 32 байт)
**Где:** [upo.rs:27](runtime/src/upo.rs#L27)

**Layout:**
```rust
pub struct DynamicTrace {
    // ПРОСТРАНСТВО (12 байт)
    pub x: i32,               // 4b - Координата X
    pub y: i32,               // 4b - Координата Y
    pub z: i32,               // 4b - Координата Z

    // ХАРАКТЕРИСТИКИ (8 байт)
    pub weight: f32,          // 4b - Вес/интенсивность
    pub frequency: f32,       // 4b - Частота колебаний

    // ВРЕМЯ (8 байт)
    pub created_at: u64,      // 8b - COM event_id создания
    pub last_update: u64,     // 8b - COM event_id обновления

    // МЕТАДАННЫЕ (4 байт + padding)
    pub source_type: u8,      // 1b - Источник
    pub source_id: u32,       // 4b - ID источника
    pub flags: u8,            // 1b - ACTIVE/FADING/LOCKED
    pub resonance_class: u8,  // 1b - Класс резонанса
}
```

**Итого:** 4 + 4 + 4 + 4 + 4 + 8 + 8 + 1 + 4 + 1 + 1 = **43 байт контента**
**Реально:** 56 байт (padding)
**Нужно убрать:** 24 байта

## Варианты оптимизации Event (56 → 32 байт)

### Вариант 1: Удалить pulse_id (рекомендуется)
**Убираем:** `pub pulse_id: u64` (8 байт)

**Обоснование:**
- pulse_id связывает событие с Heartbeat
- Можно вычислить из event_id через формулу: `event_id / heartbeat_interval`
- Heartbeat V2.0 использует event_count, не timestamp

**Потенциальные проблемы:**
- Требуется доступ к HeartbeatConfig для вычисления
- Небольшой overhead при запросе pulse_id

**Экономия:** 8 байт → 48 байт total

---

### Вариант 2: Удалить parent_event_id
**Убираем:** `pub parent_event_id: u64` (8 байт)

**Обоснование:**
- Цепочка событий может храниться в COM Timeline отдельно
- Каждое событие имеет event_id - монотонный индекс
- Parent можно вычислить через Timeline lookup

**Потенциальные проблемы:**
- Нарушает самодостаточность Event
- Требует доступ к Timeline для восстановления цепочки
- Может усложнить debugging

**Экономия:** 8 байт → 48 байт total

---

### Вариант 3: Сократить payload_hash до u32
**Изменяем:** `pub payload_hash: u64` → `u32` (экономия 4 байта)

**Обоснование:**
- Для детерминизма достаточно 32-bit hash (CRC32, FNV-1a)
- События короткоживущие, коллизии маловероятны
- 32-bit hash используется в Token/Connection

**Потенциальные проблемы:**
- Теоретически возможны коллизии (вероятность ~1/4B)
- Меньшая криптостойкость (но для Event не критично)

**Экономия:** 4 байта → 52 байт total

---

### Вариант 4: Объединить domain_id и event_type в u32
**Было:** `domain_id: u16` + `event_type: u16` = 4 байта
**Станет:** `packed_info: u32` (domain_id: 16 bits, event_type: 16 bits)

**Обоснование:**
- Логически связанные поля
- Можно упаковать битовыми операциями
- Сохраняет тот же объем информации

**Потенциальные проблемы:**
- Усложняет доступ к полям (требуются bitwise операции)
- Ухудшает читаемость кода

**Экономия:** 0 байт (не помогает, уже packed)

---

### Вариант 5: Удалить payload_size
**Убираем:** `pub payload_size: u32` (4 байта)

**Обоснование:**
- Размер payload известен из EventType
- Каждый тип события имеет фиксированный размер
- Можно получить через `event_type.payload_size()`

**Потенциальные проблемы:**
- Требует маппинг EventType → size
- Может быть неудобно для переменных payload

**Экономия:** 4 байта → 52 байт total

---

### Вариант 6: Комбинированная оптимизация (РЕКОМЕНДУЕТСЯ)

**Удаляем:**
1. `pulse_id: u64` → вычисляется (8 байт)
2. `payload_size: u32` → из EventType (4 байт)
3. `_reserved: [u8; 2]` → не нужен (2 байта)

**Сокращаем:**
4. `payload_hash: u64` → `u32` (4 байта)

**Итого:** 8 + 4 + 2 + 4 = **18 байт экономии**
**Результат:** 56 - 18 = **38 байт** (ещё 6 байт до цели)

**Дополнительно:**
5. Упаковать `priority` + `flags` в `u16` вместо отдельных `u8` (нет экономии, но compact)

**Финальный layout (32 байта):**
```rust
pub struct Event {
    pub event_id: u64,        // 8b
    pub parent_event_id: u64, // 8b
    pub payload_hash: u32,    // 4b ← было u64
    pub target_id: u32,       // 4b
    pub source_id: u32,       // 4b
    pub domain_id: u16,       // 2b
    pub event_type: u16,      // 2b
    // Убрано: pulse_id (8b), payload_size (4b), _reserved (2b), 4 байта hash
}
```

**Вычисляемые поля:**
- `pulse_id` → `event_id / heartbeat_config.interval`
- `payload_size` → `EventType::payload_size(event_type)`

## Варианты оптимизации DynamicTrace (56 → 32 байт)

### Вариант 1: Удалить last_update (рекомендуется)
**Убираем:** `pub last_update: u64` (8 байт)

**Обоснование:**
- Trace короткоживущий (несколько секунд)
- Можно использовать только `created_at`
- Обновления можно трекать через TTL или flags

**Потенциальные проблемы:**
- Теряется информация о времени последнего обновления
- Может усложнить fadeout логику

**Экономия:** 8 байт → 48 байт total

---

### Вариант 2: Сократить координаты до i16
**Было:** `x: i32, y: i32, z: i32` = 12 байт
**Станет:** `x: i16, y: i16, z: i16` = 6 байт

**Обоснование:**
- Диапазон i16: -32768..32767
- Достаточно для экранных координат (1920x1080 → fits)
- Если нужен больший диапазон, можно масштабировать

**Потенциальные проблемы:**
- Ограничивает размер пространства
- Может потребовать масштабирование при больших мирах

**Экономия:** 6 байт → 50 байт total

---

### Вариант 3: Упаковать координаты в битовые поля
**Было:** 3 × i32 = 12 байт
**Станет:** `packed_coords: u64` (21 bits per coordinate)

**Обоснование:**
- 21 bits дают диапазон ±1M (достаточно для большинства случаев)
- Упаковка: [21 bits X | 21 bits Y | 21 bits Z | 1 bit unused]

**Потенциальные проблемы:**
- Сложная упаковка/распаковка
- Ухудшает читаемость и производительность

**Экономия:** 4 байта → 52 байт total

---

### Вариант 4: Удалить created_at, использовать только last_update
**Убираем:** `pub created_at: u64` (8 байт)

**Обоснование:**
- Для fadeout достаточно одного timestamp
- created_at редко используется после создания

**Потенциальные проблемы:**
- Теряется информация о возрасте trace
- Может быть нужна для определения TTL

**Экономия:** 8 байт → 48 байт total

---

### Вариант 5: Упаковать метаданные
**Было:**
- `source_type: u8` (1 байт)
- `source_id: u32` (4 байта)
- `flags: u8` (1 байт)
- `resonance_class: u8` (1 байт)

**Станет:** `metadata: u64`
- source_type: 8 bits
- source_id: 32 bits
- flags: 8 bits
- resonance_class: 8 bits
- unused: 8 bits

**Обоснование:**
- Логически связанные поля
- Можно упаковать в u64

**Потенциальные проблемы:**
- Усложняет доступ к полям
- Ухудшает читаемость

**Экономия:** Нет (уже compact, padding убирается в других местах)

---

### Вариант 6: Комбинированная оптимизация (РЕКОМЕНДУЕТСЯ)

**Изменяем:**
1. `x: i32, y: i32, z: i32` → `i16, i16, i16` (6 байт экономии)
2. `created_at: u64` → удаляем (8 байт)
3. `last_update: u64` → переименовываем в `timestamp` (0 байт, только одно поле)

**ИЛИ:**

**Альтернатива - упаковка:**
1. `x, y, z` → `packed_position: u64` (21 + 21 + 21 bits) (4 байта экономии)
2. Оставляем `created_at` + `last_update` для точности

**Рекомендуемый layout (32 байта):**
```rust
pub struct DynamicTrace {
    pub timestamp: u64,       // 8b ← было created_at + last_update
    pub x: i16,               // 2b ← было i32
    pub y: i16,               // 2b ← было i32
    pub z: i16,               // 2b ← было i32
    pub weight: f32,          // 4b
    pub frequency: f32,       // 4b
    pub source_id: u32,       // 4b
    pub source_type: u8,      // 1b
    pub flags: u8,            // 1b
    pub resonance_class: u8,  // 1b
    pub _padding: u8,         // 1b (для alignment)
}
```

**Экономия:** 6 (coords) + 8 (timestamp) = 14 байт
**Результат:** 56 - 14 = **42 байта** → с padding убирается до **40 байт** (ещё 8 до цели)

**Более агрессивный вариант (32 байта точно):**
```rust
pub struct DynamicTrace {
    pub timestamp: u64,       // 8b
    pub packed_position: u64, // 8b (21+21+21 bits для x,y,z)
    pub weight: f32,          // 4b
    pub frequency: f32,       // 4b
    pub source_id: u32,       // 4b
    pub metadata: u32,        // 4b (source_type:8, flags:8, resonance:8, unused:8)
}
```

**Экономия:** 12 (coords) + 8 (timestamp) + 3 (metadata packing) = **23 байта**
**Результат:** Ровно **32 байта** ✅

## Рекомендации

### Event → 32 байт
**Вариант 6 (комбинированный):**
- Удалить: `pulse_id`, `payload_size`, `_reserved`
- Сократить: `payload_hash: u64` → `u32`
- Вычисляемые: pulse_id через event_id, payload_size через EventType

**Impact:**
- ✅ Достигается цель 32 байта
- ✅ Сохраняется критичная информация (event_id, parent_event_id)
- ⚠️ Требует доступ к HeartbeatConfig и EventType для вычислений
- ✅ Минимальное влияние на производительность

### DynamicTrace → 32 байт
**Вариант 6 (агрессивная упаковка):**
- Координаты: упаковать в `packed_position: u64` (21 bits each)
- Время: одно поле `timestamp: u64`
- Метаданные: упаковать в `metadata: u32`

**Impact:**
- ✅ Достигается цель 32 байта
- ⚠️ Требует упаковка/распаковка координат (bitwise operations)
- ⚠️ Диапазон координат ±1M (достаточно для большинства случаев)
- ⚠️ Усложняет код доступа к полям

**Альтернатива (менее агрессивная, 40 байт):**
- Координаты: `i16` (диапазон ±32K)
- Время: одно поле `timestamp: u64`
- Проще в использовании, но на 8 байт больше
