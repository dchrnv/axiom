# Anchor Tokens V1.0 — Калибровка семантического пространства

**Версия:** 1.0  
**Дата:** 2026-04-13  
**Назначение:** Привязка абстрактных координат AXIOM к реальному смыслу  
**Для:** Claude Sonnet (реализация) + chrnv (наполнение якорных файлов)

---

## 1. Проблема

Сейчас TextPerceptor превращает "лёд" в хэш → случайные координаты. "Солнце" может оказаться рядом с "лёд". Оси X/Y/Z не имеют семантики. Shell L1-L8 заполняется эвристиками. Домены обрабатывают всё одинаково.

Результат: система работает механически правильно (reflex, tension, persistence), но **не понимает смысл** того что обрабатывает.

## 2. Решение: якорные токены

Якоря — фиксированные токены с заданными координатами и Shell-профилями, которые определяют семантику пространства. Новые токены позиционируются **относительно якорей**.

Три уровня якорей:

### 2.1 Осевые якоря (6 штук — полюса пространства)

Определяют смысл осей X/Y/Z:

```
X+: Аполлон  (порядок, структура, логика, холод, кристалл)
X-: Дионис   (хаос, творчество, спонтанность, жар, поток)
Y+: Эрос     (жизнь, связь, рост, когерентность)
Y-: Танатос  (смерть, распад, разрушение, деструкция)
Z+: Власть   (сила, магнитуда, яркость, громкость)
Z-: Ничто    (тишина, пассивность, пустота, ноль)
```

### 2.2 Слоевые якоря (по набору на каждый L1-L8)

Определяют что каждый семантический слой "видит":

```
L1 Physical:   камень, вода, воздух, огонь, земля, дерево, металл
L2 Sensory:    красный, синий, громко, тихо, горячий, холодный, сладкий
L3 Motor:      бежать, хватать, толкать, тянуть, стоять, лежать
L4 Emotional:  радость, страх, гнев, грусть, удивление, отвращение
L5 Cognitive:  мысль, идея, вопрос, ответ, гипотеза, вывод, сравнение
L6 Social:     привет, спасибо, друг, враг, просьба, приказ, обещание
L7 Temporal:   сейчас, вчера, завтра, всегда, никогда, скоро, давно
L8 Abstract:   истина, ложь, бесконечность, ноль, причина, следствие
```

### 2.3 Доменные якоря (по набору на каждый ASHTI домен)

Определяют "компетенцию" домена — что он умеет обрабатывать:

```
EXECUTION(1):  действие, результат, выполнить, отменить, повторить
SHADOW(2):     если, допустим, возможно, альтернатива, сценарий
CODEX(3):      правило, закон, норма, исключение, нарушение
MAP(4):        факт, место, расстояние, направление, карта
PROBE(5):      проверить, измерить, запросить, данные, источник
LOGIC(6):      потому что, следовательно, если-то, противоречие
DREAM(7):      ассоциация, метафора, символ, образ, странный
ETHICS(8):     хорошо, плохо, справедливо, опасно, допустимо
```

## 4. Как якоря работают в системе

### 4.1 Загрузка при старте

```
Boot sequence (после GENOME, до CLI):
1. Загрузить axes.yaml → создать 6 осевых токенов в SUTRA(100)
2. Загрузить layers/L1..L8.yaml → создать слоевые якоря в SUTRA(100)
3. Загрузить domains/D1..D8.yaml → создать доменные якоря в соответствующих доменах
4. Все якоря получают: mass=255 (максимальный), temperature=0 (неизменяемый), state=Locked
```

Якоря НЕ затухают (state=Locked, temperature=0). Они — постоянные ориентиры.

### 4.2 Позиционирование нового токена

TextPerceptor при `perceive(text)` теперь работает так:

```
1. Взять текст, разбить на слова
2. Для каждого слова — найти ближайшие якоря:
   a. Точное совпадение (word == anchor.word)
   b. Совпадение с aliases
   c. Substring matching (MVP)
   d. [Будущее] Cosine similarity через embeddings
3. Вычислить позицию как взвешенное среднее позиций совпавших якорей
4. Вычислить Shell как взвешенное среднее Shell совпавших якорей
5. Если нет совпадений — fallback на FNV-1a хэш (текущее поведение)
```

Пример:

```
Ввод: "ледяной кристалл"
  → "ледяной" matches "холодный" (substring "лед" ≈ "холод" — по tags: температура)
  → "кристалл" matches "порядок" (alias "кристалл")
  → Position = avg(cold_pos, order_pos) = avg([15000,-10000,-10000], [30000,0,0])
            = [22500, -5000, -5000]
  → Shell = avg(cold_shell, order_shell)
  → Результат: токен в области "порядок + холод" — семантически верно!
```

### 4.3 Доменная специализация

Когда паттерн проходит через ASHTI(1-8), каждый домен "видит" его через свои якоря:

```
DREAM(7) получает "ледяной кристалл":
  → Ищет резонанс с доменными якорями DREAM
  → "кристалл" резонирует с "символ" (кристалл как символ совершенства)
  → Обрабатывает метафорически

LOGIC(6) получает "ледяной кристалл":
  → Ищет резонанс с доменными якорями LOGIC
  → "ледяной" → "причина" (почему ледяной? температура < 0)
  → Обрабатывает аналитически

EXECUTION(1) получает "ледяной кристалл":
  → "кристалл" → "действие" (что делать с кристаллом?)
  → Обрабатывает практически
```

Каждый домен "понимает" паттерн по-своему — через свой набор якорей.

## 5. Реализация

### 5.1 Новые структуры

```rust
// В axiom-core или axiom-config
pub struct Anchor {
    pub id: String,
    pub word: String,
    pub aliases: Vec<String>,
    pub tags: Vec<String>,
    pub position: [i16; 3],
    pub shell: [u8; 8],
}

pub struct AnchorSet {
    pub axes: Vec<Anchor>,           // 6 осевых
    pub layers: [Vec<Anchor>; 8],    // По набору на L1-L8
    pub domains: [Vec<Anchor>; 8],   // По набору на D1-D8
}

impl AnchorSet {
    pub fn load(config_dir: &Path) -> Result<Self, ConfigError>;

    /// Найти ближайшие якоря для текста
    pub fn match_text(&self, text: &str) -> Vec<AnchorMatch>;

    /// Вычислить позицию на основе совпавших якорей
    pub fn compute_position(&self, matches: &[AnchorMatch]) -> [i16; 3];

    /// Вычислить Shell на основе совпавших якорей
    pub fn compute_shell(&self, matches: &[AnchorMatch]) -> [u8; 8];
}

pub struct AnchorMatch {
    pub anchor: &Anchor,
    pub score: f32,     // 0.0..1.0 — сила совпадения
    pub match_type: MatchType,
}

pub enum MatchType {
    Exact,      // word == input
    Alias,      // alias == input
    Substring,  // input contains word/alias
    Tag,        // shared tag
}
```

### 5.2 Интеграция с TextPerceptor

```rust
impl TextPerceptor {
    pub fn new(anchors: Arc<AnchorSet>) -> Self {
        Self { anchors }
    }

    pub fn perceive(&self, text: &str) -> UclCommand {
        let matches = self.anchors.match_text(text);

        let (position, shell) = if matches.is_empty() {
            // Fallback: FNV-1a хэш (текущее поведение)
            (hash_to_position(text), heuristic_shell(text))
        } else {
            // Якорное позиционирование
            let pos = self.anchors.compute_position(&matches);
            let shell = self.anchors.compute_shell(&matches);
            (pos, shell)
        };

        build_inject_token_command(100, position, shell, ...)
    }
}
```

### 5.3 Загрузка якорей в домены

При старте:

```rust
fn load_domain_anchors(engine: &mut AxiomEngine, anchor_set: &AnchorSet) {
    for (i, domain_anchors) in anchor_set.domains.iter().enumerate() {
        let domain_id = 101 + i as u16; // EXECUTION=101 .. ETHICS=108
        for anchor in domain_anchors {
            let mut token = Token::new();
            token.position = anchor.position;
            token.mass = 255;           // Максимальная гравитация
            token.temperature = 0;       // Неизменяемый
            token.state = State::Locked; // Не двигается, не затухает
            // Shell пишется в Connection/Shell-кэш
            engine.inject_anchor(domain_id, token, &anchor.shell);
        }
    }
}
```

## 6. Порядок работы

### Фаза 1: Структура + загрузка (Sonnet)

1. `AnchorSet`, `Anchor`, `AnchorMatch` — структуры данных
2. `AnchorSet::load()` — парсинг YAML файлов
3. Загрузка axes.yaml при старте — 6 токенов в SUTRA
4. CLI команда `:anchors` — показать загруженные якоря
5. Тесты: загрузка, валидация, match

### Фаза 2: Интеграция с TextPerceptor (Sonnet)

1. `AnchorSet::match_text()` — поиск совпадений
2. `compute_position()`, `compute_shell()` — вычисление на основе якорей
3. TextPerceptor использует AnchorSet вместо чистого хэша
4. Тесты: "порядок" → позиция рядом с осевым якорем X+
5. CLI команда `:match "текст"` — диагностика маппинга

### Фаза 3: Слоевые + доменные якоря (Sonnet + chrnv)

1. Загрузка layers/L1..L8.yaml
2. Загрузка domains/D1..D8.yaml
3. Инжекция доменных якорей в соответствующие домены
4. Тесты: доменные якоря влияют на resonance_search

### Фаза 4: Наполнение (chrnv)

Это твоя работа — заполнить YAML файлы. Начни с малого:
- axes.yaml — 6 якорей (уже описаны выше)

## 7. Инструкция для chrnv: как заполнять якорные файлы

### Принцип

Для каждого слова-якоря ты определяешь:
- **position [x, y, z]** — где в пространстве осей оно находится. Используй систему осей:
  - X: порядок (+) ↔ хаос (-)
  - Y: жизнь (+) ↔ смерть (-)
  - Z: сила (+) ↔ тишина (-)
- **shell [L1..L8]** — какие семантические слои "загораются" (0=не затронут, 255=максимум)
- **aliases** — синонимы и однокоренные слова
- **tags** — категории для группировки

### Подсказки

Когда определяешь position, задай себе три вопроса:
1. Это **порядок или хаос**? "Математика" → X+, "Танец" → X-
2. Это **жизнь или разрушение**? "Цветок" → Y+, "Яд" → Y-
3. Это **громко или тихо**? "Взрыв" → Z+, "Шёпот" → Z-

Когда определяешь shell:
- L1 physical: затрагивает тело/материю?
- L2 sensory: можно увидеть/услышать/потрогать?
- L3 motor: связано с движением/действием?
- L4 emotional: вызывает эмоцию?
- L5 cognitive: требует мысли/анализа?
- L6 social: связано с общением?
- L7 temporal: привязано ко времени?
- L8 abstract: абстрактное понятие?

### Объём

Начни с **axes.yaml** (6 якорей — уже описаны в этом документе, можно взять как есть).
Затем **один layer файл** (рекомендую L5_cognitive — 10-15 слов).
Затем **один domain файл** (рекомендую D1_execution — 5-10 слов).
Итого: первая итерация — **~30 якорей**.

Добавлять больше постепенно, наблюдая через CLI (`:match "текст"`) как система реагирует.

## 8. Инварианты

1. **Якоря неизменяемы.** mass=255, temperature=0, state=Locked. Физика поля не двигает их.
2. **Якоря не затухают.** min_intensity не применяется к Locked токенам.
3. **Fallback на хэш.** Если текст не совпал ни с одним якорем — используется FNV-1a (текущее поведение). Система не ломается без якорей.
4. **GENOME разрешает.** Инжекция якорей проходит через GUARDIAN. Якорные файлы — часть конфигурации, не нарушают конституцию.
5. **Hot reload.** При изменении YAML файла и ConfigWatcher — якоря перезагружаются. Новые добавляются, существующие обновляются, удалённые — помечаются для деактивации (не удаляются немедленно).
