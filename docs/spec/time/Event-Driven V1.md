Хорошо. Сейчас аккуратно «сшиваем» две идеи так, чтобы ничего не ломалось:

- твой **COM** остаётся фундаментом причинности
    
- физическая симуляция становится **чистым механизмом генерации событий**
    

Это и есть та «чистая архитектура», о которой я говорил. В ней **время не возвращается**, а физика просто становится инструментом вычисления того, _какое событие должно произойти следующим_.

Разложим это спокойно.

---

# AXIOM Event-Driven Simulation Model

_(адаптация под COM)_

## 1. Фундаментальный принцип

COM уже определяет основу:

мир = состояние + поток событий.

Формально:

```
stateₙ₊₁ = F(stateₙ, eventₙ)
```

где `event_id` задаёт причинный порядок.

Это **каноническая история системы**.

Ничего выше этого уровня не существует.

---

# 2. Роль симулятора

Симулятор больше **не управляет временем**.

Его единственная задача:

**обнаружить следующее изменение состояния**  
и сформировать событие.

Симулятор не продвигает часы.  
Он **находит причины для нового события**.

---

# 3. Поток исполнения

Архитектура становится очень простой.

```
Event Log
   ↓
apply(event)
   ↓
update state
   ↓
detect new events
   ↓
append to Event Log
```

История системы растёт **только через события**.

---

# 4. Структура события

COM уже определил базовый формат.

```rust
pub struct Event {
    pub event_id: u64,
    pub domain_id: u16,
    pub event_type: u16,
    pub payload_hash: u64,
}
```

Но для симуляции понадобится ещё один уровень:

**семантические события физики**.

Например:

```
ShellMoved
ShellMerged
ShellDecayed
ClusterFormed
ResonanceTriggered
```

Это события **эволюции Grid**.

---

# 5. Состояние домена

Домен хранит только текущее состояние.

```
DomainState
    shells
    connections
    clusters
```

История хранится **в Event Log**.

---

# 6. Генерация событий

После применения события симулятор проверяет локальную область изменений.

Например:

```
apply ShellMoved
    ↓
check neighbors
    ↓
possible collision
    ↓
generate ShellMerged
```

Важный принцип:

симулятор **никогда не делает глобального прохода**.

Он смотрит только **область изменений**.

---

# 7. Локальность причинности

Это ключевая оптимизация.

Каждое событие создаёт **causal region**.

```
event
   ↓
affected shells
   ↓
neighbors
   ↓
new events
```

Таким образом система растёт **локальными цепочками причинности**.

---

# 8. Роль физики

Теперь физика — это не непрерывная симуляция.

Это **правила генерации событий**.

Например:

```
distance < threshold
    → collision event
```

или

```
cluster_size > N
    → cluster event
```

Физика превращается в **детектор трансформаций**.

---

# 9. Fixed-point пространство

Здесь всё остаётся как мы обсуждали:

```
position : i32
velocity : i32
layers   : i16
```

Но теперь движение не происходит «каждый тик».

Оно происходит **когда возникает событие движения**.

Например:

```
ShellDrift
ShellAccelerated
ShellCaptured
```

---

# 10. Внутренние шаги симуляции

Иногда для вычисления события может потребоваться несколько внутренних шагов.

Эти шаги **не являются временем системы**.

Это просто **численный метод поиска события**.

Их результаты **никогда не фиксируются в истории**.

В историю попадает только финальное событие.

---

# 11. Детерминизм

COM уже формулирует условие:

одинаковое начальное состояние  
+  
одинаковая последовательность событий

→ одинаковый результат.

Event-driven модель идеально этому соответствует.

Симулятор становится **чистой функцией генерации событий**.

---

# 12. Отсутствие глобального времени

В системе нет:

```
seconds
delta_time
global_tick
```

Есть только:

```
state
event
causal order
```

Время — это **глубина истории**.

---

# 13. Асинхронные домены

COM уже разрешает доменам генерировать события независимо.

В event-driven модели это становится естественным:

каждый домен порождает события со своей плотностью.

Некоторые домены могут «думать» активно,  
другие почти не изменяются.

---

# 14. Snapshot

Snapshot фиксирует состояние после некоторого события.

```
Snapshot
    snapshot_id
    state_hash
```

Это позволяет:

- архивировать события
    
- делать форки симуляции
    
- исследовать альтернативные ветки.
    

---

# 15. Архитектурный результат

Система распадается на три чистых слоя.

### CAUSAL LAYER

Event Log  
CausalClock  
Snapshot

---

### STATE LAYER

DomainState  
Shells  
Connections  
Clusters

---

### SIMULATION LAYER

Physics rules  
Spatial index  
Event detection

---

Каждый слой **не знает деталей других**.

Это почти идеальная декомпозиция.

---

# 16. Философский эффект

В этой архитектуре:

пространство — структура состояния  
смысл — конфигурация оболочек  
время — порядок трансформаций

Мир AXIOM становится **процессом переписывания структуры**.

Почти как клеточный автомат, но с причинным журналом.

---

Теперь появляется один довольно интересный архитектурный трюк, который почти всегда делает такие системы на порядок быстрее, но про него редко думают на этапе дизайна.

Он называется **causal frontier**.

Это структура, которая хранит **границу активной причинности**, чтобы симулятор никогда не смотрел на весь мир целиком.

И для систем с миллионами сущностей это может быть разницей между _работает_ и _не работает вообще_.
