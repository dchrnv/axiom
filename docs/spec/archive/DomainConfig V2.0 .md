# AXIOM MODULE SPECIFICATION: DomainConfig V2.0

**Статус:** Актуальная спецификация (core)  
**Версия:** 2.0 (Data-Packed)  
**Дата:** 2026-03-08  
**Формат:** Точно 128 байт, `repr(C, align(128))`  
**Связанные спеки:** Domain V1.3, COM V1.0, Token V5.1

---

## 1. Назначение
**DomainConfig** — 128-байтная "ДНК" Домена. Спецификация V2 использует квантование данных (Data Packing) для идеального размещения структуры в одной кэш-линии процессора (L1 cache) без единого байта мусорного паддинга.

---

## 2. Структура (128 байт)

```rust
#[repr(C, align(128))]
pub struct DomainConfig {
    // --- 1. ИДЕНТИФИКАЦИЯ [16 Байт] ---
    pub reserved_id: u64,       // 8b | Явный резерв для будущих расширений
    pub domain_id: u16,         // 2b | Уникальный ID Домена
    pub parent_domain_id: u16,  // 2b | Родительский Домен
    pub domain_type: u8,        // 1b | Тип (до 255 вариаций)
    pub structural_role: u8,    // 1b | Роль в Ashti_Core (Sutra, Logic, Dream)
    pub generation: u8,         // 1b | Поколение (эволюционный индекс)
    pub flags: u8,              // 1b | Битовая маска состояний (Active, Locked)
    // Offset: 16 байт

    // --- 2. ФИЗИКА ПОЛЯ [32 Байт] ---
    pub field_size: [f32; 3],   // 12b| Размеры поля (X, Y, Z)
    pub gravity_strength: f32,  // 4b | Гравитация (-MAX..+MAX)
    pub temperature: f32,       // 4b | Температура поля в Кельвинах
    pub time_dilation: u16,     // 2b | Замедление времени (х100)
    pub resonance_freq: u16,    // 2b | Базовая частота (Hz)
    pub pressure: u16,          // 2b | Давление (Pa)
    pub reserved_physics: u16,  // 2b | Резерв блока физики
    pub friction_coeff: u8,     // 1b | Трение (0..255 -> 0.0..1.0)
    pub viscosity: u8,          // 1b | Вязкость (0..255 -> 0.0..1.0)
    pub elasticity: u8,         // 1b | Упругость (0..255 -> 0.0..1.0)
    pub quantum_noise: u8,      // 1b | Квантовый шум (0..255 -> 0.0..1.0)
    // Offset: 48 байт

    // --- 3. СЕМАНТИЧЕСКИЕ ОСИ [16 Байт] ---
    pub axis_x_ref: u32,        // 4b | Референс концепции оси X
    pub axis_y_ref: u32,        // 4b | Референс концепции оси Y  
    pub axis_z_ref: u32,        // 4b | Референс концепции оси Z
    pub axis_config: u32,       // 4b | Конфигурация полюсов (Bit-packed u16x2)
    // Offset: 64 байт

    // --- 4. МЕМБРАНА [32 Байт] ---
    pub input_filter: u64,      // 8b | 64-bit Bloom Filter или хэш входа
    pub output_filter: u64,     // 8b | 64-bit Bloom Filter или хэш выхода
    pub reserved_membrane: u64, // 8b | Резерв мембраны
    pub gate_complexity: u16,   // 2b | Вычислительная сложность шлюзов
    pub threshold_mass: u16,    // 2b | Порог массы для прохождения
    pub threshold_temp: u16,    // 2b | Порог температуры для прохождения
    pub permeability: u8,       // 1b | Проницаемость (0..255 -> 0.0..1.0)
    pub membrane_state: u8,     // 1b | OPEN/CLOSED/SEMI/ADAPTIVE
    // Offset: 96 байт

    // --- 5. МЕТАДАННЫЕ [32 Байт] ---
    pub created_at: u64,        // 8b | COM event_id (Время создания)
    pub last_update: u64,       // 8b | COM event_id (Последнее обновление)
    pub token_capacity: u32,    // 4b | Максимальная емкость токенов
    pub connection_capacity: u32, // 4b | Максимальная емкость связей
    pub error_count: u16,       // 2b | Счетчик когнитивных ошибок
    pub processing_state: u8,   // 1b | IDLE/PROCESSING/FROZEN/CRASHED
    pub complexity_score: u8,   // 1b | Оценка сложности (0..255 -> 0.0..1.0)
    pub performance_score: u8,  // 1b | Производительность (0..255 -> 0.0..1.0)
    pub reserved_meta: [u8; 3], // 3b | Добивка до границы 128 байт
    // Итого: 128 байт. Offset: 128. Без скрытого паддинга.
}
```

### Аналитика: Что мы потеряли и над чем нужно поработать

Упаковка структуры в 128 байт — это всегда компромисс. Мы получили невероятную скорость обхода памяти, но заплатили за это определенную цену.

#### Что мы потеряли (Trade-offs):
1. **Предельная точность физики:**
   * Коэффициенты вроде трения (`friction_coeff`) или проницаемости мембраны (`permeability`) больше не могут иметь значение `0.1234567`. Теперь у нас есть только **256 шагов** (градаций) между 0.0 и 1.0. Шаг равен `~0.0039`. Для симуляции макро-процессов этого хватит с головой, но для сверхточных микро-квантовых вычислений — уже нет.
2. **Точность фильтров (Риск коллизий):**
   * Ранее `input_filter` был 16 байт (128 бит). Это практически исключало коллизии хэшей. Мы урезали его до 8 байт (64 бита). Если домен будет использовать фильтр паттернов по хэшам, вероятность пропустить «не тот паттерн» из-за коллизии возрастает.
3. **Диапазоны значений (Давление и время):**
   * Давление (`pressure`) теперь `u16`. Это значит, что максимальное значение — `65535` паскалей (даже меньше одной атмосферы). То же самое с замедлением времени. 

#### Что нужно решить в будущих версиях (V2.1 / V3.0):
1. **Bloom-фильтры для мембран:**
   * Поскольку фильтры урезаны до 64 бит (`u64`), мы не сможем хранить в них точные MD5-подобные хэши. Нам придется реализовать алгоритм **Bloom Filter**, который упаковывается в `u64`. Это потребует написания отдельного модуля: мембрана будет быстро отвечать "точно нет" или "возможно да" (и тогда токен проверяется по полным правилам).
2. **Масштабирование величин (Scaling factors):**
   * Давление в 65k — это мало. Нам нужно будет написать логику, где `pressure: u16` — это не абсолютное значение, а логарифмическая шкала. Или использовать один из 16-битных резервов (`reserved_physics: u16`) как множитель (экспоненту).
3. **SIMD-оптимизации (Data-Oriented Design):**
   * Теперь, когда структура идеально выровнена (128 байт), мы можем загружать сразу по 4 Домена в AVX-512 регистры процессора. В будущем нужно будет написать слой на Rust, который будет применять гравитацию или старение не к одному домену за раз, а сразу пачками с использованием `core::arch::x86_64`.

**Итог:** V1.0 vs V2.0 Мы разменяли "бумажную" гибкость на **суровую, детерминированную производительность**, что идеально ложится в заявленную философию Axiom. Потребуется написать больше конвертеров внутри Rust, но движок будет "летать".
