# Ashti_Core V2.2

*Архитектурная композиция из одиннадцати специализированных Доменов (Domain V1) в рамках одного фрактального уровня, управляемая конституционным слоем GENOME и контролируемая GUARDIAN.*

**Статус:** Актуальная спецификация (core)  
**Версия:** 2.2.0  
**Дата:** 2026-05-12  
**Связанные спеки:** GENOME V1.0, GUARDIAN V1.0, Domain V1.3, DomainConfig V2.1, Arbiter V2.1, Causal Frontier V2.0, Shell V3.0, Heartbeat V2.0, COM V1.0, Time Model V1.0, Axiom_Sentinel_V1_1

---

## 1. Что изменилось в V2.2

V2.1 → V2.2 (Axiom Sentinel V1.1):

- **`process_parallel_limited`** — новый публичный метод AshtiCore для Layer Priority режима. Делегирует в `Arbiter::route_token_limited` с ограниченным `max_role`.
- Связанные спеки обновлены: Arbiter V2.1, Axiom_Sentinel_V1_1.

---

## 1.1 Что изменилось в V2.1

Ashti_Core V2.0 описывала 11 доменов, двойной путь обработки и CODEX/GUARDIAN как концепции. V2.1 вносит следующие изменения:

1. **GENOME** — явный конституционный слой над всей архитектурой (GENOME V1.0). Загружается первым. Read-only. Определяет права, протоколы, инварианты.

2. **GUARDIAN** — полноценная спецификация (GUARDIAN V1.0). Два источника правил: GENOME (абсолютный) + CODEX (пластичный). Единственный писатель в CODEX.

3. **CODEX (Домен 3)** — уточнена роль: пластичный закон, аналог ADNA из NeuroGraph. GUARDIAN — единственный модуль с правом Write. CODEX подчинён GENOME.

4. **EXPERIENCE (Домен 9)** — расширена внутренняя структура: добавлены REFLECTOR (статистика, профилирование) и SKILLSET (кристаллизованные скиллы) как функциональные зоны внутри домена, не отдельные домены.

5. **Полный цикл** — обновлён с учётом GENOME enforcement и GUARDIAN как явного участника.

Количество доменов: **11** (без изменений). Структурные роли: без изменений.

## 3. Компоненты архитектуры

### 3.1 GENOME (Конституция)

Не домен. Конфигурационный слой, загружаемый первым. Полное описание — GENOME V1.0.

Ключевые правила, определённые GENOME для Ashti_Core:
- SUTRA(0) — единственный с правом WRITE на создание токенов.
- GUARDIAN — единственный с правом WRITE на CODEX(3).
- event_id монотонно возрастает (Time Model V1.0).
- min_intensity > 0 в EXPERIENCE(9) — ничто не забывается.
- Все маршруты данных между доменами описаны в GENOME. Неописанные — запрещены.

### 3.2 GUARDIAN (Контроллер)

Не домен. Над-доменный процесс. Полное описание — GUARDIAN V1.0.

Функции в контексте Ashti_Core:
- Проверка рефлексов перед отправкой в MAYA(10).
- Сканирование узоров в доменах 1-8 на соответствие GENOME + CODEX.
- Ингибирование нарушающих паттернов (установка INHIBITED, не удаление).
- Управление правилами CODEX(3): усиление, ослабление, создание, ингибирование.
- Enforcement прав доступа и протоколов (из GENOME).

### 3.3 SUTRA (Домен 0) — Источник

Единственный источник истины. Статичная библиотека токенов. Единственный домен с правом WRITE (определено в GENOME). Все остальные домены работают через REFERENCE (ссылки на sutra_id).

Поле в перманентном кристаллическом состоянии: максимальная гравитация, нулевое затухание.

Зеркальность с EXPERIENCE(9): SUTRA хранит что система **знает** (вечные смыслы). EXPERIENCE хранит что система **пережила** (историю проявлений).

### 3.4 EXPERIENCE (Домен 9) — Память и опыт

Экземпляр Domain V1, сконфигурированный как ассоциативная память: низкая гравитация, минимальное затухание (min_intensity > 0), высокий резонанс, средняя температура, высокая вязкость.

**V2.1: Три функциональные зоны внутри домена 9:**

**Зона 1: Резонансный поиск (базовая)**
Механизм из V2.0: входящий паттерн резонирует с существующими следами. Три уровня ответа: рефлекс (высокий weight), ассоциация (средний), тишина (слабый/отсутствующий).

**Зона 2: REFLECTOR (статистика и профилирование)**
Накапливает данные для обучения системы:
- Счётчик успешности рефлексов: сколько раз рефлекс совпал с результатом ASHTI(1-8).
- Семантические профили: какие Shell-профили (L1-L8) чаще приводят к успеху в каких доменах.
- Данные для адаптации порогов Arbiter: если рефлексы EXECUTION(1) систематически верны, reflex_threshold может быть снижен (через GUARDIAN → DomainConfig).
- Временные метрики: причинный возраст самых успешных рефлексов (как быстро формируется надёжный навык).

REFLECTOR — не отдельный домен. Это дополнительные структуры данных внутри DomainState домена 9.

**Зона 3: SKILLSET (кристаллизованные навыки)**
Хранит группы устойчивых, подтверждённых следов:
- Скилл = кластер токенов со связями, где все токены имеют высокий weight и стабильные межсвязи.
- Скилл может быть активирован как единое целое (не поэлементно).
- Скилл может быть экспортирован для передачи между экземплярами AXIOM.
- Механизм кристаллизации: когда группа следов достигает устойчивой конфигурации (высокий weight, стабильные связи, подтверждённые N раз), GUARDIAN может пометить её как скилл.

SKILLSET — не отдельный домен. Это особая зона в поле домена 9, где кристаллизованные кластеры стабилизированы высокой массой и низкой температурой.

**Три режима ответа (без изменений от V2.0):**
Рефлекс, Ассоциация, Тишина. Механизм обучения через обратную связь. min_intensity > 0 (GENOME-инвариант).

### 3.5 ASHTI (Домены 1-8) — Процессоры

Восемь экземпляров Domain V1 с различной конфигурацией. Без изменений от V2.0.

| №  | Название      | Координаты | Функция |
|----|---------------|------------|---------|
| 1  | EXECUTION     | `(+,+,+)`  | Реализация решений |
| 2  | SHADOW        | `(-,+,-)`  | Симуляция угроз и сценариев |
| 3  | CODEX         | `(-,-,+)`  | Пластичный закон (управляется GUARDIAN) |
| 4  | MAP           | `(+,-,+)`  | Статичная база фактов |
| 5  | PROBE         | `(+,+,-)`  | Зондирование среды через гипотезы |
| 6  | LOGIC         | `(-,+,+)`  | Точные вычисления и дедукция |
| 7  | DREAM         | `(-,-,-)`  | Фоновая оптимизация, рекомбинация |
| 8  | VOID          | `(+,-,-)`  | Неопределённость, аномалии |

**V2.1: Уточнение роли CODEX (Домен 3):**

CODEX — пластичный закон, аналог ADNA из NeuroGraph. Содержит поведенческие правила, которые система выучила или получила. Отличия от других доменов ASHTI:

- GUARDIAN — единственный модуль с правом Write на CODEX (GENOME-правило).
- Правила CODEX ингибируются если противоречат GENOME (GUARDIAN проверяет).
- CODEX обрабатывает паттерны как и другие домены 1-8 (получает из Arbiter, возвращает в MAYA). Но его результат имеет особый вес при проверке GUARDIAN.
- DREAM(7) может предлагать изменения правил CODEX через обратную связь → GUARDIAN → CODEX.

### 3.6 MAYA (Домен 10) — Интерфейс

Без изменений от V2.0. Выход уровня. Принимает предсказание от EXPERIENCE(9) и результат от ASHTI(1-8), формирует итоговую проекцию.

Двойной вход: рефлекс может прийти раньше результата 1-8. MAYA сравнивает их. Результат сравнения → Arbiter → обратная связь → EXPERIENCE(9).

### 3.7 Arbiter

Без изменений от Arbiter V1.0. Над-доменный маршрутизатор. Получает Arc<Genome> при создании для проверки протоколов.

### 3.8 Связующие шины

Внешние процессы, обеспечивающие коммуникацию между доменами. Домены не общаются напрямую. Все допустимые маршруты описаны в GENOME.

```
                  (усиление/ослабление правил)
```

| Маршрут | Содержание | Условие | Описан в GENOME |
|---------|-----------|---------|-----------------|
| 0 → 9 | Ссылка на токен | Всегда при новом событии | Да, mandatory |
| 9 → Arbiter | Результат резонансного поиска | Всегда | Да, mandatory |
| Arbiter → GUARDIAN | Рефлекс на проверку | Если GUARDIAN_CHECK_REQUIRED | Да |
| GUARDIAN → Arbiter | Approve / Veto | Если был запрос | Да |
| Arbiter → 10 | Рефлекс (одобренный) | Только если GUARDIAN одобрил | Да |
| Arbiter → 1-8 | Паттерн + подсказка | Всегда | Да, mandatory |

## 5. Полный цикл работы уровня (обновлён)

### 5.1 Boot sequence

```
1. Загрузить GENOME → Arc<Genome>
2. Создать GUARDIAN (получает Arc<Genome>)
3. Создать домены 0-10 (каждый получает Arc<Genome> для чтения своих прав)
4. Создать Arbiter (получает Arc<Genome>)
5. Инициализировать COM (CausalClock)
6. GUARDIAN валидирует все DomainConfig на соответствие GENOME
7. Система готова
```

### 5.2 Обработка одного паттерна (мыслительный акт)

1. **Событие** → Токен создаётся или активируется в SUTRA(0).

2. **Передача** → Ссылка на токен передаётся в EXPERIENCE(9).

3. **Резонансный поиск** → Домен 9 ищет резонанс. Результат: рефлекс / ассоциация / тишина. REFLECTOR фиксирует запрос.

4. **Маршрутизация** → Ответ передаётся Arbiter. Arbiter классифицирует ответ per-domain (рефлекс для одного, ассоциация для другого, тишина для третьего) на основе DomainConfig V2.1.

5. **Проверка рефлекса** → Если рефлекс найден и бит GUARDIAN_CHECK_REQUIRED установлен → Arbiter запрашивает GUARDIAN. GUARDIAN проверяет: GENOME (протокол, доступ) + CODEX (поведенческие правила). Результат: Approve или Veto.

6. **Быстрый путь** → Одобренный рефлекс отправляется в MAYA(10) как предварительная реакция.

7. **Медленный путь (всегда)** → Arbiter формирует пакеты для ASHTI(1-8): паттерн + подсказка (если ассоциация). Количество подсказок ≤ max_concurrent_hints.

8. **Обработка** → Каждый домен ASHTI обрабатывает паттерн согласно своей физике.

9. **Надзор** → GUARDIAN сканирует формирующиеся узоры, сверяет с GENOME + CODEX. Ингибирует нарушающие.

10. **Консолидация** → Узоры стекаются в MAYA(10).

11. **Сравнение** → MAYA сравнивает рефлекс (если был) с результатом 1-8.

12. **Обратная связь** → Результат возвращается через Arbiter:
    - В EXPERIENCE(9): усиление/ослабление следа, запись нового опыта. REFLECTOR обновляет статистику.
    - Через GUARDIAN в CODEX(3): если паттерн повторяется и подтверждён, GUARDIAN может создать/усилить правило.

13. **Кристаллизация** (фоновый процесс) → При достижении устойчивой конфигурации группа следов в EXPERIENCE(9) кристаллизуется в скилл (SKILLSET зона).

14. **Выход** → Результат из MAYA(10) исполняется или по протоколу `10 → 0` передаётся на следующий уровень.

### 5.3 Принцип двойного пути

Медленный путь через ASHTI(1-8) работает **всегда**, независимо от наличия рефлекса. Это гарантирует постоянное обучение и коррекцию рефлексов.

Исключение (V2.2): при Layer Priority режиме (`enable_layer_priority = true`, `budget_used_fraction > 0.80`) медленный путь ограничивается доменами 1–3 через `process_parallel_limited`.

### 5.4 Публичный API AshtiCore (V2.2)

```rust
// Обычный путь (всегда используется при нормальном бюджете тика)
pub fn process_parallel(&mut self, token: Token, pool: &rayon::ThreadPool) -> RoutingResult

// Layer Priority путь (Sentinel S5): только домены 1..=max_role
pub fn process_parallel_limited(
    &mut self,
    token: Token,
    pool: &rayon::ThreadPool,
    max_role: u8,
) -> RoutingResult
```

Файл: `crates/axiom-domain/src/ashti_core.rs`

`process_parallel_limited` делегирует в `self.arbiter.route_token_limited(token, Some(pool), max_role)`. Семантика идентична `process_parallel` кроме ограничения max_role.

## 7. Метафора: Организм (обновлена)

- **GENOME** — Конституция. Неизменяемый закон, существующий до организма.
- **GUARDIAN** — Иммунная система + конституционный суд. Исполняет закон, фильтрует патогены, обновляет подзаконные акты.
- **SUTRA (0)** — ДНК. Вечный код, неизменная основа.
- **EXPERIENCE (9)** — Иммунная память + нервная система. Рефлексы (SKILLSET), статистика (REFLECTOR), ассоциации.
- **CODEX (3)** — Подзаконные акты. Пластичный, эволюционирующий закон, подчинённый конституции.
- **ASHTI (1-8)** — Органы. Каждый со своей функцией.
- **MAYA (10)** — Лицо. Проекция внутренней работы.
- **Arbiter** — Нервная маршрутизация. Направляет сигналы.

## 8. Направления развития (обновлены)

1. **GridHash-индекс для EXPERIENCE(9)** — O(1) lookup поверх физики поля для ускорения resonance_search (из IntuitionEngine NeuroGraph). Не заменяет физику — дополняет.
2. **Адаптивные пороги Arbiter** — REFLECTOR накапливает статистику, GUARDIAN корректирует пороги reflex_threshold/association_threshold в DomainConfig.
3. **Формализация кристаллизации** — критерии перехода кластера следов в скилл: минимальный weight, минимальное количество подтверждений, устойчивость кластера.
4. **Протокол обмена скиллами** — экспорт/импорт SKILLSET между экземплярами AXIOM.
5. **Эволюция GENOME** — контролируемый механизм обновления конституции (V2.0 GENOME).
6. **Causal Horizon** — архивация COM event log за горизонтом причинности (Causal Frontier V2.0, раздел 13).
