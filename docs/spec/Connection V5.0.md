# AXIOM MODULE SPECIFICATION: CONNECTION V5.0

**Статус:** Актуальная спецификация (core)  
**Версия:** 5.0.0  
**Дата:** 2026-03-01  
**Формат:** 64 байта, `repr(C, align(64))`  
**Модель времени:** COM `event_id` (причинный порядок, u64)  
**Связанные спеки:** COM, Token V5.0, Domain V1.2, UPO v2.1

---

## 1. Назначение

**Connection** — направленная связь между двумя Token внутри одного Domain, реализующая “пружину/мембрану”:

- задаёт топологию (кто с кем связан и в каком домене),
- несёт тип связи (`link_type`) и флаги,
- имеет упругую динамику (strength/stress/elasticity/ideal_dist),
- может фильтровать вертикальные/переходные взаимодействия через “шлюзы” (gates),
- хранит метаданные активности через причинное время (`event_id`).

Connection **не хранит wall-clock время**. Любая “временная” динамика выражается через **COM event_id**.

---

## 2. Структура (64 байта)

```rust
#[repr(C, align(64))]
pub struct Connection {
    // --- ТОПОЛОГИЯ (16 Байт) ---
    pub source_id: u32,      // ID источника (Token/Sutra ID)
    pub target_id: u32,      // ID цели (Token/Sutra ID)
    pub domain_id: u16,      // Домен существования связи
    pub link_type: u16,      // Тип связи (Synonym, Cause, Vertical_Drop...)
    pub flags: u32,          // ACTIVE/INHIBITED/TEMPORARY/... (битовая маска)

    // --- ДИНАМИКА (16 Байт) ---
    pub strength: f32,       // Насколько сильно связь тянет (>= 0)
    pub current_stress: f32, // Текущее напряжение/стресс (может быть >= 0)
    pub ideal_dist: f32,     // Идеальное расстояние “пружины”
    pub elasticity: f32,     // Упругость/коэффициент возврата

    // --- ШЛЮЗ (16 Байт) ---
    pub density_gate: u8,    // Порог массы (Token.mass)
    pub thermal_gate: u8,    // Порог температуры (Token.temperature)
    pub reserved_gate: [u8; 14],

    // --- МЕТАДАННЫЕ (16 Байт) ---
    pub created_at: u64,     // event_id создания (COM)
    pub last_active: u64,    // event_id последней активности/обновления (COM)
}
```

---

## 3. Инварианты

- `domain_id` определяет, где Connection “живёт”. Связь не должна одновременно принадлежать нескольким доменам.
- `created_at <= last_active` (по `event_id`).
- `strength >= 0`.
- `ideal_dist >= 0`.
- `elasticity >= 0`.
- `source_id != 0`, `target_id != 0` (0 зарезервирован как “пустой”).
- **Причинность:** обновления полей Connection сериализуются через COM — любое изменение имеет причинную позицию `event_id`.

---

## 4. Флаги (flags)

Минимальный набор для совместимости с UPO:

- `ACTIVE`: связь участвует в расчётах (UPO может фильтровать `flags & ACTIVE != 0`)

Остальные флаги (INHIBITED, TEMPORARY, DECAYING и т.п.) допускаются как расширение и должны быть задокументированы как битовая маска.

---

## 5. Тип связи (link_type)

`link_type` — это семантический код (u16), который определяет базовую природу отношения (Synonym/Cause/Vertical_Drop/…).

Принцип (как в v4): пространство типов может быть организовано категориями (например 256×256), но конкретная таблица типов — отдельный справочник/модуль.

---

## 6. Динамика “пружины/мембраны”

Интерпретация динамических полей:

- `strength`: “сила” взаимодействия (модуль)
- `current_stress`: текущее накопленное напряжение/перегрузка (для логики “рвётся/ослабевает”)
- `ideal_dist`: целевое расстояние между Token в Field
- `elasticity`: коэффициент возврата к `ideal_dist`

Gates:

- `density_gate`: пропускать взаимодействие/переходы только если `Token.mass >= density_gate`
- `thermal_gate`: пропускать взаимодействие/переходы только если `Token.temperature >= thermal_gate`

---

## 7. Связь с COM и Domain

- **COM:** `created_at` и `last_active` измеряются в `event_id` (u64), что обеспечивает детерминизм и воспроизводимость.
- **Domain:** Connection принадлежит одному `domain_id`; доменная физика (DomainConfig) может трактовать `link_type`, gates и динамику по-своему.

---

## 8. Связь с UPO

UPO использует Connection как вход:
- фильтрует по `domain_id`
- учитывает только активные связи (`flags & ACTIVE != 0`)
- может агрегировать `current_stress` и `strength` для вывода стресса домена на экран.

---

## 9. Legacy заметка (v4)

Старые документы `Connection v4.0.md` / `Connection v4 структура.md` содержат идеи про:
- ритмы, time_vector, causal_chain_id,
- wall-clock timestamps,
- “процессорную” архитектуру связи.

Эти элементы **не входят** в core Connection V5.0 (64B) и рассматриваются как возможные расширения поверх базового формата.
