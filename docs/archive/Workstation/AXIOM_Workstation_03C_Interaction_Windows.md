## Контекст Части C

Части A и B описали **окна наблюдения** — System Map, Live Field, Patterns, Dream State. Это места, где пользователь **смотрит на работу системы**.

Часть C — **окна взаимодействия**. Здесь пользователь **активно делает что-то**: даёт системе текст, импортирует файл, меняет конфигурацию, запускает бенчмарк.

Эти окна имеют общий характер:

- **Утилитарность важнее эстетики удержания внимания.** Они не должны быть красивыми "местами для жизни" — они должны быть удобными для конкретного действия. После действия пользователь обычно возвращается к окнам наблюдения.
- **Не центральные.** В отличие от System Map, эти окна **не претендуют на главную позицию** в интерфейсе. Они открываются по необходимости.
- **Сохраняют эстетику.** Минимализм, скруглённые формы, ограниченная палитра — да. Но плотность функциональности здесь выше.
- **Conversation и Benchmarks могут быть detached.** Configuration и Files — обычно нет (служебные окна, открываются ненадолго).

## 6. Conversation

### 6.1 Назначение

**Conversation** — простой утилитарный чат для подачи текста системе и просмотра отклика. **Не центр интерфейса**. Не место для долгого общения. Не продвинутый диалоговый интерфейс.

Это **отладочный канал**: подал текст, увидел, что получилось.

### 6.2 Зачем это нужно

Workstation нужен способ **подать в систему текст** и **увидеть результат**. Это базовая функция любого UI к когнитивной системе. Conversation её обеспечивает в минимальной форме.

При этом Conversation **не пытается быть Companion** (см. Документ 1, раздел 1.4). Никакой инициации со стороны системы, никакой поэтики, никаких длинных историй отношений. Это всё — будущий Companion.

Структура:

**Верхняя панель — лента диалога (~ 70-80% высоты):**
- Сообщения в обратном хронологическом порядке (новые сверху или новые снизу — см. 6.6)
- Сообщения от пользователя (You) и от системы (System)
- Каждое сообщение с timestamp

**Нижняя панель — форма ввода (~ 20-30% высоты):**
- Текстовое поле (multi-line, может разрастаться)
- Selector целевого домена
- Кнопка Submit

### 6.4 Тон сообщений системы

Сообщения от System — **технически точные, без украшений** (см. Документ 1, раздел 5.1):

✓ Хорошо:
- "Обработан текст. Создан Frame #1247."
- "Frame #1198 reactivated. New temperature: 142."
- "Frame candidate did not stabilize. Required 3 scans, observed 2."
- "Не удалось обработать текст: domain LOGIC is full (token capacity reached)."

✗ Не делаем:
- "Конечно! С удовольствием обработаю ваш текст!"
- "Я подумал о том, что вы написали..."
- "Это очень интересный текст!"
- Эмодзи

Тон — **зрелый, ровный, спокойно отчитывается о результате**. Без личности.

### 6.5 Что пользователь видит как ответ системы

При отправке текста — Workstation отправляет команду `SubmitText { text, target_domain }` в Engine. Engine обрабатывает текст (через соответствующий внутренний механизм), и возможные результаты:

**1. Текст принят, началась обработка:**
```
System ›  Обработан текст. (Frame будет проанализирован FrameWeaver на следующих сканах.)

          14:32:18
```

Это базовое подтверждение — текст принят, токены созданы. Дальнейшая обработка займёт время (FrameWeaver сканирует раз в 20 тиков по дефолту, кристаллизация требует stability_threshold).

**2. Если в ходе обработки появилось событие — оно показывается в этой же ленте:**
```
System ›  Создан Frame #1247.
          Layers: S1 (subject, predicate), S7 (reason).
          
          14:32:32   (через 14 секунд после Submit)
```

Workstation подписан на EngineEvent.FrameCrystallized и при появлении нового Frame **в результате этого текста** (если это можно установить через корреляцию с временем подачи) — отображает в Conversation.

**3. Реактивация существующего Frame:**
```
System ›  Frame #1198 reactivated.

          14:30:42
```

**4. Ничего не произошло (текст не привёл к Frame):**
- Никакого "ничего не произошло" в ленту не пишется
- Просто отсутствие реакции — нормальное состояние

**5. Ошибка обработки:**
```
System ›  Could not process text: domain LOGIC is at capacity.

          14:35:00
```

С тёплым красным акцентом (цвет внимания).

### 6.6 Направление ленты — новые сверху или снизу?

Это **архитектурный выбор**, который влияет на ощущение от окна.

**Вариант А: новые сверху, старые снизу (как в email).**
- При появлении нового сообщения — оно "падает" сверху, остальные сдвигаются вниз
- Скроллбар обычно вверху
- Не нужно скроллить, чтобы увидеть последнее
- Естественно для "просмотра ленты"

**Вариант Б: новые снизу, старые сверху (как в мессенджере).**
- При появлении нового сообщения — оно появляется внизу, скроллбар следует
- Это привычный паттерн для чата
- Текст ввода естественно находится внизу окна
- Естественно для "ведения диалога"

Я склоняюсь к **варианту Б** (как в мессенджере). Причины:
- Conversation — это диалог. Привычная модель для пользователя.
- Поле ввода внизу — логично следует за лентой.
- При scroll вверх — видишь историю, при scroll вниз — последнее.

Это ASCII-эскиз и предполагает.

### 6.7 Selector целевого домена

```
Target domain: [LOGIC (101) ▾]
```

Dropdown с одиннадцатью вариантами доменов. Дефолт — **LOGIC (101)**, потому что это естественная "точка входа" для текстов.

Возможные варианты:
- LOGIC (101) — для логических утверждений
- INTNT (105) — для текстов, выражающих намерение
- ETHIC (103) — для этических вопросов
- (любой из остальных, если нужно)

При смене домена — выбор сохраняется в текущей сессии (но не в persistent settings — после рестарта возвращается к LOGIC).

### 6.8 Submit и обратная связь

При нажатии Submit:

```
1. Поле ввода блокируется (disabled state) на доли секунды
2. Сообщение появляется в ленте с timestamp
3. Команда SubmitText отправляется в Engine
4. После получения CommandResult:
   a) Success: поле ввода разблокируется и очищается
   b) Error: поле ввода разблокируется БЕЗ очистки (текст остаётся для редактирования),
              в ленте появляется сообщение об ошибке с тёплым красным
```

**Submit по горячей клавише:** Ctrl+Enter (или Cmd+Enter на macOS). Простой Enter — переход на новую строку.

### 6.9 История чата — нет в V1.0

Conversation — **текущая сессия** только. При закрытии Workstation лента очищается. После рестарта — пустая Conversation.

Это сознательное решение V1.0:
- Long-term история чата требует Engine хранить переписку (через EXPERIENCE как поток токенов с особой природой)
- В V1.0 этого не делаем — это работа Companion в будущем
- Если очень нужна история — можно скопировать вручную из ленты до закрытия

В deferred-документе это отмечено как кандидат для V2.0.

### 6.10 Связь с Patterns

**Patterns окно показывает то же самое**, но в более структурированном виде. Conversation — для непосредственного "подал-получил-увидел". Patterns — для **системного наблюдения за всеми событиями**.

Полезный сценарий: открыть Patterns в detached на втором мониторе, Conversation в main window. Подаёшь текст — видишь его сразу в Conversation как факт подачи, и в Patterns как процесс обработки с деталями.

### 6.11 Detach-режим

Conversation **может быть detached**, особенно для сценария "Conversation в одном углу, Patterns в другом, System Map на третьем мониторе". Но это менее частый кандидат, чем System Map / Patterns.

В detached-режиме окно работает полноценно — лента + ввод. Размер окна свободно настраивается.

### 6.12 Состояния окна

**Norm:** лента заполнена, поле ввода активно.

**Empty (новая сессия):**
```
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│                                                              │
│                                                              │
│                                                              │
│           (Подайте текст системе для начала)                │
│                                                              │
│                                                              │
│                                                              │
└──────────────────────────────────────────────────────────────┘

Submit text: [____________________]    [ Submit ]
```

Текст центрирован в верхней панели, скромным серым цветом. Без фейкового "стартового сообщения от системы" — просто пусто.

**Disconnected:** поле ввода disabled, тёплый красный overlay "Engine not connected".

**Sending:** поле ввода блокировано на короткое время отправки.

### 6.13 Что НЕ делается в Conversation в V1.0

- **Нет инициации со стороны системы.** System не пишет первой, не задаёт вопросов, не обращается к пользователю.
- **Нет "богатого ответа".** Не показываем графики, embeddings, цитаты. Только короткое технически точное подтверждение.
- **Нет редактирования отправленных сообщений.** Подал — отправлено. Нельзя забрать назад.
- **Нет реакций/likes.** Не социальная сеть.
- **Нет режима voice.** Голос — для Companion.
- **Нет attached files в ленте чата.** Если нужно подать файл — это Files tab, не Conversation.
- **Нет команд через текст.** Команды (force-sleep, configure) — через UI кнопки, не через "/" команды в чате.

### 6.14 Открытые вопросы

- **Должен ли Submit очищать поле ввода всегда** или только при Success? Я склоняюсь к "очищать при Success, оставлять при Error" — позволяет пользователю отредактировать и попробовать снова.
- **Размер поля ввода** — фиксированная высота 3 строки или динамический рост? Я склоняюсь к **динамическому росту** до максимум 8-10 строк, потом скролл внутри поля.
- **Нужен ли pre-fill последнего отправленного текста?** Если пользователь хочет повторить — нажимает "↑" в поле ввода и получает последний текст. Можно сделать или не делать. Я склоняюсь к "не делать в V1.0", это deferred.

## 7. Files

### 7.1 Назначение

**Files** — окно для импорта внешних данных в систему через External Adapters. Здесь пользователь:

- Видит доступные адаптеры (PDF, plain text, JSON, etc.)
- Выбирает источник (файл/папку)
- Запускает импорт
- Следит за прогрессом
- Видит результаты

Это **третий способ подачи материала в систему** (после Conversation и будущего Vision/Audio в Companion).

### 7.2 Зачем это нужно

Одно дело — подать в систему пару фраз через Conversation. Другое — **импортировать книгу, корпус текстов или базу знаний**. Это требует адаптера (преобразователя формата в структуры AXIOM) и долгой обработки (минуты или часы).

Files даёт UI к этой подсистеме. Сам Workstation **не реализует** адаптеры — это работа Engine (через External Adapters V3.0). Workstation только **показывает их и управляет ими**.

### 7.4 Available adapters — детали

Список адаптеров приходит от Engine. Каждый адаптер — строка:

```
▸ PDF Reader   (.pdf)
```

При клике на строку — раскрывается описание:

```
▾ PDF Reader   (.pdf)
   Reads text and structure from PDF files.
   Extracts paragraphs and headings.
   Optionally extracts images.
   
   Source: axiom-runtime/external-adapters/pdf
```

Это позволяет пользователю **понять, что адаптер делает**, без чтения кода.

### 7.5 New import form — детали

**Source:**
- Кнопка `[Select file or folder...]` открывает системный файловый диалог
- После выбора — путь отображается рядом
- Поддержка drag-and-drop файла на эту область

**Adapter:**
- По умолчанию — `auto-detect` (определяется по расширению файла)
- Если auto-detect не сработал или пользователь хочет переопределить — dropdown со всеми адаптерами

**Options:**
- Список опций **зависит от адаптера**. Engine отдаёт описание опций каждого адаптера в `ListAdapters` ответе.
- Адаптер описывает свои опции через схему (тип, default value, диапазон, описание)
- Workstation рендерит UI элементы соответствующие схеме (text input, number input, dropdown, checkbox)

Пример опций для разных адаптеров:

```
PDF Reader:
  [✓] Extract images
  Language hint: [en ▾]

Plain Text:
  Chunking strategy: [paragraph / sentence / fixed_size ▾]
  Chunk size: [500] tokens (only if fixed_size)

JSON Dump:
  Schema: [generic / wikidata / custom ▾]
  Filter: [_______________] (optional regex)
```

### 7.6 Start import — что происходит

```
1. Пользователь нажимает "Start import"
2. Workstation отправляет StartImport command в Engine
3. Engine возвращает CommandResult с import_id
4. В нижней панели появляется новая строка с progress bar (0%)
5. Engine начинает работу адаптера, отправляет AdapterStarted event
6. По мере работы — AdapterProgress events
7. Workstation обновляет progress bar и счётчик добавленных токенов
8. По завершении — AdapterFinished event
9. Прогресс bar заменяется на checkmark ✓ или ✗ при ошибке
10. Импорт перемещается в "completed"
```

### 7.7 Active import — детали

```
⟳ articles_corpus.txt          [Plain Text]
   ████████░░░░░░░░░  48% (24 of 50 chunks)
   2,341 tokens added so far
   [ Cancel ]
```

- **Иконка ⟳** — анимированный indicator работы (медленное вращение, не быстрое)
- **Имя файла** — основное имя без полного пути
- **Адаптер** — какой адаптер используется
- **Progress bar** — заполненная часть и счётчик единиц
- **Tokens added so far** — сколько токенов добавлено в EXPERIENCE (через GUARDIAN, прошедших фильтр)
- **Cancel** — кнопка отмены импорта

### 7.8 Cancel import

При клике Cancel:
- Подтверждение: "Cancel import? Tokens added so far will remain in the system."
- При подтверждении — отправка `CancelImport` command
- Engine останавливает работу адаптера на следующей возможности
- Импорт перемещается в completed как "cancelled"

В V1.0 нет rollback — добавленные токены остаются. Это deferred.

### 7.9 Completed imports — детали

Завершённые импорты остаются в нижней панели для просмотра:

```
✓ wikipedia_sample.xml         [Wikipedia]
   completed 12 minutes ago
   12,840 tokens added, 23 errors
```

- **✓ для success, ✗ для error, ⊘ для cancelled**
- Имя файла, адаптер, время завершения
- Счётчики: добавленных токенов и ошибок
- При клике — раскрывается detail с deeper стат и возможным error message

При накоплении большого количества — `Show more...` для подгрузки старых.

### 7.10 Параллельные импорты

В V1.0 поддерживается множество одновременных импортов (если Engine это позволяет). Все они отображаются в "Active" части.

При высокой нагрузке (5+ параллельных импортов одновременно) — Workstation просто показывает все. Engine может throttling-ить между ними согласно своим правилам.

### 7.11 Поведение в DREAMING

Во время сна системы импорты **могут быть приостановлены** (это решение Engine, не Workstation). Если так — импорт показывается с иконкой ⏸ и пометкой "paused (engine sleeping)". При пробуждении — продолжает.

Это важно: **импорт не должен мешать сну**. Сон — приоритетное состояние.

### 7.12 Detach-режим

Files **обычно не detached**. Это служебное окно — открыл, выбрал файл, запустил, закрыл. Долгое наблюдение не требуется.

Но **технически** detach возможен — иногда при длительном импорте полезно держать окно открытым на втором мониторе и видеть прогресс.

### 7.13 Состояния окна

**Norm:** список адаптеров, форма импорта, активные/завершённые.

**No adapters available:** "No external adapters configured in engine. To enable knowledge import, configure adapters in engine settings."

**Engine disconnected:** все панели пустые, overlay "Engine not connected".

### 7.14 Что НЕ делается в Files в V1.0

- **Не реализуем сами адаптеры.** Это Engine.
- **Не редактируем добавленный материал.** После импорта — он живёт в EXPERIENCE.
- **Не делаем pause/resume импорта.** Только cancel. Pause — V2.0.
- **Не строим preview содержимого файла перед импортом.** Файл — чёрный ящик до запуска адаптера.
- **Не делаем планировщик импортов.** "Импорт каждый день в 3am" — нет в V1.0.
- **Не объединяем файлы.** Импорт одного файла за раз, каждый отдельно. Группировка нескольких — V2.0.

### 7.15 Открытые вопросы

- **Drag-and-drop из системы.** Желательно поддерживать, но iced может ограничивать. Уточнить на этапе реализации.
- **Подсказка по выбору адаптера при auto-detect.** Если расширение определяется неоднозначно (например, .json может быть и dump, и custom формат) — что показать? Я склоняюсь к "выбрать наиболее общий, при этом показать tooltip с альтернативами".
- **Время хранения completed imports.** Сколько держать в списке? 50? 100? Я склоняюсь к **100** с автоматическим выбросом старейших.

## 8. Configuration

### 8.1 Назначение

**Configuration** — окно настроек системы. Здесь пользователь:

- Видит текущую конфигурацию Engine (только чтение и редактирование разрешённых частей)
- Меняет параметры с hot-reload (без перезапуска Engine)
- Управляет Workstation-настройками (UI preferences, connection)

### 8.2 Зачем это нужно

AXIOM имеет много параметров, которые могут быть настроены: thresholds для сна, веса fatigue, intervals для FrameWeaver, правила кристаллизации, и т.д. Без UI конфигурации все эти параметры пришлось бы менять через CLI или вручную в YAML файлах.

Configuration tab даёт **визуальный доступ** к этим параметрам с возможностью их менять и видеть эффект сразу (благодаря hot-reload в Engine).

### 8.3 Композиция

Структура:

**Левая панель — Sections (~ 25%):**
- Дерево категорий
- Группы Engine и Workstation
- Модифицированные секции отмечены (●)

**Правая панель — текущая категория (~ 75%):**
- Параметры выбранной категории
- Кнопки в нижней части: Reset to defaults, Apply changes, Discard

### 8.4 Категории конфигурации

**Engine (требует engine connection):**

- **Dream Phase** — пороги, fatigue weights, cycle limits (см. эскиз выше)
- **FrameWeaver** — scan_interval_ticks, stability_threshold, promotion rules
- **Domains** — DomainConfig для каждого из 11 доменов (вложенный subtree)
- **GUARDIAN** — настройки CODEX, ingibition rules
- **Persistence** — Memory Persistence settings, save intervals

**Workstation (локальные настройки):**

- **Connection** — engine address, port, reconnect behavior
- **Display** — UI preferences, theme variant (если будут варианты), animation speed
- **Benchmarks** — bench-history.md location, default iterations

### 8.5 Hot-reload

Большинство Engine параметров поддерживают hot-reload (можно менять без перезапуска Engine). При нажатии "Apply changes":

1. Workstation отправляет `UpdateConfig` команду в Engine
2. Engine применяет изменения к runtime config
3. Engine отвечает CommandResult
4. Если Success — пометка "● modified" исчезает с категории

Параметры **которые не поддерживают hot-reload** — помечены в UI:

```
Domains > LOGIC (101) > token_capacity:  [50000]   [restart required]
```

При изменении такого параметра — предупреждение: "This change requires engine restart to take effect. The current value will be used until restart."

### 8.6 Reset to defaults

Кнопка возвращает все параметры **в текущей категории** к default-значениям.

При нажатии — диалог подтверждения: "Reset Dream Phase settings to defaults?"

Это локальная операция — изменения становятся pending до Apply.

### 8.7 Discard

Отменяет несохранённые изменения в текущей категории.

### 8.8 Apply changes — детали

Применение изменений может быть **по категории** или **глобальным**:

- **По категории** (default): кнопка в правой панели применяет только изменения текущей категории
- **Глобальное**: меню → Configuration → Apply all changes — применяет все pending изменения во всех категориях

Глобальное Apply полезно когда пользователь сделал изменения в нескольких категориях и хочет применить все сразу.

### 8.9 Validation

Перед отправкой Apply changes:

- Workstation проверяет валидность значений (например, число в правильном диапазоне)
- Если ошибка — выделяет невалидное поле тёплым красным с tooltip с описанием
- Apply отключается до исправления

Engine также делает валидацию на своей стороне — если по какой-то причине проверка Workstation пропустила невалидное значение, Engine отвечает Error в CommandResult.

### 8.10 Detach-режим

Configuration **не имеет** detach. Это служебное окно (см. Документ 2 раздел 3.5).

### 8.11 Состояния окна

**Norm:** показывается дерево категорий и параметры выбранной.

**Engine disconnected:** только Workstation секции доступны для редактирования. Engine секции — серые с пометкой "Engine not connected".

**Loading config:** при первом подключении или после reconnect — placeholder "Loading configuration..." пока snapshot не пришёл.

### 8.12 Что НЕ делается в Configuration в V1.0

- **Не редактируем GENOME правила.** GENOME — frozen после boot, по дизайну. Configuration tab показывает GENOME read-only, но не позволяет менять.
- **Не редактируем Schema Configuration через текстовый редактор YAML.** В V1.0 — только UI редактор. Текстовый — V2.0.
- **Не делаем undo/redo для конфигурации.** Изменения накапливаются, Apply применяет, Discard отменяет — без полной истории изменений.
- **Не делаем профили конфигурации.** "Профиль для разработки", "профиль для production" — V2.0.

### 8.13 Открытые вопросы

- **Какие именно параметры показывать?** Не все Engine параметры стоит выставлять в UI — некоторые слишком низкоуровневые (например, точные оффсеты в memory persistence). Решается на этапе реализации с учётом review каждого параметра.
- **Как показывать Domains config?** 11 доменов с большим количеством параметров каждый. Возможно, master-detail view внутри категории Domains.

### 9.7 Cancel running benchmark

- Подтверждение (короткий popup): "Cancel benchmark? Partial results will not be recorded."

- Результаты не сохраняются

### 9.8 Detach-режим

### 9.9 Состояния окна

**No history:** "No benchmark runs yet. Run a benchmark to start tracking performance."

**Engine disconnected:** Run panel disabled с overlay "Engine not connected".

**Bench-instance starting:** прогресс "Starting bench-instance engine..."

**Bench-instance running:** прогресс бенчмарка с возможностью cancel.

### 9.10 Что НЕ делается в Benchmarks в V1.0

- **Конструктор кастомных сценариев.** Только список из 6 предустановленных.
- **Сравнение конфигураций side-by-side.** Запустил бенчмарк → изменил конфиг → запустил снова. Сравнение визуально через тренд, не через UI side-by-side.
- **Автоматический бенчмарк по расписанию.** "Каждое утро в 6am" — V2.0.
- **CI integration.** Workstation — для интерактивного использования.

### 9.11 Открытые вопросы

- **Как показывать длительные бенчмарки.** Если бенчмарк идёт 5 минут — пользователь должен иметь представление о прогрессе. Я предлагаю прогресс bar в строке бенчмарка с обновлением каждую секунду от bench-instance.
- **Поддержка диапазонов.** Хочет пользователь запустить hot_path_tick для 1k, 5k, 10k, 50k итераций сразу — это конструктор сценариев, V2.0.

## Резюме Части C

В этой части определены четыре окна взаимодействия:

1. **Conversation** — простой утилитарный чат для подачи текста и получения отклика. Не центр интерфейса. Без long-term истории, без поэтики, без инициативы со стороны системы.

2. **Files** — окно External Adapters для импорта внешних данных. Список адаптеров, форма нового импорта, прогресс активных, завершённые с результатами.

3. **Configuration** — настройки Engine (с hot-reload) и Workstation. Дерево категорий слева, параметры справа. Apply changes / Reset / Discard.

4. **Benchmarks** — запуск предустановленных бенчмарков, история результатов в bench-history.md, экспорт. Через bench-instance Engine.

Все четыре — служебные. Открываются по необходимости. **Не претендуют на главную позицию** в интерфейсе. Сохраняют общую эстетику минимализма и спокойствия.

После согласования Части C — переходим к Части D: общие компоненты, сводка, итоговое резюме всех окон.
