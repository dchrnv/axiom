## Контекст Части B

Часть A описала Welcome (первое впечатление) и System Map (главное окно). Это два окна, которые **дают общую картину**: видишь, что система живёт, в каком она состоянии, какие у неё потоки.

Часть B — **углублённое наблюдение**. Когда пользователь хочет увидеть **детали**: как именно движутся токены в полях, какие синтаксические узоры собираются, что происходит во сне.

Эти три окна имеют общий характер:

- **Плотность информации выше**, чем в System Map. Здесь можно показывать много деталей.
- **Эстетика та же** — школа Cupertino, минимализм, ограниченная палитра. Несмотря на плотность.
- **Главное — наблюдение**, не действие. Эти окна минимально интерактивны — в основном смотришь.

Все три могут быть detached на второй монитор — что особенно полезно для Patterns и Live Field, которые могут работать в качестве "экранов слежения" во время чтения текстов или импорта.

## 3. Live Field

### 3.1 Назначение

**Live Field** — буквальная физическая визуализация полей доменов AXIOM. Здесь пользователь видит **движение токенов**: где они находятся в трёхмерном пространстве домена, как они движутся, как формируются связи.

Это самое **физическое** окно из всех. В System Map были схематичные блоки доменов — здесь домен показан как **поле с летающими в нём точками**.

### 3.2 Зачем это нужно

Архитектура AXIOM построена на идее, что токены живут в трёхмерных полях с физикой (gravity, friction, temperature, cooling rate). Это не абстракция — это реальное представление в коде. Без Live Field пользователь видит только метрики (количество токенов, средняя температура), но не **видит саму физику в действии**.

Live Field превращает физику в видимое.

### 3.3 Что показывается

В каждый момент времени Live Field показывает **один выбранный домен** (пользователь выбирает из списка):

- 3D-поле домена с осями (отображаются как лёгкая координатная сетка)
- Токены в нём — как **точки** с цветом и размером
- Связи между токенами — как **тонкие линии** (опционально, можно отключить если перегружено)
- Anchor-токены — выделены особо (большие, статичные)

**Цвет точки = семантический слой**. У каждого токена есть Shell-профиль с активациями по 8 слоям (L1 physical, L2 sensory, ..., L8 abstract). На Live Field точка окрашивается в **доминирующий слой** — какой из 8 слоёв сейчас доминирует в её shell.

Восемь слоёв получают восемь оттенков (внутри ограниченной палитры — оттенки нейтральной базы с лёгкой примесью одного из цветов состояний). Конкретное соответствие — этап дизайна.

**Размер точки = масса токена.** Большой токен виден сразу. Маленькие — мелкие точки. Anchor-токены имеют максимальный размер и не движутся.

**Прозрачность точки = температура.** Высокая температура — насыщенная точка. Низкая (затухающий токен) — почти прозрачная. Это позволяет **видеть, что важно**: яркие активные токены, тающие в фон затухающие.

### 3.5 Камера и навигация

В V1.0 камера простая:
- **Drag мышью** — поворот вокруг центра поля
- **Scroll** — приближение/отдаление
- **Двойной клик в пустом месте** — сброс камеры в дефолтное положение

Без сложных режимов (orbit/walk/fly), без сохранения видов, без presets. Просто — взгляд на поле с возможностью повернуть.

**Дефолтное положение** — изометрический вид с лёгким наклоном. Видно все три оси.

### 3.6 Display options

```
[✓] Show connections      — показывать линии связей
[✓] Show anchors          — выделять anchor-токены
[✓] Layer color coding    — окрашивать токены по доминирующему слою
[ ] Highlight recent      — подсвечивать токены, появившиеся за последние N секунд
```

**Show connections** — может перегружать вид при тысячах связей. По умолчанию включено для маленьких доменов (< 500 токенов), отключено для больших. Пользователь может переключить.

**Show anchors** — anchor-токены отображаются как чуть более крупные и более насыщенные точки (даже если их температура низкая). Выключение прячет это выделение, anchor становится одной из точек.

**Layer color coding** — главный механизм понимания "что есть что". Если выключить — все точки в нейтральном тоне, видно только движение и плотность, но не семантику.

**Highlight recent** — подсвечивает свежие токены (появились за последние 10 секунд). Полезно при подаче нового материала — видишь, куда он лёг. По умолчанию выключено.

### 3.7 Анимация и динамика

Live Field — самое **динамическое** окно. Здесь токены реально движутся.

- **Frame rate** — 30-60 fps. Зависит от производительности машины.
- **Плавность** — движение интерполируется между кадрами Engine. Engine может слать обновления раз в N тиков, но визуализация показывает плавное движение между ними.
- **Связи** — линии следуют за движущимися токенами в реальном времени.
- **Появление токена** — fade-in анимация (~ 200ms), не резкое возникновение
- **Исчезновение токена** — fade-out (~ 500ms), плавно растворяется

### 3.8 Domain stats — числовые показатели

Показатели выбранного домена:

```
Tokens:        742           — общее количество (без removed)
Connections:   1280          — общее количество активных связей
Avg temp:      142 / 255     — средняя температура
Layer activations:           — sparkline по 8 слоям
  L1 ▂▃▃▁▁▁▁▁
```

Layer activations — sparkline-микрографики **по каждому слою отдельно**, не один общий. Это позволяет видеть, какие слои активны: например, в LOGIC доменe L5 (cognitive) и L8 (abstract) высокие, а L1-L2 (physical/sensory) низкие.

### 3.9 Производительность

Live Field — **самое дорогое окно в Workstation**. 3D-рендеринг точек, связей, анимация — это нагрузка на UI thread.

Стратегии управления нагрузкой:

- **Tier visualization.** Если в домене > 5000 токенов — снижать качество рендеринга (без анимации появления, simplified shading, possible chunking).
- **Frustum culling.** Не рендерить точки вне поля зрения камеры.
- **Connection limit.** Если связей > 10000 — рендерить только связи к токенам в поле зрения.
- **Adaptive frame rate.** Если FPS падает ниже 20 — автоматически отключать "Show connections" и "Highlight recent".

Это всё — **этап реализации**. На этапе дизайна важно знать, что такие меры предусмотрены.

### 3.10 Detach-режим

Live Field — **второй кандидат на detach** после System Map.

Сценарий: пользователь хочет наблюдать движение токенов в EXPERIENCE во время большого импорта. Detach Live Field на второй монитор, выбрать EXPERIENCE — и видно, как материал заполняет поле в реальном времени.

В detached-режиме окно работает как **живой пейзаж домена**. С навигацией камерой, всеми display options, всем функционалом.

### 3.11 Состояния окна

**Norm:** домен выбран, токены отображаются.

**No domain selected:** placeholder с текстом "Select a domain to view its field" и подсказкой.

**Domain has no tokens:** placeholder с текстом "Domain is empty" + базовые stats (0 tokens, 0 connections).

**Disconnected:** последнее состояние замороженное с overlay "Connection lost".

### 3.12 Что НЕ делается в Live Field в V1.0

- **Не редактируется** — нет добавления/удаления токенов через клик. Это работа Engine.
- **Нет search/filter** — ищем токен по ID, фильтруем по слою. Сложно сейчас, в V2.0.
- **Нет multi-domain view** — одно окно — один домен. Хочешь видеть два — открой Live Field в двух detached-окнах.
- **Нет сохранения сцены** — закрыл окно, открыл заново — видишь текущее состояние, не "то, что было час назад".
- **Нет 2D-проекций** — только 3D. Слишком много вариантов для V1.0.

### 3.13 Открытые вопросы

- **3D-движок.** Iced сам по себе не имеет полноценного 3D. Возможные варианты: использование `iced::widget::canvas` с собственным lightweight 3D-рендером, или интеграция с `wgpu` напрямую через iced wgpu backend. Решается на этапе реализации.
- **Координаты осей.** Domain имеет 3D-пространство, но что значат оси X/Y/Z в каждом домене? Это вопрос к Engine, не к Workstation, но в UI имеет смысл показывать **подписи осей** (например, в LOGIC: X = "Generality", Y = "Confidence", Z = "Time of formation"). Это требует Engine отдавать описание осей в DomainSnapshot.
- **Кастомизация цветов слоёв.** В V1.0 — фиксированные дефолты. Кастомизация — V2.0.

## 4. Patterns

### 4.1 Назначение

**Patterns** — окно семантических и синтаксических узоров в реальном времени. Здесь видно:

- Какие Frame кристаллизуются (выходы FrameWeaver)
- Какие синтаксические подтипы (S1-S8) активны прямо сейчас
- Какие семантические слои (L1-L8) доминируют в системе
- Когда происходят реактивации существующих Frame
- Когда GUARDIAN отклоняет создание узоров

Это окно **специально для понимания того, как система структурирует входящий опыт**. Когда пользователь подаёт текст в Conversation или импортирует корпус через Files — Patterns показывает, что система **извлекла** из этого материала.

### 4.2 Зачем это нужно

FrameWeaver V1.1 — серьёзная подсистема. Без Patterns пользователь не видит её работы напрямую. Видит только результаты в EXPERIENCE через метрики ("1247 frames"). Patterns даёт **прозрачность процесса**: видно, как именно собираются узоры, какие участники, какие связи.

Это особенно важно при отладке: если пользователь подал текст и ожидал кристаллизации Frame, но её не произошло — Patterns покажет, **почему** (стабильность не достигнута, GUARDIAN отклонил, не хватило участников).

### 4.4 Active layers — детали

Sparkline для каждого слоя — последние 60 секунд активности этого слоя. Текущий уровень обозначен текстом (silent / low / medium / high / highest) — это **категории, не точные числа**, которые читаются с одного взгляда.

Семантические L1-L8 — это слои Shell (см. Shell V3.0):
- L1 physical, L2 sensory, L3 motor, L4 emotional, L5 cognitive, L6 social, L7 temporal, L8 abstract

Синтаксические S1-S8 — это слои FrameWeaver V1.1:
- S1 core arguments, S2 modification, S3 structural, S4 pragmatic, S5 temporal frame, S6 spatial frame, S7 causal internal, S8 meta

**Точки рядом с syntactic** — индикаторы текущей активности:
- ●●● — высокая активность (highest)
- ●● — средняя
- ● — лёгкая
- (пусто) — silent

Это даёт **двойную считываемость**: можно посмотреть на sparkline для тренда, или на точки для текущего состояния.

### 4.5 Recent Frames — детали

Каждое событие — карточка одного из четырёх типов:

**1. Кристаллизация (●):**
```
● Frame #1247  syntactic  3 participants
   domains: 101, 109
   layers: S1, S7    just now
```

- Иконка ● в нейтральном тоне (или в цвете завершения = шалфей?)
- Frame ID, тип, количество участников
- Список доменов, в которых живут участники
- Какие синтаксические слои присутствуют
- Время

**2. Реактивация (↻):**
```
↻ Frame #1198  reactivated (temperature: 87 → 142)
   12 sec ago
```

- Иконка ↻ — стрелка по кругу
- Frame ID, изменение температуры
- Время

**3. GUARDIAN veto (⊗):**
```
⊗ Frame candidate vetoed by GUARDIAN
   reason: "missing layer S1"
   34 sec ago
```

- Иконка ⊗ — крестик
- Цвет — тёплый красный (внимание)
- Причина отклонения
- Время

**4. Промоция в SUTRA (↑):**
```
↑ Frame #1042  promoted to SUTRA  (rule: deep_causal)
   1 min ago
```

- Иконка ↑
- Цвет — зелёный шалфей (завершение)
- Frame ID, имя правила промоции
- Время

### 4.7 Поведение при появлении новых событий

**Кристаллизация Frame:**
- Карточка появляется наверху ленты с fade-in (~ 300ms)
- Существующие карточки плавно сдвигаются вниз
- Без звука, без всплывающего уведомления
- В Active layers панели — соответствующие sparkline получают новую точку

**Большой объём событий (например, во время импорта):**
- Лента может быстро меняться
- Throttling — при > 5 событий в секунду показывается агрегированно: "5 frames crystallized in last second"
- При расслаблении темпа — возвращается к индивидуальному отображению

### 4.8 Domain selector — нет в V1.0

В Live Field был выбор домена. В Patterns по умолчанию — **показ всей системы целиком**. Можно отфильтровать по домену, но фильтр в V1.0 простой: галочки сверху ленты (deferred — оставить на V2.0 если будет нужно).

### 4.9 Detach-режим

Patterns — **третий кандидат на detach** (после System Map и Live Field).

Особенно полезен в сценарии "наблюдение во время взаимодействия с системой":
- Detached Patterns на втором мониторе
- Подаёшь текст через Conversation в main window
- Видишь, как растут активации в Patterns в реальном времени
- Видишь, какие Frame возникают

Это **прямой канал понимания системы**.

### 4.10 Состояния окна

**Norm:** активные слои показываются, лента событий заполнена.

**No recent activity:** "No recent pattern activity. Try submitting text in Conversation or importing a file."

**No frames yet:** "System is running but no frames have been crystallized yet. As FrameWeaver scans for patterns, they will appear here."

**Disconnected:** последнее состояние замороженное.

### 4.11 Что НЕ делается в Patterns в V1.0

- **Не редактируем Frame.** Не удаляем, не модифицируем. Это работа Engine.
- **Не управляем правилами кристаллизации.** Конфигурация правил — в Configuration tab, не здесь.
- **Не строим графы Frame.** Только список. Граф взаимосвязей Frame между собой — V2.0.
- **Не сохраняем историю сессии в файл.** Лента — это runtime буфер. После рестарта — пуста.

### 4.12 Открытые вопросы

- **Цвета индикаторов кристаллизации.** Я склоняюсь к нейтральному тёмному (графитовому) для иконки ●, шалфейному для ↑ (промоция), тёплому красному для ⊗ (veto), нейтральному с лёгким акцентом для ↻ (реактивация). Точные оттенки — этап дизайна.
- **Активность layers — как именно нормализовать число событий в "silent / low / medium / high"?** Это зависит от темпа системы. Нужны хорошие пороги, которые могут быть разными для разных слоёв (например, S5 silent в большинстве случаев — это нормально).
- **Размер ленты Recent Frames.** В памяти держим до 100-200 карточек, под "Show more..." подгружаем глубже из 1000-event-buffer. Достаточно?

## 5. Dream State

### 5.1 Назначение

**Dream State** — окно текущего состояния DREAM Phase. Здесь видно:

- В каком состоянии система прямо сейчас (WAKE / FALLING_ASLEEP / DREAMING / WAKING)
- Текущий уровень fatigue с детализацией по факторам
- Идущий DreamCycle, если система спит сейчас
- История последних снов (DreamReports)
- Кнопки force-sleep и wake-up (явное управление)

Это окно **специфично для DREAM Phase V1.0**. Без него пользователь видит состояние сна только косвенно (через цвет мандалы на System Map). Здесь — детальная картина.

### 5.2 Зачем это нужно

DREAM Phase — серьёзная и тонкая подсистема. Понимание, **когда система засыпает и почему**, **что происходит во сне**, **как часто промоутируются Frame** — критично для развития проекта.

Также Dream State позволяет **управлять циклом сна вручную** для тестирования: заставить заснуть для проверки, разбудить для прерывания, увидеть результаты.

### 5.4 Current state — детали

В состоянии **WAKE** (нормальная работа):
```
       WAKE
    
    for 23 minutes
    
    [Force sleep]
```

Время с момента последнего пробуждения. Кнопка **[Force sleep]** — для явного засыпания (отправляется ExplicitCommand trigger в DreamScheduler). При нажатии — диалог подтверждения:

```
"Force engine to sleep?"
The current dream cycle will be initiated after the
next tick.
[ Cancel ]  [ Sleep now ]
```

В состоянии **DREAMING**:
```
      DREAMING
    
    for 1 minute 23 seconds
    
    Stage: Processing (4 of 12 proposals)
    
    [Wake up]
```

Текущий этап DreamCycle (Stabilization → Processing → Recombination → Consolidation), сколько proposals обработано из всего количества. Кнопка **[Wake up]** — для прерывания (отправляется critical priority signal).

В состоянии **FALLING_ASLEEP** или **WAKING**:
```
    FALLING_ASLEEP
    
    transitioning...
```

Без кнопок — это короткие переходные состояния, длительностью секунды.

### 5.5 Fatigue panel — детали

```
Total: 47 / 255  ▂▂▃▃▃▂▂▁

By factor:
• Uncrystallized candidates:  8  / 80
• EXPERIENCE pressure:       24  / 100
• Heavy proposals:            6  / 60
• Causal horizon growth:      9  / 30
```

Total — общий fatigue (0-255), с sparkline тренда последние 60 секунд.

By factor — каждый из четырёх компонентов FatigueWeights:
- **Uncrystallized candidates** — сколько кандидатов в FrameWeaver висит без кристаллизации (вес 80 default)
- **EXPERIENCE pressure** — насколько EXPERIENCE приближается к token_capacity (вес 100)
- **Heavy proposals** — сколько тяжёлых предложений в очереди DreamPhase (вес 60)
- **Causal horizon growth** — насколько быстро растёт горизонт без переработки (вес 30)

Это даёт **прозрачность**: видно, **почему** растёт fatigue. Если EXPERIENCE pressure вырос до 80 из 100 — пора засыпать.

При переходе fatigue через threshold (180/255 default) — fatigue панель меняет тон, фон становится тёплым янтарным как сигнал "близко к засыпанию".

### 5.6 Recent dreams — детали

Лента DreamReport-ов в обратном хронологическом порядке. Каждый dream:

- **Иконка ● или ⊘**: ● completed normally, ⊘ interrupted
- **Номер сна**: уникальный counter за время работы engine
- **Старт-конец-длительность**
- **Триггер**: idle / fatigue / explicit
- **Причина пробуждения**: cycle_complete / critical_signal / timeout / guardian_override
- **Результаты обработки**: processed / approved / vetoed / deferred
- **Промоции**: сколько Frame продвинуто в SUTRA в этом сне
- **Изменение fatigue**: до и после

### 5.7 Прогресс DreamCycle во время DREAMING

Когда система **спит сейчас** — Dream State показывает прогресс:

```
      DREAMING
    
    for 1 minute 23 seconds
    
    ┌────────────────────────────────────────┐
    │ Stabilization ✓                        │
    │ Processing    [▓▓▓▓░░░░░░] 4/12        │
    │ Consolidation ○                        │
    └────────────────────────────────────────┘
    
    Current proposal:
      Promotion of Frame #1287
      Rule: deep_causal
      Priority: 80
    
    [Wake up]
```

- Зелёная галочка — этап завершён
- Прогрессбар — текущий этап с прогрессом
- Пустой кружок — этап впереди
- Текущий обрабатываемый proposal с типом, правилом и приоритетом

**Это единственное место в Workstation, где есть прогрессбар.** Остальные индикации — sparklines, иконки, цвета. Здесь прогрессбар оправдан, потому что DreamCycle — это **дискретный процесс с известным концом**. Это не "прогресс жизни системы", это конкретная процедура.

### 5.8 Detach-режим

Dream State — может быть detached, но не самый частый кандидат. Полезно для длительного наблюдения за циклами сна (например, в сценариях стресс-тестирования или когда импортируется большой корпус и хочется видеть, как система им переваривает).

### 5.9 Состояния окна

**Norm (WAKE):** текущее состояние, fatigue, лента снов.

**Norm (DREAMING):** текущее состояние, fatigue (заморожена на момент засыпания), прогресс DreamCycle.

**No dreams yet:** "System has not slept yet. As fatigue accumulates or system goes idle, the first dream will begin."

**Disconnected:** последнее состояние замороженное.

### 5.10 Что НЕ делается в Dream State в V1.0

- **Не редактируем правила сна.** Это в Configuration tab.
- **Не показываем рекомбинацию во сне.** В V1.0 этап Recombination — заглушка. В V2.0 будет визуализация.
- **Не визуализируем DREAM(107) физикой.** Если хочется видеть физику DREAM-домена — Live Field с выбором DREAM.
- **Не сохраняем DreamReport в файл.** Они хранятся в EXPERIENCE как токены (см. DREAM Phase V1.0 раздел 9.3).

### 5.11 Открытые вопросы

- **Точные пороги fatigue для "nearly sleeping" предупреждения.** 80%? 90%? Нужен ли вообще этот предупредительный режим?
- **Force sleep — нужно ли подтверждение всегда, или можно настроить "just do it"?** Я склоняюсь к подтверждению по умолчанию, но возможна опция в Configuration "skip confirmation for force-sleep".
- **Длина ленты Recent dreams.** Я предлагаю показывать последние 20 в окне, остальные через "Show more". При очень частых снах (каждые 5 минут) это будет день истории.

## Резюме Части B

В этой части определены три окна углублённого наблюдения:

1. **Live Field** — буквальная физика. 3D-визуализация полей доменов с летающими токенами. Самое дорогое окно. Главный кандидат на detach для слежения за импортом или большой нагрузкой.

2. **Patterns** — синтаксические/семантические узоры в реальном времени. Показывает работу FrameWeaver: какие Frame кристаллизуются, реактивируются, отклоняются GUARDIAN, промоутируются. Прямой канал понимания, как система структурирует опыт.

3. **Dream State** — текущее состояние DREAM Phase. Fatigue с разбивкой по факторам. Прогресс текущего DreamCycle. История DreamReports. Управление сном вручную (force-sleep, wake-up).

Все три окна — глубокие, плотные, спокойные. Они дополняют System Map: System Map даёт **общую картину**, эти три — **детальные взгляды на конкретные аспекты**.

После согласования Части B — переходим к Части C: окна взаимодействия (Conversation, Files, Configuration, Benchmarks).
